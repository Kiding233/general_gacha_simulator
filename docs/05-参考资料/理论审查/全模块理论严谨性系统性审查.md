### 2.1 经验分布模块 (distribution.py)

**功能**：EmpiricalDistribution（分位数/CDF/CVaR）、JointSamples、WorstCaseAnalysis、freedman_diaconis_bins

#### 合格项

- **分位数估计 (R-type 7)**：`idx = p·(n-1)` 线性插值。与 `numpy.percentile` 和 R `quantile()` 默认行为一致。Hyndman & Fan (1996) 分类为 R-type 7，是统计软件的事实标准。对连续分布的 i.i.d. 样本无偏。

- **Freedman-Diaconis 分箱规则**：`h = 2·IQR·n^(-1/3)`。Freedman & Diaconis (1981) 推导了 L₂ MISE 最优性，IQR 替代标准差增强了对离群值的鲁棒性。对于抽卡模拟数据（资源分布通常有界），IQR 是比标准差更好的离散度度量。

- **CDF 二分查找**：O(log n)，数值正确。

#### 理论缺陷

**缺陷 1：CVaR 的 ceil 偏差（中等）**

```python
cutoff = int(math.ceil(alpha * self._n))
```

离散样本的 CVaR 理论定义为对 VaR 以下样本的条件期望。`ceil` 导致尾部比例系统性膨胀：n=50, α=0.01 时，`ceil(0.5)=1`，取 2% 尾部而非 1%。正确的估算应将分数权重计入：

```
CVaR = (1/(α·n)) · [Σ_{i=1}^{⌊α·n⌋} X_{(i)} + (α·n - ⌊α·n⌋) · X_{(⌊α·n⌋+1)}]
```

在典型模拟规模（n > 1000）下可忽略，但在小样本使用中保守偏倚（高估风险）。

**缺陷 2：空 EmpiricalDistribution 静默返回 0（低等）**

`n=0` 时，`mean()`、`median()`、`quantile()`、`cvar()` 均返回 `0.0` 而非 `NaN`。下游代码无法区分「分布均值为 0」和「分布为空」。对于条件分布（如 `conditional_second_below_quantile(alpha=0.001)`），过滤后的空集不会被检测到。

#### 文献对照

| 方法 | 文献 | 一致性 |
|------|------|--------|
| R-type 7 分位数 | Hyndman & Fan (1996), *The American Statistician* | ✅ 一致 |
| CVaR（预期亏损）| Rockafellar & Uryasev (2000), *Journal of Risk* | ⚠️ 离散化偏差 |
| Freedman-Diaconis 分箱 | Freedman & Diaconis (1981), *Z. Wahrscheinlichkeitstheorie* | ✅ 一致 |

### 2.2 GDR 系统 (gdr.py)

**功能**：UNIFIED_GDR_REGISTRY 17 个指标、SuccessChecker、compute_weighted_gdr

**注**：实际注册表含 17 个条目，非 CLAUDE.md 和计划文档中记载的 13 个。

#### 合格项

- **目标达成类 GDR**（target_achievement、target_collection、all_targets）：截断计数 `min(count, target)` 防止超额计入，边界处理正确。

- **SuccessChecker**：统一的成功/失败判定接口，被所有分析面板复用，消除了重复逻辑。

- **draw_conversion_efficiency 的池类型过滤**：正确排除了兑换池和资源池的抽数，仅计入抽卡池。

#### 理论缺陷

**缺陷 3：resource_per_card 返回 inf 破坏下游聚合（严重）**

```python
if obtained <= 0: return float('inf')
```

任何包含此值的 `mean()` 或 `sum()` 都得到 `inf`，静默破坏所有聚合统计量。应返回 `float('nan')` 或使用哨兵值并在聚合时排除。

**缺陷 4：weighted_satisfaction 缺乏公理化效用理论基础（中等）**

该指标假设跨卡片的可加拟线性效用函数，无交互项。权重来自用户任意输入，无校准原则。当所有权重默认为 1.0 时退化为 Σ(2·got - target)，未归一化——跨不同 target_specs 配置不可比较。在期望效用理论中，desire 和 miss_cost 通过参考点关联（前景理论表明损失厌恶系数 λ > 1），而当前设计将它们视为独立参数。

**缺陷 5：total_card_value 对非目标卡隐式赋值 1.0（中等）**

```python
total_value += cnt * card_value_weights.get(card_id, 1.0)
```

对所有未配置 `card_value_weights` 的卡（包括所有非目标卡）隐式赋予权重 1.0。这意味着普通 SSR 的价值等于任意一张目标卡的 1 个副本——这几乎不可能是用户的意图。

**缺陷 6：资源单态假设（中等）**

所有基于资源的 GDR（resource_remaining、resource_consumed、resource_efficiency、resource_per_card、draw_conversion_efficiency）仅跟踪 `'draw_resource'` 键。多货币场景（免费石/付费石/门票）下无法区分，一个 GDR 值混入了不可比的经济维度。

**缺陷 7：旧 general_drop_rate.py 中 DropRateBetweenT1T2 忽略 t 参数（低等）**

`compute(t, history)` 方法忽略传入的 `t`，始终计算固定窗口 `[t1, t2)` 内的出率。这看起来是一个 bug 或设计意图与实现不一致。

**缺陷 8：weapon_character_ratio 的 weapon_character_map 无配置入口（已知）**

CLAUDE.md 和计划文档中已记录。GDR 始终返回 0.0。

#### 文献对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| 广义出率指标 | 无标准定义（项目独创） | N/A |
| 加权满意度 | 期望效用理论 (von Neumann-Morgenstern, 1944) | ⚠️ 可加性假设过强 |
| 渴望-未命中权重 | 前景理论 (Kahneman & Tversky, 1979) | ❌ 未建模损失厌恶 |

### 2.3 保底系统 (pity.py)

**功能**：PityEngine、SoftPityBehavior（线性/指数/阶梯）、HardPityBehavior、多保底叠加

#### 合格项

- **概率转移公式**：`new_p_target = min(current + progress·(1-current), 1.0)` ——「渐近填充」方案。非目标概率按比例缩放 `scale = (1-new)/(1-current)`，保持概率之和为 1.0。数学上正确。

- **马尔可夫性质**：每个保底计数器根据当前计数器值独立转移，联合向量过程也是马尔可夫的。

- **保底行为管道**：多个保底按 pity.txt 中的定义顺序依次调整概率分布。这在定义明确时是正确的。

#### 理论缺陷

**缺陷 9：get_pity_probabilities 对 _pity_state 产生副作用（严重）**

`StrategyContext.get_pity_probabilities()` 调用 `PityEngine.before_draw()`，**这会递增保底计数器**。任何查询概率但不实际抽卡的策略（特别是 PityReserveStrategy）都会静默产生「幽灵抽卡」——计数器递增但无实际抽卡发生。

具体影响链：
1. PityReserveStrategy 在每个决策步对所有当前池调用 `get_pity_probabilities`
2. 每个调用都触发 `before_draw` → 所有活跃保底计数器 +1
3. 若计数器从 79 开始，两次查询后变为 81 → 越过了软保底起点 80
4. 实际抽卡时保底状态已失真

**缺陷 10：保底目标 ID 不在池奖励中时概率未归一化（严重）**

`_apply_targeted` 和 `HardPityBehavior.apply` 中，`total_target_weight` 包含池奖励中不存在的 ID。这些 ID 消耗了权重分母但不分配概率，导致总概率 < 1.0。在硬保底场景（100% 目标概率）中，若部分目标 ID 不在此池，最终概率和可能远小于 1。

**缺陷 11：多保底管道顺序无优先级逻辑（中等）**

行为顺序完全由 `pity.txt` 中的出现顺序决定。当软保底和硬保底同时触发且目标集不同时，后处理的行为可能覆盖前者的调整。无冲突解决机制。

#### 文献对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| 软保底概率递增 | 伯努利试验序贯概率模型 | ✅ 正确 |
| 保底计数器的马尔可夫性 | 离散状态马尔可夫链 | ✅ 成立 |
| 多保底叠加 | 无标准文献（项目特定） | ⚠️ 需明确语义 |

### 2.4 策略系统 (strategy.py)

**功能**：STRATEGY_REGISTRY 7 种策略、StrategyContext、Action 类型

#### 合格项

- **策略注册表 + 工厂模式**：`create_strategy(name, params)` 提供统一接口，扩展性良好。

- **StrategyContext 封装**：将运行时信息集中传递，减少了全局变量依赖。

#### 理论缺陷

**缺陷 12：所有策略均为贪婪/启发式，不满足贝尔曼最优性（严重）**

7 种策略均为一阶前瞻（或零阶）：检查当前条件，立即行动，不递归评估未来状态的期望价值。这等价于在部分可观测环境中做贪婪决策。

具体后果：
- 没有策略能权衡「现在花费资源抽此池」vs「等待更好的池子」
- 没有策略能预测保底计数器在未来池子中的价值
- `future_schedules` 已传入 StrategyContext 但**没有任何策略使用它**

**缺陷 13：策略之间无统一效用函数，不可比（中等）**

各策略服务于不同的隐式目标（成功率最大化/资源最小化/风险规避），无共同度量标准。「策略 A 优于策略 B」的断言在不指定元目标的情况下无定义。

**缺陷 14：Information incompleteness — 策略缺乏前瞻信息（中等）**

StrategyContext 不提供：
- 未来资源收入预测（每日任务奖励）
- 未来池子的奖励分布（只有 pool_id 和时间）
- 时间贴现因子
- 池间保底链接关系

#### 理论澄清

策略系统采用**基于规则的启发式**而非期望效用最大化，这一选择本身是合理的——抽卡游戏的最优策略通常是 NP-hard 的（多池多保底多资源多目标的随机动态规划），实际游戏中玩家也使用启发式。理论审查的重点不是要求实现贝尔曼最优，而是**明确文档化这一设计决策**，并在 UI 中将策略输出标注为「启发式结果」而非「最优策略」。

#### 文献对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| 启发式决策 | Simon (1956) 满意原则 | ✅ 合理选择 |
| 贝尔曼最优性 | Bellman (1957) 动态规划 | ❌ 不满足 |
| 效用理论 | von Neumann-Morgenstern (1944) | ❌ 未建模 |

### 2.5 过程分析 (process_trace.py + process_analysis.py)

**功能**：事件推断（双路径）、AA/BB/AB/BA 交叉分析、5 种事件模式、transition_flags

#### 合格项

- **事件分类体系**：7 种事件类型基于三个正交维度（是否有抽卡 × 是否获得目标 × 是否触发保底）加兑换池独立分支，覆盖完备。

- **双路径等价性**：`_infer_from_draw_sequence` 和 `_infer_from_aggregate` 在核心判定逻辑上等价（draws、target_count、pity_count、skip/ignore 判定均一致）。

- **transition_flags 逐卡独立判断**：修正了旧版 `sum(counts) >= sum(requirements)` 的假阳性 bug，与 GDR SuccessChecker 逻辑一致。

- **AB/BA 互为贝叶斯关系**：`P(success|event) = P(event|success)·P(success)/P(event)`，两种交叉分析构成完整的预测-诊断框架。

#### 理论缺陷

**缺陷 15：资源池事件被跳过，破坏分析完整性（中等）**

两路径均 `if pool_type == '资源': continue`。资源池在过程分析中完全不可见——不产生事件条目、不影响模式计算。用户可能误认为资源池未被模拟。

**缺陷 16：skip/ignore 判定使用已获得卡而非池可掉落列表（中等）**

`pool_card_counts` 仅记录玩家在该池实际获得的卡。若玩家在含有目标卡的池子中 0 抽（应标记为 `skip`），`pool_card_counts[pool_id]` 为 `{}`，导致 `any(cid in target_ids for cid in {})` 返回 False，被错误标记为 `ignore`。两路径共享此缺陷。

**缺陷 17：条件概率估计无小样本保护（中等）**

`compute_ab` 中 `P(success|event_pattern)` 在事件模式仅出现 1-2 次时方差极大。`compute_ba` 中 `ratio = p_given_success / p_given_failure` 在分母为 0 时返回 `inf`。无拉普拉斯平滑、无置信区间、无最小样本量守卫。

| 事件序列分类 | 离散时间随机过程状态标注 | ✅ 合理 |
| 条件概率交叉分析 | 贝叶斯定理 | ✅ 正确 |
| 模式组合（raw/sequence/set/count_set）| 降维与符号化 | ✅ 合理 |

### 2.6 流式分析 (streaming.py)

**功能**：SharedResultCollector、StreamingAnalyzer、各 extractor

#### 合格项

- **StreamingSuccessCounter 无偏性**：样本均值是 Bernoulli(p) 的 MLE，达到 Cramér-Rao 下界。

- **extract_aggregate / extract_process**：纯确定性变换，逻辑正确。

- **SharedResultCollector 内存模型**：在当前单线程使用模式下安全（`on_result` 在父进程主循环中顺序调用）。

#### 理论缺陷

**缺陷 18：SharedResultCollector 缺少线程安全声明（低等）**

`n_results += 1` 非原子操作，`add_extractor` 与 `on_result` 之间存在竞态条件。当前单线程使用中安全，但类本身未声明此约束。未来若切换为多线程模式会产生难以调试的数据竞争。

**缺陷 19：热图资源分箱上限导致尾部聚集（低等）**

`max_resource = initial_resource * 2` 为分箱上限。每日资源获取可能使实际资源超出此值——所有超出的值被 clip 到最后一个分箱，导致该分箱异常偏高。

### 2.8 最差影响分析 (worst_impact.py)

**功能**：ConditionalResourceDistribution、虚拟池生成、连续池成功分布

#### 理论缺陷

**缺陷 20：保底引擎通配符匹配可能静默失败（严重）**

`_build_pity_engine` 使用 `fnmatch(pool_id, pity.pools_pattern)` 匹配虚拟池。若原始保底定义了特定池模式（如 `pools=event_banner`），虚拟池 ID `_worst_impact_pool_0` 不匹配，该保底被静默丢弃。模拟在**完全没有保底**的状态下运行——连续成功次数会被严重低估。

**缺陷 21：MAX_POOLS=99 硬截断（中等）**

成功次数分布在 99 处被截断，对于储蓄极高的玩家产生向下偏倚。`expected_pools` 估计被低估。

**缺陷 22：虚拟池同质化假设（中等）**

所有 99 个虚拟池共享相同的持续时间（21 天）、成本、奖励结构、概率分布。仅 featured 卡 ID 不同。实际游戏中这些参数随池变化，外推分析在参数变化时失效。

**缺陷 23：无资源收入假设（低等）**

`resource_gain=None` 意味着模拟假设玩家无后续收入，仅靠当前储蓄。这确保了最差情况下界，但绝对连续成功数被低估。

**缺陷 27：条件分布分位数 → 新模拟预算的逻辑跳跃（中等）**（2026-05-26 补充审查发现）

`WorstImpactAnalyzer.analyze()` 的流程为：(1) 从模拟结果中提取条件于失败（或全部）的资源分布 → (2) 取 α 分位数 q_α → (3) 以 q_α 为初始预算，从清空历史开始运行 99 个虚拟池的新模拟。

这等价于假设：

> E[未来连续成功池数 | 初始预算 = q_α, 清空历史] ≈ E[未来连续成功池数 | 剩余资源 = q_α, 坏运气历史]

但这两个期望不等价。原模拟中剩余 q_α 资源的失败路径经历了坏运气——保底计数器可能已接近软保底触发、兑换机会可能已消耗。新模拟从零开始——保底归零、兑换完整——条件比真实失败场景更**有利**。后果：可能**高估**连续成功池数。

**缺陷 28：虚拟池同质化消除「高成本池连续出现」的失败模式（中等）**（2026-05-26 补充审查深化缺陷 22）

除缺陷 22 已指出的成本/概率/持续时间同质化外，此假设还系统性地消除了一种关键失败模式：**连续高成本池的出现**。在真实游戏中，若连续两个池均为高成本（如 280 钻/抽）+ 低概率（如 0.3% 基础出率），玩家在有 N 资源预算时的实际连续成功概率远低于同质化假设下的估计。虚拟池同质化分析在池间方差越大的游戏经济体系中偏差越严重。

#### 文献对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| 条件资源分布 | 条件概率直接估计 | ✅ 正确 |
| 连续池成功 | 序贯决策随机过程 | ⚠️ 保底匹配可能缺失 |
| 虚拟池外推 | 无标准文献 | ⚠️ 同质化假设过强 |

### 2.9 退路搜索 (retreat_search.py)

**功能**：Pareto 前沿搜索（额外资源 vs 目标卡数量）

#### 理论缺陷

**缺陷 24：退路搜索的「Pareto 最优」定义不严格（严重）**

该算法搜索的不是真正的 Pareto 前沿，而是沿贪婪路径的子集近似：
- 真正的 Pareto 前沿需考虑 2^N 个卡子集，算法仅探索 N+1 个（全量 → 逐个移除）
- 不移除支配点（如卡更少且资源更多的点未被过滤）
- 本质是等概率水平集而非 Pareto 前沿

**缺陷 25：上界搜索可能失败（中等）**

从 50 抽开始每次翻倍，最坏情况 `50·2^10 = 51200` 抽。若实际需求超过此值，二分搜索在无效区间上运行，返回的点声称 `extra_resource = r_hi` 但 `success_probability < threshold`。

**缺陷 26：二分搜索在 p≈threshold 时因噪声振荡（中等）**

每步决策基于 500 次模拟的随机估计。在 `true_p ≈ threshold` 处，SE≈0.01，二分决策约 50% 概率出错。前沿点在不同运行间不稳定。

#### 文献对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| Pareto 前沿 | 多目标优化 (Miettinen, 1999) | ❌ 子集近似非严格前沿 |
| 二分搜索 | 标准数值方法 | ⚠️ 随机目标函数下不稳健 |
| 等概率水平集 | 条件概率密度水平集 | ✅ 语义匹配 |

## 三、未实施计划审查

### 3.1 P3：Bootstrap 稳定性分析

**方案**：不重新跑模拟，对已有 N 条结果做有放回抽样，BCa 置信区间，m-out-of-n 回退

#### 理论评估

**优点**：
- 文献基础扎实——BCa (Efron, 1987)、m-out-of-n (Bickel & Sakov, 2008)、Hill 估计量 (Hill, 1975)
- 零额外模拟成本（仅对已有结果重抽样）
- 四种方法覆盖轻尾到厚尾的递进覆盖

**关键缺失**：
- **m-out-of-n 的 m 选择策略**：m-out-of-n 方法最大的实践难点是子样本量 m 的选择。m 选太小增加方差，选太大保留偏差。文献中 m = n^k (k=0.7~0.9) 为经验范围，但需数据驱动选择。计划未讨论。
- **Hill 估计量的 k 选择**：Hill 估计量对顺序统计量个数 k 极其敏感。不同 k 可从 α=1.5 变到 α=5。无 k 选择策略，厚尾检测不可靠。
- **BCa 在离散 Bootstrap 分布下可能失效**：罕见事件（p~0.1%）的 Bootstrap 分布高度离散，BCa 的偏差校正依赖连续 CDF 反函数。

**综合评级**：★★★☆（框架正确，两个关键选择策略缺失）

---

### 3.2 P4：自适应模拟次数与方差缩减

**方案**：RSE 自适应停止、对偶变量法、GPD 尾部拟合 (EVT)

#### 理论评估

**核心问题：对偶变量法的种子互补方案不正确（严重）**

计划采用的 `seed_b = MAX_UINT32 - seed_a` 不能保证负相关。真正的对偶变量法要求在**单次模拟内部**对每对抽卡使用配对的均匀随机数 `(u, 1-u)`。种子互补产生的是两个独立随机流，其协方差为 0，方差缩减效果为零。

**其他问题**：
- **EVT 自动阈值选择的 R² 方法非标准**：EVT 文献中无此方法，统计性质未知。标准方法为 Hill 图、参数稳定性图。
- **多重 GDR 指标的停止偏误**：任一指标 RSE 达标即停止 → 报告的 RSE 被条件化于停止时刻，存在向下偏误（序贯分析中的 stopping bias）。
- **对偶变量不适用于分位数和稀有事件概率**：计划在 Tooltip 中提及分位数限制，但未讨论稀有事件场景。

**综合评级**：★★☆☆（EVT 部分可接受，对偶变量方案需彻底重设计）

### 3.3 P12：模拟性能优化

**方案**：算法优化 (Phase 1) → Numba 数值化内核 (Phase 2) → 共享内存并行 (Phase 3)

#### 理论评估

这是一个工程优化方案，统计理论风险低。需关注的技术要点：

- **Numba-Python 回调开销**：每次抽卡从 Numba 调用 Python 策略函数，边界穿越 ~50-100ns/次。对大规模模拟可能抵消 Numba 内核的加速效果。
- **随机数状态一致性**：Numba 内核和 Python 回退路径必须在相同 seed 下产生相同结果序列。
- **Worker extractor 合并的时序问题**：不同 Worker 独立维护的累积快照合并后不保留全局时间顺序，对依赖时序的分析（转变分析）产生影响。

**综合评级**：★★★☆（方案合理，工程风险可控）

### 3.4 P13：调试模式

**方案**：单步抽卡、状态快照、GDR 实时显示、策略决策日志

#### 理论评估

这是软件工程方案，不涉及统计推断。需关注：

- **GDR 中间值的误导性**：「每目标卡资源消耗」在获得第一张目标卡前为 inf，「所有目标达成率」在达成前始终为 0。用户可能基于中间值做出错误判断。需在 UI 中明确标注中间值的语义。
- **state 深拷贝性能**：每次单步执行需 deepcopy 完整模拟状态，在交互式步进场景中可能产生可感知的延迟。

**综合评级**：★★★★（工程方案完整，需补充 GDR 中间值语义文档）

## 五、方法论总结

### 5.1 项目整体理论水平

项目在**工程正确性**方面表现良好——算法实现基本正确、边界处理大部分到位、API 设计合理。但在**统计理论深化**方面存在系统性的薄弱环节：

1. **二元响应普遍使用线性模型**：脆弱性分析的 LLR、GDR 的加权满意度均使用线性/拟线性模型处理本质上的非线性/非高斯数据。局部逻辑回归和适当的效用建模是标准的改进方向。

2. **点估计缺乏不确定性量化**：过程分析的条件概率、退路搜索的 Pareto 前沿、最差影响的连续池分布均为点估计，无置信区间。P3 (Bootstrap) 计划旨在解决此问题但尚未实施。

3. **启发式决策缺乏形式化框架**：策略系统采用规则式启发式，这在工程上是合理选择，但需在文档中明确标注「非最优」。贝尔曼框架虽不实际实现，但可作为评估启发式质量的基准。

4. **独创指标缺乏外部校准**：GDR 系统的 17 个指标为项目独创，无外部文献或业界标准对照。对其「正确性」的判断只能基于内部一致性，而非外部验证。

### 5.2 与标准统计实践的差距

| 维度 | 标准实践 | 当前状态 |
|------|---------|---------|
| 二元响应建模 | 逻辑回归 / 局部似然 | 线性回归 / LLR |
| 带宽选择 | 交叉验证 / 插入法 | Silverman 规则 |
| 置信区间 | Bootstrap / 渐近正态 | 多数模块无 CI |
| 多重检验 | Bonferroni / FDR / 同时置信带 | 无校正 |
| 小样本保护 | 拉普拉斯平滑 / 贝叶斯收缩 | 无 |
| 效用建模 | 期望效用 / 前景理论 | 临时可加权重 |

### 5.3 对后续工作的建议

1. **优先修复第一级缺陷**（产生错误结果）：保底归一化、保底副作用、最差影响保底匹配、脆弱性 r_grid
2. **实施 P3 (Bootstrap) 前先确定 m 和 k 的选择策略**
3. **重新设计 P4 对偶变量实现方案**（单次模拟内配对随机数）
4. **为 GDR 系统补充公理化文档**（每个指标的定义、值域、假设、适用场景）
5. **为所有条件概率估计增加最小样本量守卫和平滑选项**
6. **在策略系统文档中明确标注启发式性质**
