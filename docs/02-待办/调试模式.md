# 调试模式

> 提炼自 `03-归档/plans/调试模式实施计划.md` | 2026-05-28 v2 | 未实施

## 问题

缺少一个能逐步调试抽卡逻辑的面板。当前只能批量模拟然后看聚合结果，无法观察单步抽卡的保底触发、GDR变化、策略决策等过程细节。

## 方案

新增「调试模式」Tab，鸣潮式布局：左侧卡池选择竖列 + 右侧展示区（上半区游戏界面 + 下半区调试信息面板），QScrollArea 单页滚动。

核心改造：在 `GachaService` 中抽取 `_execute_action()` 共享逻辑 → 新增 `prepare_debug_session()` + `run_simulation_step()` → 每步返回 `StepSnapshot`（含操作结果、GDR实时值、原始数据repr）。

## 四阶段

| 阶段 | 内容 | 估时 |
|------|------|------|
| Phase 1 | 核心单步逻辑——`_StepState`、`_execute_action()`、增量CompactResult、GDR实时计算、7种原始数据采集 | 1-2天 |
| Phase 2 | 上半区UI——卡池选择列、展示区（资源条/标题/保底/卡面/操作按钮）、抽卡结果2×5网格 | 2-3天 |
| Phase 3 | 下半区调试信息——调试选项栏、等待操作、池快照网格、操作日志、GDR全表17项、原始数据折叠卡片 | 2-3天 |
| Phase 4 | 集成——MainWindow新增Tab、ConfigStore接入、端到端测试 | 1天 |

## 依赖

- 无强制依赖
- P10（Worker全局变量消除）完成后可简化 `_execute_action()` 抽取，但非必须

## 风险

- 中低。`_execute_action()` 抽取引入微量方法调用开销（预期 < 0.5%）
- 十连抽采用 UI 层循环 + 锁池方案，不修改引擎
- 测试策略：N次单步执行后的最终状态与 `run_simulation()` 一次执行N步完全一致

## 归属模块

`panels/配置/` —— 调试模式是新Tab，当前无对应面板文件夹。
