## 一、问题概述

当前项目有 20+ 个活跃计划（P4-P37），计划文件的审查与完善完全依赖人工。一次完整的计划审查需要：（1）检查修改对其他面板/子系统的影响面；（2）逐维度检查计划的逻辑闭合、边界覆盖、向后兼容；（3）判断计划是否可实施（变更粒度、依赖顺序、回滚路径）；（4）将计划逐条映射到实际代码并做全链条逻辑审计；（5）同步模块状态矩阵。

人工执行上述审查耗时且容易遗漏。需要构建一个全自动的计划审查工作流，一键触发，完成所有审查阶段并直接改进计划文件。

### 1.1 核心需求

- 对任意计划文件（P 编号），一键运行全流程审查
- 审查即改进——发现问题后直接修改计划文件，不另行输出报告
- 解决 LLM 自我审查偏差——发现、修复、验证必须由不同 agent 执行
- 循环至收敛——不得因「看起来差不多了」而提前终止
- 审查完成后自动同步模块状态矩阵

## 二、方案：五阶段对抗验证流水线

### 2.1 总体架构

```
阶段0: CLASSIFY（分类）
  │  1 个 agent，判断计划复杂度 + 变更性质 + 动态生成维度清单 + skip_stages
  │  一个轻量 agent 在分类器前执行 Grep 扫描（搜索计划中函数/类在代码库的出现位置），作为维度生成的辅助信号
  │
阶段1: FAN-OUT IMPACT（扇出影响面）
  │  N 个 agent 并行检查 N 个影响面维度（N 由分类器动态决定）
  │  每个 agent 深耕一个子系统/面板，找出计划未提及的波及面
  │  输出：合并后的影响面发现列表
  │
阶段2: ADVERSARIAL LOOP（对抗循环）← 核心
  │  ┌──────────────────────────────────────────────┐
  │  │ 发现者 (Finder)  →  只发现，不修改            │
  │  │   输出含 relevant_code_snippet（附带原始代码） │
  │  │ 修复者 (Fixer)   →  只修改，不发现            │
  │  │   优先用片段验证 → 调用 Write/Edit 直接改文件    │
  │  │   返回确认信息（修改段落 + REVIEW-FIX 标注）                 │
  │  │ 验证者 (Verifier)→  只验证，默认拒绝          │
  │  │   Read 读修复后文件 → 判决 PASS/FAIL/PARTIAL    │
  │  │   ↑           ↓           ↓                   │
  │  │   └── 有 FAIL/PARTIAL ─────┘                  │
  │  │   └── 全 PASS + 连续2轮无新发现+无遗留 → 退出  │
  │  │   熔断器: 分类器设定的最大轮数上限             │
  │  └──────────────────────────────────────────────┘
  │
阶段3: FEASIBILITY GATE（可行性门控）
  │  1 个 agent，6 项深度检查（L1 存在性 + L2 有效性）
  │  ┌─────────────────────────────────────────────────┐
  │  │ PASS → 进入阶段 4（或 skip_stages 跳过）         │
  │  │ NEEDS_FIX → 【门控内部修复循环】                  │
  │  │   门控建议 → 修复者 → 验证者 → 重新门控           │
  │  │   （无 Finder 参与，gateFixAttempts 独立计数）     │
  │  │ FAIL → 【退回阶段 2】                             │
  │  │   完整 Finder→Fixer→Verifier 对抗循环             │
  │  │   （totalBacktracks++，受全局熔断约束）           │
  │  └─────────────────────────────────────────────────┘
  │
  【人工确认门】（仅 complex + breaking 计划触发）
  │  展示阶段1-3的审查摘要 → 等待用户输入 → 确认后进入阶段4
  │  用户可选择：确认进入 / 回阶段2指定方向 / 终止
  │
阶段4: CODE AUDIT（代码审计）
  │  4a: 代码映射——逐条修改映射到实际代码位置 + 当前逻辑 + 上下游影响
  │  4b: 全链条逻辑审计——从用户操作入口追踪到最终输出，检查类型/空值/信号断裂
  │  有阻塞断裂 → 退回阶段2（最多回 1 次）
  │
阶段5: MATRIX SYNC（矩阵同步）
  │  1 个 agent，对比计划元数据 ↔ 模块状态矩阵，自动更新不一致行
  │
  ▼
汇总输出 → 追加「自动化审查记录」章节到计划文件末尾

═══════════════════════════════════════

全局熔断器：totalBacktracks = 0（仅在 FAIL→退回阶段2 时递增，门控内部修复循环 NEEDS_FIX 不递增）
  上限 = classification.max_adversarial_rounds + 2
  门控内部修复循环独立计数 gateFixAttempts，上限 = classification.max_gate_retries
  超限 → 写入阻塞项章节，终止工作流
  这防止阶段3→2→3→2→4→2→3→2... 的无限交替循环
```

### 2.1.1 Workflow 平台约束与设计适配

本工作流运行在 Claude Code Workflow 沙箱中，存在以下关键约束：

| 约束 | 影响 | 适配方案 |
|------|------|---------|
| 脚本层**无文件 I/O** | 无法 `writeFile()` / `readFile()` / `appendFile()` | 所有文件操作委托给 agent（Fixer → Write/Edit，Verifier → Read） |
| 脚本层**无 Bash/Grep** | 阶段 0 的「轻量 Grep 扫描」无法在脚本层执行 | 改为一个轻量 agent 调用（`agent("扫描计划文件中提到的函数/类在代码库中的出现位置", {schema: GREP_SIGNAL_SCHEMA})`） |
| 脚本层**无 AskUserQuestion** | 人工确认门无法在 Workflow 内部阻塞等待 | 方案 A（推荐）：拆为两个工作流，人工确认在两次调用之间；方案 B：移除阻塞确认，改为日志警告 |
| agent 调用可能返回 `null` | agent 超时/跳过/API 错误 | 所有 `agent()` 返回值用 `.filter(Boolean)` 防护，单维度失败不阻塞其他维度 |

**脚本层职责重新定义**：
- ✅ 编排控制流（while 循环、收敛判定、熔断计数）
- ✅ 数据合并（`parallel` 结果 flat + filter）
- ✅ 状态追踪（`pendingUnresolved`、`dryRounds`、`totalBacktracks`）
- ❌ 文件读写 → 委托 agent
- ❌ 代码搜索 → 委托 agent
- ❌ 用户交互 → 拆分为多次工作流调用

### 2.2 设计原则（社区调研依据）

| 原则 | 来源 | 实现方式 |
|------|------|---------|
| **No Self-Review** | Giovanni/Forge/Codex-Claude-Cursor 三系统收敛 | 发现者、修复者、验证者三角色分离；门控器只诊断不修改 |
| **Gate, Don't Suggest** | Forge | 门控器 6 项全 PASS 才放行，输出 PASS/NEEDS_FIX/FAIL |
| **Adversarial Default** | Giovanni | 验证者默认立场是「找理由拒绝」，需看到确凿证据才 PASS |
| **Loop-Until-Dry** | Anthropic 动态工作流模式 | 脚本控制循环，连续 2 轮无新发现 + 无遗留问题 → 收敛 |
| **Artifact-Based Communication** | AgentMesh 学术框架 | Agent 间传递结构化 JSON（schema 约束），不用自然语言 |
| **Cross-Context Verification** | RepoAI（ScienceDirect 2026） | 发现与验证分属不同 agent 上下文，消除自我评估偏差 |
| **Circuit Breaker** | Forge/TAKT/ai-orchestrator | 局部（阶段2 maxRounds）+ 全局（totalBacktracks 上限）双层熔断 |
| **Re-plan, Don't Re-try** | AGENTS.md 社区最佳实践 | 熔断后写入阻塞项章节，交还人工决策 |
| **Human-in-the-Loop** | Giovanni/Forge | complex + breaking 计划在代码审计前暂停确认 |
| **Idempotency** | 本设计新增 | 重复运行时折叠历史记录，标注前缀隔离，支持续跑 |

### 2.3 两层审查分工

审查体系分为两层，互补不重叠：

| | 阶段1 — 影响面分析 | 阶段2 — 发现者审查 |
|---|---|---|
| **看什么** | 实际 `.py` 文件 | 计划 `.md` 文件 + 靶向代码 |
| **问什么** | 「这个修改会波及哪些代码？」 | 「计划已写的内容是否与代码事实一致？」 |
| **怎么读** | 扇出——N 个 agent 各深耕一个子系统 | 靶向——只读计划明确提到的文件/函数 |
| **审查维度** | 分类器动态生成（子系统 + 面板 + 动态） | 固定 8 维度（受 change_nature 调节） |
| **输出** | 计划未覆盖的波及面 | 事实错误 + 逻辑断裂 + 边界遗漏 + 文档缺失 + 根因深度 |

---

### 2.4 幂等性设计

工作流可能对同一计划文件多次运行。需处理以下情况：

**启动时检测**（由一个轻量 agent 在阶段 0 前执行）：
1. agent 读取计划文件，检测是否已有 `<!-- REVIEW-FIX: -->` 标注或「自动化审查记录」章节
2. 如果存在：
   - agent 将上一轮审查记录折叠到 `<details>` 区域，追加标题「第 N-1 次审查（YYYY-MM-DD）」
   - agent 将上一轮的 `<!-- REVIEW-FIX: -->` 标注重新编号为 `<!-- REVIEW-FIX-PREV: -->`，避免与新一轮混淆
   - 返回检测结果给脚本层，脚本层输出日志：「检测到第 N 次运行，上次审查记录已折叠」
3. 如果文件已有「⚠️ 自动化审查阻塞项」章节：
   - agent 检测阻塞项内容并返回给脚本层
   - **Workflow 限制**：脚本层无法交互询问用户，因此默认行为为「重新开始」（清理旧阻塞项章节）
   - 如需「从阻塞处继续」，用户需在触发工作流时传入 `resumeFromBlock: true` 参数

**多轮标注追踪**：每次审查使用独立的标注前缀（如 `<!-- REVIEW-R2-FIX: -->`），确保发现者和验证者能区分本轮修改与历史修改。

## 三、各阶段详细设计

### 3.1 阶段 0：分类器（Classify）

**职责**：判断计划复杂度 + 变更性质 + 动态生成维度清单。后续所有阶段的参数由分类器输出决定，脚本层不硬编码任何维度。

**分类器额外输入**：除计划文件全文外，一个轻量 agent 在分类器前执行 Grep 扫描——搜索计划中提到的所有函数/类/方法名在代码库中的出现位置（仅统计文件分布，不读取完整代码）。将此扫描结果作为分类器的辅助信号，帮助其判断：（1）计划涉及的实际代码范围是否超出计划文件所述；（2）是否需要新增维度覆盖未被计划提及但代码中引用了的函数/类。此举修正了「计划可能漏写涉及的 `.py` 文件而分类器无法察觉」的盲区。

**输出**：

```json
{
  "plan_identity": {
    "file": "P31 最差影响分析剩余缺陷改进计划.md",
    "module": "panels/最差影响/",
    "p_number": "P31",
    "priority": "P0"
  },
  "complexity": "simple" | "medium" | "complex",
  "change_nature": "evolutionary" | "breaking" | "greenfield",
  "dimensions": [
    { "type": "subsystem", "name": "GDR系统", "rationale": "..." },
    { "type": "panel", "name": "最差影响", "file": "gui/worst_impact_panel.py" },
    { "type": "dynamic", "name": "迁移路径与废弃警告", "rationale": "..." }
  ],
  "excluded": [
    { "name": "并行模拟", "rationale": "..." }
  ],
  "recommendation": {
    "enable_fan_out": true,
    "max_adversarial_rounds": 3,
    "max_gate_retries": 2
  },
  "skip_stages": []
}
```

**`skip_stages` 字段**（新增）：允许分类器标记无需运行的阶段。适用场景：

| 条件 | 跳过 | 理由 |
|------|------|------|
| `change_nature` = `greenfield` | `["code_audit"]` | 无现有代码可审计 |
| `complexity` = `simple` 且无 `.py` 文件在维度清单中 | `["fan_out_impact", "code_audit"]` | 纯文档/理论计划：影响面扇出成本高收益低（N 个 agent 扫描 N 个子系统，但计划不涉及代码）；代码审计无对象 |
| `complexity` = `simple` 且 `change_nature` = `evolutionary` | `["code_audit"]`（可选） | 小幅渐进式变更，代码审计收益低 |
| `complexity` = `simple` 且仅涉及 1 个面板 | `["fan_out_impact"]`（可选） | 单一面板变更影响面窄，阶段 2 发现者已覆盖靶向代码检查 |

脚本层在执行前检查 `skip_stages`：
- 含 `"fan_out_impact"` → 跳过阶段 1，直接进入阶段 2
- 含 `"code_audit"` → 阶段 3 后直接跳到阶段 5

**变更性质影响后续行为**：

| change_nature | 发现者兼容检查行为 |
|--------------|-------------------|
| evolutionary | 检查旧配置/旧缓存/旧API是否仍可用 |
| breaking | 不检查兼容，改为检查「是否提供迁移路径/废弃警告/版本号变更」 |
| greenfield | 完全跳过兼容维度 |

**维度生成规则**：
- 交叉子系统（始终包含）：数据流、配置系统、策略系统、保底系统、GDR系统、并行模拟、流式分析、图表引擎、分布估计、停止条件——但仅限计划触及的
- 面板维度：计划列出的每个 `.py` 面板文件 → 独立 agent
- 间接面板（计划未列但通过 MainWindow 信号连接）→ 合并为一个维度
- 动态维度：新子系统、新外部依赖、破坏性 API 变更、配置格式变更、性能变更、纯文档/理论计划
- 排除维度：明确列出与计划完全无关的默认维度

**脚本层合理性检查**（分类器输出后立即执行）：

分类器可能误判。脚本层在收到分类器输出后进行两项校验：

1. **复杂度校验**：统计计划文件的客观指标
   - 涉及 `.py` 文件数 ≥ 5 但 `complexity` = `simple` → 自动修正为 `medium`
   - 列出缺陷数 ≥ 6 但 `complexity` ≠ `complex` → 自动修正为 `complex`
   - 修正时输出警告日志：「分类器判定 {原值}，基于客观指标修正为 {新值}」

2. **维度清单校验**：
   - 计划列出的 `.py` 文件不在任何维度中 → 追加「遗漏面板」维度
   - 分类器排除的子系统如果在计划文件中被明确提及 → 恢复该维度

### 3.2 阶段 1：扇出式影响面分析（Fan-Out Impact）

**职责**：N 个 agent 并行，每个深耕一个维度，找出计划未提及的波及面。**只发现不修改。**

**提示词框架**（以「数据流影响」为例，其余维度结构相同、检查内容不同）：

```
你是影响面分析员，只负责检查【数据流影响】这一个维度。

你会收到：计划文件全文

你的任务：阅读计划中提出的修改方案，逐条判断它们是否会影响【数据流影响】维度。
不要修改计划文件，只输出发现。

检查清单：
1. 计划中的修改是否会影响 CompactResult 的字段定义？
2. 是否会影响 SimulationEnv 的结构？
3. 是否会影响 SharedResultCollector 的 extractor 逻辑？
4. 是否涉及序列化/反序列化（to_dict/from_dict）的变更？

输出 JSON：
{
  "dimension": "数据流影响",
  "findings": [
    {
      "id": "IMPACT-001",
      "severity": "高" | "中" | "低",
      "blocking": true/false,
      "description": "...",
      "affected_files": ["..."],
      "rationale": "..."
    }
  ],
  "no_findings": false
}
```

**脚本层合并逻辑**：
```javascript
const impactResults = await parallel(
  dimensions.map(d => () => agent(d.prompt, { schema: IMPACT_SCHEMA }))
)
const allFindings = impactResults.filter(Boolean).flatMap(r => r.findings)
```

### 3.3 阶段 2：对抗循环（Adversarial Loop）

**三个角色严格分离**：

| 角色 | 职责 | 立场 | 读写权限 |
|------|------|------|---------|
| 发现者 | 读计划 + 靶向代码，只输出问题列表 | 默认怀疑 | 只读计划文件 |
| 修复者 | 读问题列表 + 计划，修改计划文件 | 忠实执行 | 可写计划文件 |
| 验证者 | 读修改后计划 + 原始问题，逐项验证 | 默认拒绝 | 只读计划文件 |

#### 发现者（Finder）—— 8 个审查维度

1. **影响面追踪**：逐条检查阶段 1 的发现是否在计划中有响应
2. **事实验证**：计划提到的函数/类/方法是否真实存在？签名是否匹配？
3. **逻辑闭合**：A 步骤的输出类型和 B 步骤的输入类型是否匹配？修改顺序是否合理？
4. **边界覆盖**：空状态、极端值、用户中断、配置缺失
5. **兼容策略**：受 change_nature 调节
6. **文档同步**：是否标注了需要更新的文档文件和章节？
7. **前置依赖**：模块状态矩阵中对应计划的状态验证
8. **根因深度**：计划的修复方案是针对症状还是根因？评估症状修复/机制修复/架构修复

**靶向读代码指令**：
```
代码读取规则：
- 只读计划文件中明确列出的文件
- 对每个文件，只读计划明确提到的函数/类/方法
- 不扩展搜索——发现 Y 调用了 Z 但计划没提 Z，这是阶段1的工作
- 你的重点：计划说改 A，A 是否真的存在？计划的描述与实际签名是否一致？
```

**代码片段附带机制**：
发现者输出的每个问题必须附带 `relevant_code_snippet` 字段——将问题相关的原始代码片段（读到的函数签名、类定义、调用链等）直接附在问题中。修复者优先使用此片段验证问题真实性，避免重复读取同一代码文件。仅在片段缺失或不清晰时，修复者才自行打开代码文件。这消除了 Finder 和 Fixer 之间的代码版本竞争条件（两阶段之间代码可能被外部修改），并减少约 30% 的 token 消耗（避免 Fixer 重复读取 Finder 已读过的代码）。

#### 修复者（Fixer）

**输入**：问题列表（含 `relevant_code_snippet`） + 计划文件 + 涉及的代码文件（仅当片段缺失时靶向读取）

```
你是计划修复员。唯一职责：根据问题列表修改计划文件。

你会收到三份输入：
1. 发现者的问题列表（含每个问题的 `relevant_code_snippet`——发现者已读取的代码片段）
2. 当前计划文件全文
3. 涉及问题的代码文件（仅当 `relevant_code_snippet` 缺失或不清晰时才需打开验证）

处理规则：
- 优先使用 `relevant_code_snippet` 验证问题真实性，无需重新读取代码
- 仅在片段缺失、片段不完整、或怀疑片段已过时时，才自行打开对应代码文件
- 返回格式：{ "modified_content": "完整的修改后计划文件全文" }

修复模式受 change_nature 和 complexity 联合调节：
```

**修复模式参数**（来自分类器）：

| 模式 | 触发条件 | 修复规则 |
|------|---------|---------|
| conservative | simple + evolutionary | 最小改动——只改受影响的段落，保留章节结构，不删内容 |
| moderate | medium | 可重组段落、重写不清的表述，但保留章节标题 |
| aggressive | complex 或 breaking | 可重组章节结构、重写段落、添加新建议、调整实施优先级。仅不可改变核心架构决策（如方案A/B选择） |

**通用规则（所有模式）**：
1. 逐条处理，每条修复后用 `<!-- REVIEW-FIX: ISSUE-XXX -->` 标注
2. 冲突处理——两条问题的修复互相矛盾时选更保守方案，标注「⚠️ 待人工裁决」
3. 涉及代码事实的问题 → 打开对应代码文件验证后再改
4. 发现者建议的修复方向仅供参考，可选择更好的方案
```

#### 验证者（Verifier）

```
你是计划验证员。唯一职责：逐项验证修复是否到位。

对每个 ISSUE-XXX：
1. 定位计划文件中的 <!-- REVIEW-FIX: ISSUE-XXX --> 标注
2. 对照原始问题描述，判断修改是否真正解决了问题
3. PASS 需要提供证据（修改后的具体文本片段）
4. FAIL 需要解释为什么不行
5. PARTIAL 需要列出 resolved_items（已修复）和 unresolved_items（仍未修复）

PARTIAL 判决适用场景：
- 原问题包含多个子项（如"3个函数签名需要更新"），修复者只更新了2个
- 原问题含"A且B且C"，修复者解决了A和B但遗漏了C
- 修复方向正确但不完整

特别注意：
- 「过度修复」（改了不该改的）→ FAIL
- 「表面修复」（改措辞但逻辑问题仍在）→ FAIL
- 默认立场：不信任修复——需要确凿证据才给 PASS 或 PARTIAL
```

**脚本层 PARTIAL 处理**：

```javascript
// 验证后处理——PARTIAL 只保留未解决的子项到下一轮
pendingUnresolved = verifierResult.verifications
  .filter(v => v.verdict === 'FAIL')
  .map(v => v.issue_id)

const partialItems = verifierResult.verifications
  .filter(v => v.verdict === 'PARTIAL')
// PARTIAL: 保留 unresolved_items 到下一轮，避免已修复子项被重复处理
for (const p of partialItems) {
  pendingUnresolved.push({
    issue_id: p.issue_id,
    sub_items: p.unresolved_items  // 仅未修复的子项
  })
}
```

#### 脚本层循环控制

```javascript
let dryRounds = 0, totalRounds = 0
let pendingUnresolved = []  // 上一轮未解决的验证 FAIL/PARTIAL
let seenIssueKeys = new Set()  // 跨轮去重：已见过的 issue 摘要（维度::标题::涉及文件）
const maxRounds = classification.max_adversarial_rounds

while (dryRounds < 2 && totalRounds < maxRounds) {
  totalRounds++
  
  // 发现（注入上一轮未解决的问题 + 跨轮去重信号）
  const finderResult = await agent(FINDER_PROMPT, {
    schema: FINDER_SCHEMA,
    planFilePath: planFilePath,
    pendingUnresolved: pendingUnresolved,
    seenIssueKeys: [...seenIssueKeys]  // 跨轮去重信号
  })
  if (!finderResult) { log(`⚠️ 第 ${totalRounds} 轮 Finder 返回 null，跳过`); continue }
  
  // 跨轮去重：按「维度::标题::涉及文件」过滤已见等价问题
  const trulyNewIssues = finderResult.new_issues.filter(issue => {
    const key = `${issue.dimension || ''}::${issue.title || ''}::${(issue.affected_files || []).join(',')}`
    if (seenIssueKeys.has(key)) return false
    seenIssueKeys.add(key)
    return true
  })
  
  // 收敛判定：本轮无新发现 AND 无未解决的验证 FAIL
  const hasUnresolved = pendingUnresolved.length > 0
  if (trulyNewIssues.length === 0 && !hasUnresolved) {
    dryRounds++
    if (dryRounds >= 2) break  // 连续两轮既无新发现也无遗留 → 收敛
    continue
  }
  
  // 有新发现或有遗留 → 不能干燥
  dryRounds = 0
  
  // 合并本轮的修复任务（去重后新发现 + 上轮未解决子项）
  const issuesToFix = [
    ...trulyNewIssues,
    ...pendingUnresolved.map(item => ({
      issue_id: item.issue_id,
      sub_items: item.sub_items || null,
      source: 'previous_round_unresolved'
    }))
  ]
  
  // 修复（Fixer agent 直接调用 Write/Edit 工具修改计划文件）
  // 返回确认信息（modified_sections, applied_labels）
  const fixerResult = await agent(FIXER_PROMPT, {
    planFilePath: planFilePath,
    issues: issuesToFix,
    codeSnippets: finderResult.relevant_code_snippets,
    changeNature: classification.change_nature,
    complexity: classification.complexity
  }, { schema: FIXER_SCHEMA })
  if (!fixerResult) {
    log(`⚠️ 第 ${totalRounds} 轮 Fixer 返回 null——未解决问题保留到下一轮`)
    pendingUnresolved = issuesToFix.map(i => ({ issue_id: i.issue_id || i.id }))
    continue
  }
  log(`  第 ${totalRounds} 轮修复：${fixerResult.modified_sections?.length || 0} 处修改`)
  
  // 验证（Verifier agent 直接 Read 修复后的计划文件）
  const verifierResult = await agent(VERIFIER_PROMPT, {
    schema: VERIFIER_SCHEMA,
    planFilePath: planFilePath,
    originalIssues: issuesToFix
  })
  if (!verifierResult) {
    log(`⚠️ 第 ${totalRounds} 轮 Verifier 返回 null——假设全部未解决`)
    pendingUnresolved = issuesToFix.map(i => ({ issue_id: i.issue_id || i.id }))
    continue
  }
  
  // FAIL → 完整问题保留到下一轮
  // PARTIAL → 仅未解决的子项保留到下一轮（避免已修复子项重复处理）
  // PASS → 清除
  pendingUnresolved = [
    ...verifierResult.verifications
      .filter(v => v.verdict === 'FAIL')
      .map(v => ({ issue_id: v.issue_id })),
    ...verifierResult.verifications
      .filter(v => v.verdict === 'PARTIAL')
      .map(v => ({ issue_id: v.issue_id, sub_items: v.unresolved_items }))
  ]
  
  const passCount = verifierResult.verifications.filter(v => v.verdict === 'PASS').length
  const partialCount = verifierResult.verifications.filter(v => v.verdict === 'PARTIAL').length
  const failCount = verifierResult.verifications.filter(v => v.verdict === 'FAIL').length
  log(`  第 ${totalRounds} 轮验证: ${passCount}P ${partialCount}PART ${failCount}F → ${pendingUnresolved.length} 遗留`)
}

// 熔断检查
if (dryRounds < 2 && totalRounds >= maxRounds) {
  appendCircuitBreakerSection(planFilePath, {
    rounds: totalRounds,
    unresolved: pendingUnresolved,
    reason: `已执行 ${maxRounds} 轮对抗循环仍未收敛，${pendingUnresolved.length} 个问题未解决`
  })
}
```

### 3.4 阶段 3：可行性门控（Feasibility Gate）

**职责**：6 项深度检查，每项含 L1 存在性 + L2 有效性判定。全 PASS 才放行。

**门控器只做诊断，不直接修改计划文件**——这遵循 No Self-Review 原则。如需拆分任务或补充细节，门控器输出修改建议，由修复者执行，验证者验证。

| # | 检查项 | L1 存在性 | L2 深度判定 |
|---|--------|:-------:|:---------:|
| 1 | 变更粒度 | 每个任务 ≤ 1h | 任务过大时输出拆分建议（split_suggestions）；描述模糊时输出补充建议（clarify_suggestions）。不直接修改计划 |
| 2 | 文件清单 | 每个步骤标注文件+函数 | 随机抽样 3 个文件验证代码一致性 |
| 3 | 依赖顺序 | 无环 | 输入/输出类型匹配 |
| 4 | 风险缓解 | 阻塞项全响应 | 缓解措施可操作性 + 严重度匹配 |
| 5 | 回滚路径 | 有回滚策略 | 每步是否可独立回滚 |
| 6 | 测试策略 | 每步有验收方式 | 验收方式含具体输入/期望输出 + 覆盖边界 |

**输出**：
```json
{
  "checks": [
    {
      "id": "变更粒度",
      "verdict": "PASS" | "NEEDS_SPLIT" | "NEEDS_CLARIFY" | "FAIL",
      "detail": "...",
      "split_suggestions": [  // NEEDS_SPLIT 时提供
        { "original_task": "...", "suggested_splits": ["子任务1", "子任务2"] }
      ],
      "clarify_suggestions": [  // NEEDS_CLARIFY 时提供
        { "original_task": "...", "suggested_clarification": "..." }
      ]
    }
  ],
  "overall": "PASS" | "NEEDS_FIX" | "FAIL",
  "failures_blocking": []
}
```

**处理流程**：

```
门控器输出 overall 判定
  │
  ├── PASS → 进入阶段 4（或 skip_stages 含 "code_audit" 则跳阶段 5）
  │
  ├── NEEDS_FIX（含 NEEDS_SPLIT / NEEDS_CLARIFY）
  │     │  门控器已诊断出具体修改方案（split_suggestions / clarify_suggestions）
  │     │  走「门控内部修复循环」——不退回阶段 2，无 Finder 参与：
  │     │
  │     │  门控建议 → 注入修复者 → 修复者执行修改 → 验证者验证修改
  │     │       ↑                                              ↓
  │     │       └──────────── 修改未通过 ←─────────────────────┘
  │     │       └──────────── 修改通过 → 重新门控（检查修复后是否满足门控标准）
  │     │
  │     │  此循环内部计数 `gateFixAttempts`，上限 = classification.max_gate_retries
  │     │  超限 → 将未解决的 NEEDS_FIX 升级为 FAIL → 走 FAIL 路径
  │     │  注意：门控内部修复循环不递增 `totalBacktracks`
  │     │
  │     └── 重新门控再次返回 NEEDS_FIX 且 gateFixAttempts 未超限 → 继续循环
  │
  └── FAIL
        │  门控器无法自行诊断出修改方案（如：计划整体结构需要重组、
        │  核心架构决策存在矛盾、多个修改建议互相冲突无法裁定）
        │  走「退回阶段 2」——完整 Finder→Fixer→Verifier 对抗循环：
        │
        │  门控 FAIL 项 → 转为发现者问题（注入下一轮 Finder 的 pendingUnresolved）
        │       → 退回阶段 2（完整对抗循环）
        │       → 对抗循环收敛后 → 重新进入阶段 3 门控
        │
        │  totalBacktracks++（全局退回计数器递增）
        │  超限 → 写入「自动化审查阻塞项」章节，工作流终止
```

**两种路径的区分要点**：

| | 门控内部修复循环 | 退回阶段 2 |
|---|---|---|
| 触发条件 | `overall` = `NEEDS_FIX` | `overall` = `FAIL` |
| 参与者 | 修复者 + 验证者 | 发现者 + 修复者 + 验证者（完整三角色） |
| Finder 是否参与 | 否（门控器已诊断） | 是（需重新发现问题范围） |
| 计数变量 | `gateFixAttempts`（独立） | `totalBacktracks`（全局） |
| 上限来源 | `max_gate_retries` | `globalBacktrackLimit` |
| Token 成本 | 低（2 agent/轮） | 高（3-5 agent/轮 × N 轮） |
| 适用场景 | 门控器能说出"具体缺什么" | 门控器只能说"这不行"但说不清为什么 |

**人工确认门（Human Gate）**：

> ⚠️ **Workflow 平台限制**：Workflow 脚本无法调用 `AskUserQuestion` 或阻塞等待用户输入。原设计中「脚本暂停 → 展示摘要 → 等待用户选择」无法在单个 Workflow 中实现。

触发条件：分类器输出 `complexity` = `complex` 且 `change_nature` = `breaking`。

**替代方案 A（推荐）：拆分为两个工作流**

将原有单次调用拆为两次：
```
工作流 1: plan-review-phase0-3
  阶段0 → 阶段1 → 阶段2 → 阶段3
  → 阶段 3 PASS 后，输出审查摘要 JSON（保存到 .claude/plan_review_state.json）
  → 若为 complex+breaking 计划，工作流 1 在此终止，等待用户确认

【用户在两次调用之间查看摘要，做出决策】

工作流 2: plan-review-phase4-5
  读取 .claude/plan_review_state.json
  → 阶段 4（代码审计）
  → 阶段 5（矩阵同步）
  → 追加审查记录
```

实现细节：
- `plan-review-phase0-3` 在阶段 3 PASS 后，检查 `classification.complexity === 'complex' && classification.change_nature === 'breaking'`
  - 若是：生成审查摘要，调用一个 agent 将摘要写入 `.claude/plan_review_state.json`，然后终止（不报错）
  - 若否：直接进入阶段 4-5（无需人工确认）
- 用户手动触发 `plan-review-phase4-5` 继续，或通过 `/loop` 在确认后自动触发

**替代方案 B：移除阻塞确认，改为日志警告**

- complex+breaking 计划在阶段 3 后不暂停，直接输出日志：「⚠️ 此为 complex+breaking 计划，建议人工复核后再继续代码审计」
- 照常进入阶段 4
- 优点：简单，无需拆分工作流
- 缺点：违背 Human-in-the-Loop 设计原则（Giovanni/Forge）

**替代方案 C：用 Skill 而非 Workflow 实现**

- Skill 可以调用 `AskUserQuestion` 实现真正的交互确认
- 代价：失去 Workflow 的 resumeFromRunId、结构化 schema 强制、确定性重放等特性
- 可作为长期方案，但 v1.0 优先用方案 A

**v1.0 实际采用方案 B（日志警告）**。理由：单工作流简洁性 + token 成本可控 + C2 matrix-syncer 提供二次检查。complex+breaking 计划在阶段 3 通过后输出警告日志，照常进入阶段 4。人工确认可通过 `/workflow plan-review` 两次调用实现（第一次 phase0-3 → 检查输出 → 第二次 resume）。方案 A 的双工作流拆分留作未来可选增强。

### 3.5 阶段 4：代码审计（Code Audit）

**4a：代码映射**

逐条映射格式：
```
修改条目 #N
  - 计划描述：从「实施路线」摘录原文
  - 代码位置：文件路径:行号范围
  - 当前逻辑：读代码后写（3-5 句话），不猜测
  - 计划要求：具体改什么
  - 改法说明：基于当前代码的改动方式
  - 上下游影响：谁调用它 / 它调用谁 / 数据类型变化
  - 改动风险：行数 / 是否破坏接口 / 级联改动
```

**冲突检测**：逐对检查修改条目间是否存在矛盾（A 改签名，B 以旧签名调用）。

**传递缺口**：检测计划已提修改之间的传递性依赖——A 改了，A 的调用方 B 也需要改，但计划没提 B。

**4b：全链条逻辑审计**

从用户操作入口开始，沿数据流逐环节模拟完整执行路径。在每个修改点停下来验证：
- 类型断裂：上游输出类型 X，下游期望类型 Y，无转换
- 空值断裂：上游可能返回 None，下游直接调 .attr
- 权限断裂：子线程尝试访问 GUI 控件
- 信号断裂：信号签名变化但槽函数未同步
- 文件断裂：路径逻辑与项目路径系统不一致

**审计后处理**：

有阻塞断裂 → 检查全局熔断器：

```javascript
let totalBacktracks = 0  // 全局退回计数器（阶段3/4 → 阶段2）
const globalBacktrackLimit = classification.max_adversarial_rounds + 2

if (auditResult.blocking_breaks > 0) {
  totalBacktracks++
  if (totalBacktracks > globalBacktrackLimit) {
    // 委托 agent 将阻塞项写入计划文件的「⚠️ 自动化审查阻塞项」章节
    await agent(BLOCK_WRITER_PROMPT, {
      planFilePath: planFilePath,
      reason: `全局熔断：${totalBacktracks} 次退回阶段2，超过上限 ${globalBacktrackLimit}`,
      unresolved: auditResult.execution_path.flatMap(s => s.breaks)
    })
    log(`🛑 全局熔断触发：${totalBacktracks}/${globalBacktrackLimit} 次退回 → 阻塞项已写入计划文件`)
    return  // 终止工作流
  }
  // 将断裂点转为发现者问题 → 退回阶段 2（完整对抗循环）
  log(`🔄 代码审计发现 ${auditResult.blocking_breaks} 处阻塞断裂 → 退回阶段 2`)
}
```

阶段 3 退回时同样递增 `totalBacktracks` 并检查全局上限。

### 3.6 阶段 5：矩阵同步（Matrix Sync）

从计划文件提取元数据（P 编号、状态、涉及文件、一句话描述、优先级），与 `docs/00-meta/模块状态矩阵.md` 逐字段对比，不一致则更新。额外检查前置计划的状态标记是否滞后。

## 五、测试策略

| 测试场景 | 预期行为 |
|---------|---------|
| simple/evolutionary 计划（如 P30 文档补充） | 维度清单短（3-5 维），maxRounds=2，保守修复模式 |
| medium/breaking 计划（如 P33 GDR 重构） | 含迁移路径检查，maxRounds=3，中等修复模式 |
| complex/breaking 计划（如 P26 21项缺陷） | 全开，maxRounds=4，扇出 10+ agent，激进修复模式 |
| 阶段 1 发现阻塞项 → 计划未响应 | 阶段 2 发现者标记为 ISSUE → 修复者加入计划 |
| 发现者连续 2 轮无新发现 + 无未解决遗留 | 收敛退出 |
| 验证者 FAIL → 未解决问题保留到下一轮 | pendingUnresolved 累积，修复者合并处理新发现+遗留 |
| 分类器误判 complexity（涉及5+文件判为simple） | 脚本层自动修正 + 输出警告日志 |
| 分类器遗漏计划中列出的面板 | 脚本层追加「遗漏面板」维度 |
| 门控 NEEDS_FIX → 修复者执行 → 验证者验证 | 修复→验证→重新门控，不退回阶段2 |
| 门控 FAIL → 退回阶段 2 | 转为发现者问题，完整对抗循环 |
| 阶段 3→2→3 超过 maxGateRetries | 写入阻塞项章节，工作流终止 |
| 4b 发现阻塞断裂 → 全局熔断检查 | totalBacktracks++，超限则终止；未超限则退回阶段2 |
| 阶段 3→2→3→4→2→3 交替退回超 totalBacktracks 上限 | 全局熔断触发 → 写入阻塞项章节 |
| complex+breaking 计划门控通过 | 暂停展示审查摘要 → 等待用户确认 |
| 用户确认「继续」→ 进入阶段 4 | 人类确认后继续 |
| 用户选择「回退+指定方向」 | 用户指定方向转为发现者问题 → 退回阶段2 |
| 第二次运行同一计划 | 历史审查记录折叠，标注前缀隔离，提示继续或重新开始 |
| 计划标记为「已完成」 | 矩阵同步自动更新状态 |
| 不传参数调用工作流 | 自动读取 `00-本周聚焦.md` 定位计划文件 |
| 验证者 PARTIAL → 仅未解决子项保留下一轮 | `unresolved_items` 注入 Finder，已修复子项不重复处理 |
| 纯文档计划（simple + greenfield，无 .py 涉及） | `skip_stages` = `["code_audit"]`，阶段 3 后直接跳阶段 5 |
| 门控 NEEDS_FIX → 门控内部修复循环 | 修复者→验证者→重新门控，gateFixAttempts++，不递增 totalBacktracks |
| 门控内部修复循环超 maxGateRetries | NEEDS_FIX 升级为 FAIL → 退回阶段 2，totalBacktracks++ |
| 发现者附带 relevant_code_snippet → 修复者验证 | 修复者优先用片段验证，仅在缺失时读代码文件 |
| Finder agent 返回 null（超时/跳过/API 错误） | 脚本层 null-guard → 日志警告 → 跳过本轮发现 → continue，不阻塞收敛 |
| Fixer agent 返回 null | 日志警告 → 未解决问题全部保留到 pendingUnresolved → continue |
| Verifier agent 返回 null | 日志警告 → 假设全部未解决 → 保留到 pendingUnresolved → continue |
| 阶段 1 单个维度 agent 返回 null | 该维度结果被 `.filter(Boolean)` 丢弃 → 不影响其他维度 → 日志记录缺失维度 |
| Finder 跨轮发现语义相同但表述不同的等价问题 | `seenIssueKeys` 去重：按「维度::标题::涉及文件」过滤 → 等价问题不重复处理 |
| 分类器 Grep 信号 agent 返回 null | 脚本层以空信号传给分类器 → 分类器降级为仅基于计划文件全文判断 |
| Fixer agent 修改文件时遇到权限/锁定错误 | Fixer 返回 `write_success: false` + 错误描述 → 问题保留到下一轮 |
| simple 计划触发 fan_out_impact 跳过 | `skip_stages` 含 `"fan_out_impact"` → 阶段 1 跳过 → 直接进入阶段 2 |

## 七、风险

| 风险 | 等级 | 缓解 |
|------|:----:|------|
| Token 消耗大 | 中 | 分级估算：simple ~100-150K / medium ~200-300K / complex ~350-500K+（含退回循环可能翻倍）。缓解：阶段 0 复杂度分级——简单计划走精简路径（跳过阶段 4）；`resumeFromRunId` 断点续跑；token 预算不足时的降级策略：减少 maxRounds、跳过代码审计、仅运行阶段 0-2 |
| Agent 幻觉——发现者报告不存在的代码问题 | 中 | 验证者逐项 PASS/FAIL + 证据要求；发现者被指令「读代码，不猜测」；`relevant_code_snippet` 附带机制让修复者可交叉验证 |
| 熔断器误触发——正常修复需要超过 maxRounds 轮 | 低 | 分类器动态设定 maxRounds；熔断后人工介入 |
| 工作流脚本维护——Claude Code API 变更 | 低 | 脚本使用标准 API（agent/parallel/pipeline），兼容性由平台保证 |
| Workflow 沙箱无文件 I/O | 中 | 脚本层无法直接读写文件；已改为 agent 端工具调用（Write/Edit/Read）。缓解：阶段 0 前增加 agent 端能力验证——确认 agent 可成功 Read/Write 目标文件 |
| Workflow 无用户交互 API | 中 | 人工确认门无法阻塞等待；已改为方案 A——拆分为两个工作流 + 状态文件。缓解：`plan-review-phase0-3` 在需确认时正常终止（非报错），用户手动触发后续工作流 |
| 8 个 Schema 未在计划中定义 | 高 | Schema 设计是成败关键——太宽松 agent 输出噪声大，太严格 agent 频繁拒绝。缓解：v0.1 前先完成 3 个核心 Schema 草稿（FINDER/VERIFIER/GATE）；后续版本逐步补充其余 |

#### ISSUE-001: Workflow 脚本层无文件 I/O 权限 🔴 阻塞

**发现**：计划在多处假设脚本层可调用 `writeFile()`、`readFile()`、`appendCircuitBreakerSection()`。Workflow 脚本在沙箱中运行，无 Node.js API 和文件系统访问。

**修复**（`<!-- REVIEW-FIX: ISSUE-001 -->`）：

- 阶段 0 Grep 扫描：改为轻量 agent 调用
- 阶段 4 熔断器：`appendCircuitBreakerSection()` → 委托 agent（BLOCK_WRITER_PROMPT）写入

#### ISSUE-002: 人工确认门无法在 Workflow 中实现 🔴 阻塞

**发现**：Workflow 无 `AskUserQuestion` 等效 API，无法阻塞等待用户输入。原设计「脚本暂停 → 展示摘要 → 等待用户选择」不可行。

**修复**（`<!-- REVIEW-FIX: ISSUE-002 -->`）：

#### ISSUE-003: 8 个 Schema 未设计 🟡 风险

**发现**：计划列出了 IMPACT_SCHEMA / FINDER_SCHEMA / FIXER_SCHEMA / VERIFIER_SCHEMA / GATE_SCHEMA / CODE_MAP_SCHEMA / AUDIT_SCHEMA / MATRIX_SYNC_SCHEMA 共 8 个 Schema，但无一给出具体定义。Schema 设计是成败关键。
