## 一、问题诊断

### 1.1 当前 `docs/` 结构（60+ .md 文件）

```
docs/
├── archive/                          # 15 个已归档计划
├── reports/                          # 8 个调查报告
├── superpowers/
│   ├── plans/archive/                # 14 个已归档 superpowers 计划
│   └── specs/archive/                # 7 个已归档 superpowers 设计文档
├── *.md                              # ~18 个活跃计划 + 元文件（顶级散落）
├── 计划汇总.md                       # 54KB 全局汇总
├── 搁置计划记录.md                   # 8 个搁置条目
├── 优先级执行路线图.md               # 行动清单
├── 手动验收项目清单.md               # 验收清单
├── HANDOVER.md                       # 交接文档
├── implementation_plan.md            # 理论架构总图
└── 面板聚合项目文档架构提示词.md      # 参考提示词
```

### 1.2 核心痛点

1. **两套归档并存**：`docs/archive/`（15文件）+ `docs/superpowers/*/archive/`（21文件）
2. **顶级混杂**：18 个活跃计划文件 + 元文件全在 `docs/` 顶级，无法快速区分
3. **`计划汇总.md` 54KB**：每次新增/更新计划都要改，维护成本高
4. **全局视图唯一**：删掉计划汇总就没索引，保留就维护负担大
5. **搁置条目孤岛**：`搁置计划记录.md` 独立于其归属模块
6. **每个模块的基本信息、设计意图、理论假设分散**，每次对话都要重新理解

---

## 二、设计原则

1. **稳态用六文件，变化用主题计划**——模块文档描述当下的状态，改进计划保留为完整的主题文件
2. **按模块聚合**——每个面板/子系统的文档物理上放在一个文件夹
3. **单一入口**——`00-本周聚焦.md` 是每周唯一的起点
4. **全局视图精简为索引**——`模块状态矩阵.md`（2-3KB）替代 54KB 计划汇总，只保留 P编号、一句话、文件位置
5. **搁置条目独立但链接回模块**——`搁置计划记录.md` 保留，但每项标注归属模块
6. **理论独立成文件**——`01-理论.md`（假设/收敛性/局限/推导），不塞在档案里

### 六文件 vs 主题计划的角色分工

| | 六文件 | 主题计划文件 |
|---|---|---|
| **描述什么** | 稳态——模块**当前是什么样** | 变化——模块**要改成什么样** |
| **更新频率** | 实现变更时更新 | 计划推进时增量更新 |
| **谁读** | 初次接触模块时 | 执行具体改进时 |
| **跨模块时** | 各模块在相关文件中交叉引用 | **不拆散**，保留在主要归属模块 |

## 五、六文件工作流

六文件中 `02-实施.md`、`03-审计.md`、`04-问题.md`、`05-笔记.md` 是动态文件，需要明确更新时机和规则。

### 5.1 各文件更新频率

| 文件 | 更新频率 | 触发条件 |
|------|---------|---------|
| `00-档案.md` | 极低 | 设计意图变更、新功能改变模块定位 |
| `01-理论.md` | 低 | 新理论发现、审查发现新缺陷、外部研究补充 |
| `02-实施.md` | 中 | 代码重构、新功能上线、信息流变更 |
| `03-审计.md` | 低 | 首次建档时创建、重大重构后重新审计 |
| `04-问题.md` | **高** | 每次会话结束后更新、每周清理 |
| `05-笔记.md` | 高→清零 | 每次会话追加、**每周清理**（有用的迁移，没用的删除） |

### 5.2 标准工作流

**A. 开始处理某个模块时：**
1. 读 `00-本周聚焦.md` → 确认当前模块
2. 读模块 `04-问题.md` → 找到当前 P0
3. 读模块 `00-档案.md` → 理解设计意图
4. 读模块 `01-理论.md` → 了解假设和局限
5. 执行任务

**B. 完成任务后（Claude 在每个会话结束时应主动执行）：**
1. 在 `04-问题.md` 中标记已完成的 P 项
2. 如代码变更 → 更新 `02-实施.md`
3. 如发现新理论见解 → 追加到 `01-理论.md` §已知局限性
4. 本次对话关键结论 → 追加到 `05-笔记.md`（带日期）

**C. 新建模块时：**
1. 创建六文件骨架
2. `00-档案.md` 从现有材料提取
3. `01-理论.md` 从审查报告和外部研究中提取
4. `02-实施.md` 标注当前代码位置
5. `04-问题.md` 从 `计划汇总.md` 拆入对应 P 条目
6. `03-审计.md` 新建时填写「首次建档，未审计」
7. 在 `00-本周聚焦.md` 和 `模块状态矩阵.md` 注册

**D. 计划完成归档时：**
1. 主题计划文件移入 `99-历史/`
2. `04-问题.md` 中对应条目标记完成
3. `02-实施.md` 反映新实现状态
4. 如整个模块完成 → 整包移入 `03-归档/`

**E. 每周清理（Claude 在用户要求时执行）：**
1. 清理各模块 `05-笔记.md`：有用的迁移到 `01-理论.md` 或 `04-问题.md`，没用的删除
2. 清理 `04-收件箱/`：归类或删除
3. 更新 `00-本周聚焦.md`
4. 更新 `模块状态矩阵.md`（P 计数、状态）
5. 检查 `搁置计划记录.md` 中是否有可重新激活的条目

### 5.3 CLAUDE.md 需新增的工作流指令

在 CLAUDE.md 的「文档与计划约定」节后追加：

```markdown
### 文档维护工作流

- **每次会话结束时**：将关键结论写入当前模块的 `05-笔记.md`（带日期）；如有代码变更则更新 `02-实施.md`
- **新建模块时**：创建六文件骨架 → 从现有材料填充 `00-档案.md` + `01-理论.md` + `02-实施.md` → 从审查报告拆入 `04-问题.md`
- **计划完成时**：主题计划 → `99-历史/`；更新 `02-实施.md` + `04-问题.md` + `模块状态矩阵.md`
- **每周清理**：清理 `05-笔记.md` + `04-收件箱/`；更新 `00-本周聚焦.md` + `模块状态矩阵.md`
```

### 7.4 reports 智能拆分方案

审查报告不是按模块均匀分布的——有些缺陷跨模块、有些方法论论述不适合拆分。拆分原则：

- **按缺陷逐条映射**，不按章节整块切
- **跨模块缺陷**在主要归属模块记录，受影响模块在 `01-理论.md` §已知局限性 中写一行交叉引用
- **方法论/元论述**（如审查方法、评分标准）保留在原件，不拆分
- **原件保留**在 `05-参考资料/理论审查/`，作为不可篡改的历史记录

### 7.5 主题计划文件改写规则

当前每个计划文件包含多种类型的内容。迁移时需要提取而非整块搬运：

**计划文件的内容层次（以 `Bootstrap稳定性分析改进计划.md` 为例）：**

| 内容类型 | 示例 | → 目标 |
|---------|------|--------|
| **背景/动机** | "Bootstrap 为所有统计估计添加置信区间" | `00-档案.md` §1.业务问题 或 §2.设计意图 |
| **理论推导** | BCa 加速因子公式、GPD 尾部拟合原理 | `01-理论.md` §4.数学推导 |
| **现状分析** | "当前 Jackknife 上限 1000 截断" | `02-实施.md` §4.与档案意图的偏差 |
| **已知局限** | "BCa 在小 n 时不稳定" | `01-理论.md` §3.已知局限性 |
| **实施步骤** | "Phase 1: Jackknife 随机抽样" | **保留在主题计划文件** |
| **历史讨论** | 方案演变、决策记录 | **保留在主题计划文件** |

**改写原则：**

1. **灵活优先，不教条套用**——上表仅为理想化分类。每个计划文件结构不同，实际操作时以文件自身内容为准，不强行套用模板
2. **对照代码当前状态**——`02-实施.md` 必须基于实际 `.py` 文件编写，不能照搬旧计划中可能已过时的现状描述。旧计划中的「现状」可能是撰写时的快照，需重新核实
3. **主题计划文件保留原名**，只提取稳态内容到六文件
4. 六文件是**摘要**，主题计划是**全文**——不重复，六文件中写一行引用
5. 已完成的计划（如 P16/P17），提取理论教训到 `01-理论.md`，其余归档
6. 已修复的缺陷在 `01-理论.md` 中标注 `（已修复：YYYY-MM-DD，见 99-历史/xxx）`
7. **结构不匹配时，只移动不提取**——如某个计划文件几乎全部是实施步骤（无理论/现状分析），则整文件归入模块文件夹，不强行拆分
8. **不确定时保守处理**——宁可不提取、保留原文件，也不编造或使用过期信息

**改写示例——`Bootstrap稳定性分析改进计划.md`：**

| 提取内容 | → 目标 | 具体 |
|---------|--------|------|
| BCa/GPD/Hill 理论 | `01-理论.md` §4 | 公式 + 文献引用 |
| 6 项已知问题 | `01-理论.md` §3 | 当前局限（未修复前）/ 已修复标记（修复后） |
| 当前代码状态 | `02-实施.md` §4 | "Jackknife 上限 1000 截断"等 |
| 实施步骤 | 主题计划文件 §四-§六 | 保留不动 |
| 历史讨论 | 主题计划文件 | 保留不动 |
