## 一、死代码（可安全删除）

### R1：旧 history 路径函数（大部分无人调用）

`gdr.py` 中定义了 13 个基于 `history: List[InfoVector]` 的旧 GDR 函数，它们被注册为 `GDRDefinition.compute_from_history`。但实际调用路径全部走 `compute_from_compact`——`compute_from_history` 从未通过注册表被调用。

**例外**（仍有直接调用方）：
| 函数 | 调用方 | 
|------|--------|
| `resource_remaining()` | `core/distribution.py` L274, `core/risk_analysis.py` L16 |
| `all_targets_obtained()` | `core/risk_analysis.py` L50–51 |
| `_count_draws()` | `core/per_pool_analysis.py` L4 |
| `_total_resource_consumed()` | `core/per_pool_analysis.py` L4 |
| `ssr_collection_rate()` | 无（仅在注册表中） |

**其余 8 个函数**（`simple_target_achievement_rate`、`weighted_satisfaction`、`extra_target_cards`、`weighted_extra_cards`、`regret_index`、`total_card_value`、`non_pity_draws`、`pity_draws`、`resource_efficiency`、`per_pool_draw_rate`、`weapon_character_ratio`、`target_collection_rate`、`weighted_collection_rate`、`card_collection_rate`）——**无人调用**。它们只在以下位置出现：
- 自身定义
- `GDR_REGISTRY` 注册
- `compute_all_gdr` / `compute_weighted_gdr` / `compute_custom_weighted_gdr` 内部（这些函数本身也是死代码——见 R4）

**建议**：不删除（保留作为 `compute_from_history` 的备选实现），但标注为"历史路径，未激活"。

### R2：`WeightedGDRConfig` —— 完全无人使用

- 定义：`gdr.py` L43–50
- 导入：无任何文件导入
- 调用：零

**建议**：直接删除。

### R3：`SuccessChecker.from_registry()` —— 完全无人调用

- 定义：`gdr.py` L796–799
- 调用：零

```python
@classmethod
def from_registry(cls, gdr_key, target_specs, gdr_threshold=None, **kwargs):
    return cls(target_specs=target_specs, gdr_key=gdr_key,
               gdr_threshold=gdr_threshold, **kwargs)
```

这只是一个转调 `__init__` 的工厂方法——无额外逻辑。**建议**：删除。

### R4：`compute_all_gdr` / `compute_weighted_gdr` / `compute_custom_weighted_gdr` / `compute_collection_rates` —— 无人调用

这 4 个函数是旧 history 路径的批量计算入口。定义在 `gdr.py` L644–713，没有任何外部调用方。

它们依赖 `GDR_REGISTRY`（见 R5）和 `WeightedGDRConfig`（见 R2）。

**建议**：删除。

### R5：`GDR_REGISTRY` / `COMPACT_GDR_REGISTRY` —— 仅被死代码消费

- `GDR_REGISTRY`：由 `_build_legacy_registries()` 构建，仅被 `compute_all_gdr`（R4 死代码）使用
- `COMPACT_GDR_REGISTRY`：同批构建，导出到 `__init__.py` 但无任何外部引用

两个都导出为公共 API（`__init__.py` L112），但实际无人消费。

**建议**：从 `__init__.py` 移除导出（标记为内部），待 R4 完成后删除。

## 二、平行系统

### R6：`generalized_drop_rate.py` —— 独立的老 GDR 框架

这是项目早期的 GDR 抽象，使用 ABC 基类 + `compute(t, history)` 接口。与 `UNIFIED_GDR_REGISTRY`（基于 `compute_from_compact`）**完全平行、无交集**。

调用链：
```
cli.py → TargetCardCountAtT / TargetCardPercentageAtT
analysis_service.py → AnalysisService(gdr_functions: List[GeneralizedDropRate])
```

| 维度 | `generalized_drop_rate.py` | `gdr.py` (UNIFIED_GDR_REGISTRY) |
|------|--------------------------|-------------------------------|
| 数据格式 | `InfoVector`（旧、逐抽） | `CompactResult`（新、聚合） |
| 接口 | `compute(t, history)` | `compute_from_compact(compact, ...)` |
| 注册机制 | 类继承 | `GDRDefinition` + 模块级 `Dict` |
| 权重机制 | `TotalValueAtT` 组合模式 | 函数参数传入 |
| 使用范围 | CLI 少量使用 | GUI 全量使用 |

**建议**：不合并——两个框架服务于不同数据格式。但 `generalized_drop_rate.py` 的 CLI 路径可能需要评估是否迁移到 compact 路径。

## 三、GDRContext 字段未充分利用

### R7：`all_drawable_ids` / `resource_gain_per_day` 存储但无人消费

`GDRContext` 有 7 个字段，但实际使用情况：

| 字段 | 消费者 | 
|------|--------|
| `target_specs` | 几乎所有 `_gdr_*` 函数 |
| `ssr_ids` | `_gdr_ssr_collection` |
| `weapon_character_map` | `_gdr_weapon_character_ratio` |
| `initial_resources` | `compute_gdr_from_cumulative`（间接） |
| `collection_sets` | 仅旧 history 路径 `card_collection_rate` / `compute_collection_rates` |
| `all_drawable_ids` | **无**——仅存储，无 GDR 函数读取 |
| `resource_gain_per_day` | **无**——仅存储，无 GDR 函数读取 |

`all_drawable_ids` 在 `batch_simulator.py` L571 计算后存入，但无消费者。可能是为未来 GDR 预留或已废弃。

**建议**：标注未使用字段（保留但注明状态），未来清理或实现消费逻辑。

## 四、重复模式

### R8：`display_name` ↔ `key` 反向映射重复构建 4 处

每次需要根据 `display_name` 查找 `key`（或反过来），代码都手动遍历 `UNIFIED_GDR_REGISTRY` 构建映射：

| 文件 | 行号 | 构建的映射 |
|------|------|-----------|
| `analysis_panel.py` | 270–274 | `_display_to_key` + `_key_to_display` |
| `analysis_panel.py` | 1743 | `_gdr_key_by_name` |
| `analysis_panel.py` | 1900 | `key_to_display` |
| `process_analysis_panel.py` | 23–27 | `_gdr_key_to_display` |

应该在模块级构建一次，或通过 `GDRContext` / `UNIFIED_GDR_REGISTRY` 提供查询方法。

**建议**：在 `gdr.py` 添加两个工具函数（或在 `GDRContext` 更新时一并处理）：
```python
def gdr_key_by_display_name(name: str) -> Optional[str]:
    for key, defn in UNIFIED_GDR_REGISTRY.items():
        if defn.display_name == name:
            return key
    return None

def gdr_display_name(key: str) -> str:
    defn = UNIFIED_GDR_REGISTRY.get(key)
    return defn.display_name if defn else key
```

### R9：`needs_weights` / `needs_ssr_ids` / `needs_weapon_map` 元数据从未被程序化校验

`GDRDefinition` 定义了三个标记字段表明 GDR 需要哪些额外参数：

```python
needs_ssr_ids: bool = False       # 'ssr_collection' 设置为 True
needs_weapon_map: bool = False    # 'weapon_character_ratio' 设置为 True
needs_weights: str = ''           # 'weighted_satisfaction' / 'total_card_value'
```

但**没有任何代码读取这些字段来校验参数是否传入**。`compute_gdr_from_compact` 接受所有权重/ssr_ids/weapon_map 为可选参数——缺了就静默退化（权重→1.0, ssr_ids→空集, weapon_map→空dict）。

只有 `compatible_with_min_resource` 被实际检查（`plan_search_panel.py` L993, `retreat_search.py` L166）。

**P33 完成后，这个问题部分缓解**：权重不再手动传入，由 GDRContext 统一绑定，不会遗漏。但 `needs_ssr_ids` / `needs_weapon_map` 仍然静默退化——如果 GDRContext 没设 ssr_ids，用 `ssr_collection` GDR 时不会报错，只会返回错误结果。

**建议**：P33 后，GDRContext.checker() 或 compute() 方法中根据 `needs_*` 标记做运行时断言——缺了必要参数直接报错，不静默退化。

### R10：`compute_gdr_from_cumulative` 是手工适配器

```python
def compute_gdr_from_cumulative(cum_snapshot, target_specs, gdr_key,
                                 desire_weights=None, miss_cost_weights=None,
                                 card_value_weights=None, ssr_ids=None,
                                 weapon_character_map=None, ...):
    # 把 cum_snapshot 手工转换为 pseudo_compact dict
    pseudo_compact = {
        'card_counts': cum_snapshot.get('cumulative_card_counts', {}),
        'total_draws': cum_snapshot.get('cumulative_draws', 0),
        ...
    }
    return compute_gdr_from_compact(pseudo_compact, target_specs, gdr_key,
                                     desire_weights, miss_cost_weights, ...)
```

它在 `cum_snapshot` 和 `compute_gdr_from_compact` 之间做适配。但参数签名与 `compute_gdr_from_compact` 几乎一致——所有权重参数全量转发。P33 后，如果改用 GDRContext，这个适配器的权重参数也可消除。

**建议**：P33 完成后，改为 `ctx.compute(pseudo_compact, gdr_key)`。

## 五、抽象未推广（P33 核心问题）

### R11：GDRContext 只有一个构造点——其余调用方绕过了它

`GDRContext` 是整个 GDR 系统设计的"上下文载体"，但：

- **唯一构造点**：`batch_simulator.py` L576（`SimulationEnvBuilder.from_config_store` 内部）
- **11 处调用绕过了它**：直接调 `compute_gdr_from_compact` / `compute_success_probability`，手动传参

这就是 P33 要解决的核心问题——GDRContext 应该成为所有 GDR 调用的统一入口，而不是只在一个地方使用。
