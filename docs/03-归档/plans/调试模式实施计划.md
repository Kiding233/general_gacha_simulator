## 一、需求概述

新增一个「调试模式」Tab 页，提供**模拟抽卡游戏界面 + 全量调试信息**的单页可滚动面板。

### 1.1 核心操作

| 操作 | 说明 |
|------|------|
| **抽取一次** | 单步抽卡，调用 `DrawAction` |
| **抽取十次** | 十连抽，UI 层循环 10 次单抽（不修改引擎） |
| **等待 — 自定义** | 用户输入时长，执行 `WaitAction(duration)` |
| **等待 — +1h / +1d** | 快捷等待按钮 |
| **等待 — 跳到下池开放后** | 计算当前时间到下一个池 `available_from` 的差值 |
| **等待 — 跳到当前池关闭前** | 计算到当前池 `available_until` 前一瞬间的差值 |
| **历史记录** | QDialog 弹窗，逐抽记录表格，支持分页浏览 |

### 1.2 观察数据

1. 当前活跃池子及概率分布
2. 保底计数器（每个池的硬保底/软保底进度）
3. 抽卡结果（出什么卡、触发什么保底）
4. 资源变化（消耗、获得、bonus）
5. 全部 17 项 GDR 值（含阈值、成败判定、Δ 变化箭头）
6. 策略决策日志
7. 传给各分析面板的数据结构（7 种核心对象的 Python repr 原貌）

### 1.3 UI 定位

- **鸣潮式布局**：左侧卡池选择竖列 + 右侧满铺展示区
- **上半区**：模拟抽卡游戏界面（池选择 → 展示区 → 操作按钮）
- **下半区**：调试信息面板（等待控制、池快照、操作日志、GDR 表、引擎状态、原始数据）
- **QScrollArea 单页滚动**，不使用 QSplitter
- **浅色主题**，匹配项目现有 Fusion + #f5f5f5 配色
- **新 Tab 页**，不覆盖任何现有面板

### 1.4 显式搁置

## 二、架构设计

### 2.1 数据流

```
ConfigStore
  → SimulationEnvBuilder.from_config_store()
    → SimulationEnv
      → GachaService(pools, strategy, stop_cond, ...)
        → prepare_debug_session(initial_state)
          → _debug_state / _debug_stats / _debug_accumulated / ...
        → run_simulation_step(force_pity, fixed_card_id, override_pool_id)
          → _execute_action(ss)          # 共享实现
          → _build_incremental_compact() # 增量 CompactResult
          → compute_gdr_from_compact()   # GDR 计算（零修改）
          → _collect_raw_objects()       # 原始数据采集
          → StepSnapshot → DebugPanel 更新 UI
```

### 2.2 核心改造：`GachaService._execute_action()` + `run_simulation_step()`

#### 2.2.0 `_StepState` — 可变状态打包

为避免 `_execute_action()` 参数爆炸（当前循环体涉及 25+ 局部变量），引入轻量 dataclass 打包所有可变状态：

```python
@dataclass
class _StepState:
    state: GachaState
    pity_state: PityState
    stats: SimulationStats
    real_time: float
    resources: dict          # state.resources 的引用
```

`_execute_action(ss: _StepState) -> _ActionSnapshot` 就地修改 `ss`，不可变数据（pools、strategy、pity_engine、target_cards 等）通过 `self` 访问。参数从 ~15 个降为 1 个，调用开销不变（dataclass 属性访问与局部变量同等级）。

#### 2.2.1 `_execute_action()` — 抽取共享逻辑

```python
def _execute_action(self, ss: _StepState, *,
    force_pity: bool = False,
    fixed_card_id: str | None = None,
) -> _ActionSnapshot:
    """执行一步操作（Draw 或 Wait），就地修改 ss 并返回快照。

    由 run_simulation() 和 run_simulation_step() 共用。
    包含：确定活跃池 → 策略选择 Action → 执行 → 更新状态。
    不含 GDR 计算和 raw_objects 采集（由调用方附加）。
    """
```

#### 2.2.2 增量 CompactResult 构建

调试模式逐步累积，但 `compute_gdr_from_compact()` 需要一个完整 `CompactResult`。解决方案：`GachaService` 内部维护增量构建器，每步操作后更新，生成"当前进度的 CompactResult"：

```python
def _build_incremental_compact(self, ss: _StepState) -> CompactResult:
    """从 _debug_accumulated 构造当前进度的 CompactResult。
    
    包含：draw_card_ids / card_counts / total_consumed / 
    total_gained / total_draws / final_resources / pool_draw_counts 等。
    GDR 计算层只读 CompactResult，不关心是「最终」还是「进行中」——API 无需改动。
    """
```

GDR 计算层（`compute_gdr_from_compact()`、`SuccessChecker`）**不需要任何修改**——它们只读 CompactResult 字段。

#### 2.2.3 `card_defs` — 预计算 `_initial_counts`

`card_defs` 中的 `initial_count` 影响 `first_time_bonus` 计算。`GachaService.__init__()` 已将 `card_defs` 存为 `self.card_defs`，在 `prepare_debug_session()` 中预计算：

```python
def prepare_debug_session(self, initial_state):
    self._debug_initial_counts = {}
    for cd in self.card_defs:
        ic = cd.get('initial_count', 0) if isinstance(cd, dict) else getattr(cd, 'initial_count', 0)
        if ic > 0:
            cid = cd['card_id'] if isinstance(cd, dict) else cd.card_id
            self._debug_initial_counts[cid] = ic
    ...
```

`_execute_action()` 通过 `self._debug_initial_counts` 访问——不新增参数。

#### 2.2.4 完整接口总览

```python
def prepare_debug_session(self, initial_state: SimulationState):
    """初始化调试会话。"""
    self._debug_state = copy.deepcopy(initial_state)
    self._debug_pity_state = self.pity_state.clone()
    self._debug_stats = SimulationStats()
    self._debug_step = 0
    self._debug_log: list[dict] = []
    self._debug_accumulated = _IncrementalCompact()  # 增量构建器
    self._debug_initial_counts = {...}               # 从 self.card_defs 预计算
    return self._debug_state

def run_simulation_step(
    self,
    force_pity: bool = False,
    fixed_card_id: str | None = None,
) -> StepSnapshot:
    """执行一步，返回 StepSnapshot（调试模式专用）。"""
    ss = _StepState(
        state=self._debug_state, pity_state=self._debug_pity_state,
        stats=self._debug_stats, real_time=self._debug_state.real_time,
        resources=self._debug_state.resources,
    )
    action_snap = self._execute_action(ss, force_pity=force_pity, fixed_card_id=fixed_card_id)
    # 增量更新 CompactResult
    self._debug_accumulated.update(action_snap)
    compact = self._build_incremental_compact(ss)
    # GDR 计算
    gdr = compute_gdr_from_compact(compact, ...)
    # raw_objects
    raw = self._collect_raw_objects(ss)
    # 组装返回
    return StepSnapshot(
        step=self._debug_step,
        action=action_snap.action_type,
        gdr_values=gdr.values,
        gdr_thresholds=gdr.thresholds,
        gdr_success=gdr.success,
        raw_objects=raw,
        ...
    )

def run_simulation(self, initial_state, ...):
    """批量模拟（现有方法，内部改为循环调用 _execute_action()）。"""
    ss = _StepState(state=initial_state.clone(), ...)
    for iteration in range(max_iterations):
        if self.stop_condition.check(ss.state, [], ss.stats):
            break
        self._execute_action(ss)
    ...
```

**性能评估**：`_execute_action()` 引入一次 dataclass 属性访问 + 一次方法调用，每步额外开销 < 100ns。实际工作（随机数 + 字典合并 + 保底计算）在微秒级，占比 < 0.1%。实施后跑 CLI 计时确认。

### 2.2a 十连抽：UI 层循环 + 锁定当前池

当前引擎只有 `DrawAction(pool_id)`，没有多抽 Action。调试模式下十连抽直接在 UI 层循环调用 10 次 `run_simulation_step()`：

- **不需要**新增 `MultiDrawAction` 或修改 `core/action.py`
- 调试模式无并行模拟竞态问题，循环调用是安全的
- 每抽结果逐条追加到操作日志，10 抽完成后统一刷新展示区
- **锁定当前池**：十连过程中不重新调用 `strategy.select_action()`——首抽确定池子后，后续 9 抽强制使用同一 `pool_id`。避免策略中途切换池子导致十连跨池
  - 实现：在 `run_simulation_step()` 增加可选参数 `override_pool_id: str | None = None`，十连首抽后传入固定 pool_id

若未来需要引擎级十连（原子性、性能优化），再单独评估。当前范围：**仅 UI 层循环 + 锁池**。

### 2.2b 等待操作：复用已有 WaitAction

引擎已支持 `WaitAction(duration)`，三种等待模式映射：

| UI 操作 | 引擎调用 |
|---------|---------|
| 自定义时长 | `WaitAction(duration=用户输入秒数)` |
| +1h / +1d | `WaitAction(duration=3600)` / `WaitAction(duration=86400)` |
| 跳到下池开放后 | 计算 `next_pool.available_from - current_time`，取 `max(0, 差值)` |
| 跳到池关闭前 | 计算 `current_pool.available_until - current_time - epsilon` |

等待操作也走 `run_simulation_step()`，产生的 StepSnapshot 中 `action` 字段为 `"wait"`。

`StepSnapshot` 包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| `step` | int | 当前步数 |
| `action` | str | 执行的动作（`"draw"` / `"wait"` / `"exchange"`） |
| `pool_id` | str \| None | 抽卡的池子 ID（等待时为 None） |
| `drawn_card_id` | str \| None | 抽到的卡 ID（等待时为 None） |
| `drawn_card_name` | str \| None | 卡名 |
| `drawn_rarity` | str \| None | 稀有度 |
| `pity_triggered` | bool | 是否触发保底 |
| `pity_name` | str \| None | 触发的保底名 |
| `pity_counters` | dict | 各池保底计数器 `{pool_id: {pity_name: count}}` |
| `resources_before` | dict | 抽前资源 |
| `resources_after` | dict | 抽后资源 |
| `resources_changed` | dict | 资源变化（正=获得，负=消耗） |
| `bonus_applied` | dict \| None | 本次触发的额外资源 |
| `card_counts` | dict | 当前持有卡数量 |
| `total_draws` | int | 累计抽数 |
| `total_actions` | int | 累计操作数（含等待） |
| `gdr_values` | dict | 实时 GDR 值 `{gdr_key: value}` |
| `gdr_thresholds` | dict | GDR 阈值 `{gdr_key: threshold}`（v2 新增） |
| `gdr_success` | dict | GDR 成败判定 `{gdr_key: bool}`（v2 新增） |
| `strategy_decision` | str | 策略决策描述 |
| `stop_reason` | str \| None | 停止原因（未停止为空） |
| `active_pools` | list | 当前活跃池列表 |
| `timestamp` | float | 模拟时间 |
| `raw_objects` | dict \| None | 核心对象 repr（v2 新增，见 2.3 节） |

### 2.3 原始数据 Dump（v2 新增）

每次操作后，面板需要能展示以下核心对象的 Python repr 原貌。数据来源为 `StepSnapshot.raw_objects` 字典：

| key | 来源模块 | 展示内容 |
|-----|---------|---------|
| `gacha_state` | `core/state.py` | `GachaState` dataclass: resources / pity_counters / real_time / total_actions / extra_state |
| `pools` | `core/pool.py` | 当前所有 Pool dataclass: id / name / cost / rewards（限前10条+N条总计） / available_from/until / is_exchange / pool_type |
| `compact_result` | `core/result_types.py` | `CompactResult.to_dict()`：逐抽序列（仅最后 K 步）、card_counts、final_resources、total_consumed |
| `pity_engine` | `core/pity.py` | 每条保底规则的计数器当前值 / 触发阈值 / 概率分布 / 重置条件 |
| `strategy_context` | `core/strategy.py` | `StrategyContext` dataclass: state / current_pools / target_cards / acquired / pool_draw_counts / total_draws / ssr_ids |
| `stop_condition` | `core/stop_condition.py` | 类型名 / 参数 / `.check()` 当前判定 / `.description()` |
| `resource_gain` | `core/resource_gain.py` | 各收益函数定义 + SimulationStats: total_consumed / total_gained / card_counts / pity_triggers |

展示方式：
- 每个数据结构一张**可折叠卡片**（QGroupBox 或自定义 collapsible widget）
- 展开后以 `monospace` 字体显示 Python `repr()` 或 dataclass 字段展开
- 颜色标记：键名(绿色)、数值(黄色)、字符串(蓝色)、布尔(橙色)
- 默认全部折叠，按需点开
- **只读**，不可编辑
- 每次操作后自动刷新所有展开中的卡片
- Pool 的 rewards 列表限制显示数量（前 10 条 + "…共 N 条"）
- CompactResult 序列数组只显示最后 K 步

- 颜色：成功 `#4CAF50`，失败 `#f44336`，不变 `#888`
- 数据来源：`UNIFIED_GDR_REGISTRY` + `compute_gdr_from_compact()`

## 四、调试选项详述

> 以下选项放置于下半区调试信息区域的**最顶部**（等待操作栏上方），以紧凑的 QHBoxLayout + QCheckBox / QPushButton 形式呈现。默认全部关闭。

### 4.1 强制触发保底
- 勾选后，下次抽卡必定触发当前池的保底机制
- 实现：在 `run_simulation_step()` 中检测此标志，跳过随机抽取，直接返回保底卡
- 触发后**自动取消勾选**（单次生效）

### 4.2 固定出卡 ID
- 用户在下拉列表中选择卡 ID（或输入卡名），每次抽卡固定出该卡
- 实现：跳过 `pool.draw()` 随机逻辑，直接构造指定卡的结果
- 下拉列表数据来源：当前池的 rewards 列表

### 4.3 显示策略决策日志
- 每次 `strategy.select_action(ctx)` 调用后，在操作日志中追加一条特殊记录：
  - 当前 ctx 的快照（活跃池、目标完成情况、资源）
  - 策略选择的 Action 及原因
- 日志中以不同颜色/标签区分（如 `[策略]` 标签）

### 4.4 显示 GDR 计算过程
- 每次操作后，在 GDR 表下方追加一行计算过程摘要
- 对已启用的 GDR 逐项展示计算输入和输出
- 例：`资源消耗 = total_consumed['draw_resource'] = 7660`

### 4.5 显示保底概率详情
- 在引擎内部状态区域追加保底概率分布信息
- 当前保底计数器 × 概率曲线 → 下次出 SSR 的概率
- 以简短的一行文字展示（如 "下次出 SSR 概率: 0.6% → 软保底生效中 → 到 90 抽时 100%"）

### 4.6 自动模式（v2 保留）

- **自动前进**：勾选后自动逐歩执行，间隔可调（QSpinBox，默认 500ms）
- **暂停**：自动模式下暂停执行
- **重置会话**：清空当前调试会话，回到初始状态
- **跳转步数**：输入步数 N，快进执行 N 步（批量调用 `run_simulation_step()`，中间不刷新 UI）

### 4.7 实现优先级

- P0（必须）：重置会话、自动前进/暂停
- P1（重要）：强制触发保底、固定出卡、策略决策日志
- P2（可选）：GDR 计算过程、保底概率详情

## 五、实现步骤

### Phase 1：核心单步逻辑（1-2 天）

1. **`GachaService` 扩展**
   - 定义 `_StepState` dataclass + `_ActionSnapshot` dataclass（`service/gacha_service.py` 内部）
   - 抽取 `_execute_action(ss: _StepState, *, force_pity, fixed_card_id) -> _ActionSnapshot`
   - `run_simulation()` 内部改为构造 `_StepState` → 循环调用 `_execute_action()`
   - 新增 `prepare_debug_session(initial_state)`：初始化 `_debug_*` 属性 + 预计算 `_debug_initial_counts`
   - 新增 `run_simulation_step(force_pity, fixed_card_id, override_pool_id)`：构造 `_StepState` → `_execute_action()` → 附加 GDR + raw_objects
   - 实施后跑 CLI 计时确认性能无明显退化（预期 < 0.5%）

2. **增量 CompactResult 构建**
   - 实现 `_IncrementalCompact` 内部类：每步后更新 `draw_card_ids`、`card_counts`、`total_consumed` 等
   - 实现 `_build_incremental_compact(ss) -> CompactResult`
   - GDR 计算层（`compute_gdr_from_compact()`）**零修改**

3. **GDR 实时计算**
   - `run_simulation_step()` 末尾调用 `compute_gdr_from_compact(incremental_compact)`
   - 附加阈值、成败判定、Δ 变化到 StepSnapshot

4. **原始数据采集**
   - 实现 `_collect_raw_objects(ss) -> dict`：采集 7 种核心对象的 repr/dict（见 2.3 节）

5. **单元测试**
   - N 次单步执行 vs `run_simulation()` 一次执行 N 步——结果完全一致
   - 十连锁池：10 次单步 = 同池 10 抽
   - 等待操作与抽卡操作混合执行一致性

### Phase 2：UI 面板——上半区游戏界面（2-3 天）

1. **`DebugPanel` 整体骨架**
   - QScrollArea 包裹 QWidget + QVBoxLayout
   - 上半区 QHBoxLayout（侧栏 + 展示区）
   - 下半区 QVBoxLayout（调试面板组）
   - 样式：Fusion + 内联 QSS，颜色遵循 3.4 配色规范

2. **左侧卡池选择列**
   - QVBoxLayout + QPushButton × N
   - 当前选中池金色高亮边框
   - 点击切换：更新展示区标题/保底/概率信息

3. **右侧展示区**
   - 主 QWidget，QSS `background-color: #dce3ea` 纯色背景
   - 子控件绝对定位（`setGeometry` 在 `resizeEvent` 中计算）：
     - 资源条（右上）
     - 标题 + 保底（顶部居中）
     - 卡面占位（中央）
     - 历史按钮（左下）
     - 抽取一次/十次按钮（右下）
   - 抽卡结果网格：QGridLayout 2×5，覆盖在中央展示区上

4. **交互逻辑**
   - 单抽：调用 1 次 `run_simulation_step()`，更新 UI
   - 十连：首抽确定 pool_id，后续 9 抽传 `override_pool_id` 锁定池子；每抽追加日志，最后统一刷新展示区
   - 历史按钮：打开 QDialog（`debug_history_dialog.py`）
   - 等待按钮：构造 WaitAction → 调用 `run_simulation_step()`

### Phase 3：UI 面板——下半区调试信息（2-3 天）
