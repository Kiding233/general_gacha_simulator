## 二、根因分析

1. **`GDRCalculator` 被绕过**——`GDRCalculator.__init__` 已绑定权重 + key，`compute_gdr(compact)` 零参调用。但多数调用方不用它，直接调 `compute_gdr_from_compact` 然后手传参数——传了就冗余，忘了就静默退化
2. **`config_panel` 充当了不该当的数据源**——权重存在 `ConfigStore.card_weights`，读写逻辑正确，但运行时没人从 ConfigStore 取，全从 UI 控件搬运
3. **新面板复制粘贴**——每个新面板照抄 `_config_panel.get_desire_weights()` 三段式

**核心矛盾不是缺少封装——`GDRCalculator` 早就在那里了。是调用方没用它。**

### 3.1 数据流

```

                      │
                      ▼
           ConfigStore.card_weights
                      │
                      ▼
           ConfigStore.desire_weights    } 三个 property
```

## 四、核心设计：一个工厂函数消除所有重复

### 4.1 `make_gdr_calculator` —— gdr.py 新增的唯一代码

```python
# core/gdr.py —— 模块级工厂函数

def make_gdr_calculator(store, target_specs, gdr_key, *,
                         gdr_threshold=None, ssr_ids=None,
                         weapon_character_map=None) -> GDRCalculator:
    """从 ConfigStore 提取权重，构造 GDRCalculator。

    调用方只需传 store + target_specs + gdr_key。
    权重从 ConfigStore.xxx_weights 自动提取，不可能遗漏。
    """
    return GDRCalculator(
        target_specs=target_specs,
        gdr_key=gdr_key,
        gdr_threshold=gdr_threshold,
        desire_weights=store.desire_weights,
        miss_cost_weights=store.miss_cost_weights,
        card_value_weights=store.card_value_weights,
        ssr_ids=ssr_ids,
        weapon_character_map=weapon_character_map,
    )
```

**设计原则**：
- 不挂在任何类上——GDRContext 是序列化信封，GDRCalculator 不该知道 ConfigStore，ConfigStore 不该知道 GDRCalculator
- 用函数消除重复，不用类来消除重复
- 未来权重语义变更，只改这一个函数

### 4.2 ConfigStore 添加三个 property

```python
# core/config_store.py —— ConfigStore 类

@property
def desire_weights(self) -> Dict[str, float]:
    return {cid: cw.desire_weight for cid, cw in self.card_weights.items()}

@property
def miss_cost_weights(self) -> Dict[str, float]:
    return {cid: cw.miss_cost_weight for cid, cw in self.card_weights.items()}

@property
def card_value_weights(self) -> Dict[str, float]:
    return {cid: cw.card_value for cid, cw in self.card_weights.items()}
```

### 4.4 PlanSearchEngine：target_specs 动态变化

`_simulate_with_resource` 的 `target_specs` 每次搜索不同——不绑在 `__init__`，方法内动态构造：

```python
# —— 改前 ——
class PlanSearchEngine:
    def __init__(self, config_store, ..., desire_weights=None, miss_cost_weights=None, ...):
        self.desire_weights = desire_weights or {}
    def _simulate_with_resource(self, env, target_specs, resource_value):
        ...
        return compute_success_probability(histories, target_specs, self.gdr_key,
            self.gdr_threshold, self.desire_weights, self.miss_cost_weights, ...)

# —— 改后 ——
class PlanSearchEngine:
    def __init__(self, config_store, ..., add_order=None, remove_order=None):
        self.config_store = config_store
        # 不再缓存权重——target_specs 每次不同，方法内动态构造
    def _simulate_with_resource(self, env, target_specs, resource_value):
        ...
        checker = make_gdr_calculator(
            self.config_store, target_specs, self.gdr_key,
            gdr_threshold=self.gdr_threshold,
        )
        _, _, prob = checker.check_batch(histories)
        return prob
```

### 4.5 SuccessChecker → GDRCalculator 改名

`SuccessChecker` 这个名字暗示只能成败判定，但实际上它提供了 `compute_gdr()`（单次 GDR 值）和 `check_batch()`（批量统计）——是一个通用的 GDR 计算器。改为 `GDRCalculator` 反映实际功能。

```python
# core/gdr.py —— 类定义改名

class GDRCalculator:
    """绑定了 gdr_key + 权重的 GDR 计算器。构造后零参调用。"""
    def __init__(self, target_specs, gdr_key=..., gdr_threshold=None,
                 desire_weights=None, miss_cost_weights=None,
                 card_value_weights=None, ssr_ids=None, weapon_character_map=None):
        ...  # 实现不变

    def compute_gdr(self, compact_or_aggregate) -> float: ...
    def is_success(self, compact_or_aggregate) -> bool: ...
    def check_batch(self, results) -> Tuple[int, int, float]: ...

```
