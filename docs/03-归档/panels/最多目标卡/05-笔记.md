### 2026-06-08 续：待讨论项决议

五条遗留项全部确认，计划文件更新至 v8：

**① 权重迁移**：用户指出方案 1.4 节已默认为双列表，不存在迁移问题。风险表中权重迁移降为「低」。

**② Pareto 性能预估**：完整时间线 Pareto 枚举 N 张卡 × 每张一次二分搜索（~12 轮），N=20 时约 240 轮模拟。缓解方案——二分结果缓存 + 快速模式（减少模拟次数）+ UI 预估耗时提示。保留「中」风险等级。

**③ P26 依赖**：非硬依赖，可并行。P26 修引擎层算法（`core/retreat_search.py`），P8 合并 UI 层，变更区域不重叠。建议不同日合入避免冲突。

**④ cost_per_draw 多池修复**（已细化）：用户质疑「提取多个计算不是还是只能用一个吗」。修正为：退路点模式用 `from_pool_id` 定位取 cost；完整时间线模式取包含目标卡的第一个池的 cost。二分搜索用 cost 做 epsilon 精度 + 上界初始化，必须有一个值，取最相关池即可。用户始终可通过 spinbox 手动覆盖。

**⑤ 错误处理统一**：三个面板 Worker 的错误处理对比：
- Strategy/Resource：`_should_stop` 标志位轮询 + `DetailedError` 包装 + `worker.wait()`
- Retreat：`engine.stop()` 委托 + 原始异常 + 无 wait + `assert isinstance`
- 统一方案写入计划 2.6 节

**⑥ 视觉伴侣同步**：用户指出计划文件 2.1-2.5 仍为 v6 旧版步骤引导式设计，与 v8 视觉伴侣完全不同。重写整个 UI 设计章节（2.1-2.6），删除「步骤1/2/3」「渐进式披露」「条件可见（setVisible）」等概念，替换为 QGroupBox 分组 + QRadioButton + 灰显（setEnabled）+ 快照系统。

**⑦ 文档合并与拆分方案**写入计划 八：
- 新建 `方案搜索/` 目录（六骨架 + 本计划 + P26 迁入）
- `退路分析/` → 重命名为 `脆弱性分析/`，删退路搜索内容，仅保留脆弱性
- `最多目标卡/` + `最少资源/` → 整包移入 `docs/03-归档/`
- 模块状态矩阵更新条目

**⑧ tab 命名**：合并后 Tab「退路分析」→ Tab「脆弱性分析」，与目录重命名一致。

**⑨ 快照系统行为逻辑**（补充）：内存列表存储、上限 20 条、正常完成自动 push、停止/出错不产生快照、跨模式共存、翻看时搜索参数区灰显、点击搜索退出翻看模式。

**⑩ 并行进程默认值**：三个面板已统一 `max(1, (os.cpu_count() or 8) - 2)`，合并后不变，计划中补注。

**⑪ cost_per_draw——理论局限，非 bug**（覆盖 ④）：用户指出多池成本不同时，任何单一值都是错的，手动输入也不行。确认二分搜索在资源空间运行，cost 只影响 epsilon 精度（值越小越精确）和 UI 换算展示。结论：降级为阶段 E 代码归并 + tooltip 标注，不再称「修复」。风险表中移出，改为理论局限说明。
