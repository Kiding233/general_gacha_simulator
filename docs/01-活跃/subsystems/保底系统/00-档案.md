## 业务问题

抽卡游戏的核心机制之一：随着玩家连续未抽到高稀有度卡片，系统逐渐提高高稀有度的出率，直到玩家「保底」获得高稀有度卡。保底机制深刻影响玩家的抽卡策略——何时该抽、何时该等——以及模拟结果的统计分布。

需要一个**多池多保底引擎**，能够：
1. 建模软保底（概率从某抽起逐渐递增）和硬保底（达到阈值时 100% 触发）
2. 支持一个池适用多个保底机制（如同时存在 SSR 软保底和限定硬保底）
3. 按配置的顺序叠加应用多个保底
4. 支持多种重置条件（`any_ssr`/`featured`/`never`）
5. 正确处理保底目标分布的高亮（target distribution）——将增加的概率分配给特定卡片

## 设计意图

### 核心抽象

- **PityBehavior（ABC）**：保底行为抽象基类。`apply(counter_value, probabilities, extra) -> adjusted_probabilities` 方法接受当前概率分布，返回保底调整后的概率分布。`is_active(counter_value)` 查询该保底在给定计数器值下是否生效。
- **SoftPityBehavior**：软保底。从 `start_at` 开始渐进增加高稀有度概率（支持 linear/exp/step 三种函数），至 `end_at` 时达到 100%（实际到 100% 时由硬保底保证）。
- **HardPityBehavior**：硬保底。计数器达到 `threshold` 时 100% 触发，将目标卡片概率设为 1.0。
- **PityEngine**：多池多保底引擎。管理所有池的保底配置。核心方法：
  - `get_probabilities(pool_id, state, base_probabilities)`：查询保底调整后的概率分布，**不修改**计数器（满足 P16 修复——查询与修改分离）
  - `before_draw(pool_id, state, base_probabilities)`：抽卡前递增计数器 + 查询调整后概率
  - `after_draw(pool_id, state, reward_id)`：抽卡后按重置条件重置计数器
- **PityState**：单次模拟的保底计数器状态。`increment(name)` / `reset(name)` / `get(name)`。
- **PoolPitySpec**：描述单个池适用的保底机制列表及其参数。

### 保底管道机制

一个池可能适用多个保底（如先有 SSR 软保底提升概率，再有限定硬保底确保特定卡）。这些保底按 `pity.txt` 中的定义顺序构成一条**管道**——前一个保底调整后的概率分布传入下一个保底继续调整。最终概率分布是管道末端输出。

### P16 修复：查询/修改分离

P16 修复之前，`get_probabilities()` 内部会隐式修改保底计数器，导致所有调用者（包括只做概率查询的策略）都会产生副作用。修复后：
- `get_probabilities()` ——纯查询，不修改 `PityState`
- `before_draw()` ——先 `increment` 再查询
- `after_draw()` ——按重置条件 `reset`

## 技术实现

### 调用链

```
GachaService 每步决策:
  1. strategy.select_action(ctx)          # 策略可能调用 ctx.get_pity_probabilities() 查询概率
  2. pity_engine.before_draw(pool_id, state, base_probs)  # 递增计数器 + 获取调整后概率
  3. 按调整后概率执行抽卡
  4. pity_engine.after_draw(pool_id, state, reward_id)     # 按重置条件重置计数器
```

### 核心数据结构

| 类 | 职责 |
|----|------|
| `PityBehavior (ABC)` | 保底行为抽象 |
| `SoftPityBehavior` | 软保底：渐进概率增加 |
| `HardPityBehavior` | 硬保底：100% 触发 |
| `PityDefParsed` | dataclass——从 `pity.txt` 解析的保底定义 |
| `PoolPitySpec` | dataclass——单个池的保底配置（保底名称列表 + SSR/限定 ID 集合 + 解析后目标分布） |
| `PityState` | dataclass——保底计数器字典 + `increment`/`reset`/`get` |
| `PityEngine` | 多池多保底引擎——组合以上所有组件 |

### 实施历史

- **v1.0-v1.5**：单保底、单管道模式
- **v1.6**：引入多保底叠加管道机制
- **v1.8（P16 修复）**：`get_probabilities` 与 `before_draw` 分离，消除隐式副作用（缺陷已修复）
- **当前状态**：软/硬保底、多保底叠加、三种重置条件全部可用。已知理论缺陷 1 项（P30 缺陷 11——多管道优先级文档化）
