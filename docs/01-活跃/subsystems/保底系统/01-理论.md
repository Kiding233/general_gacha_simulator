## 一、核心假设

### 1.1 概率调整的可加性假设

多保底管道假设**保底效果可叠加**——保底 A 调整概率分布后，保底 B 在 A 的结果上继续调整。

**隐含假设**：
1. **调整方向一致性**：所有保底增加同一方向（提高高稀有度概率），不存在「保底 A 提高 SSR 概率，保底 B 降低 SSR 概率」的冲突。
2. **单调性保持**：概率调整后所有值的和为 1.0（归一化不变）。
3. **顺序依赖**：管道的最终概率分布依赖于保底的应用顺序。

### 1.2 保底目标分布的独立性

软保底和硬保底均支持 `target_distribution` 参数——指定概率增量分配给哪些卡片。解读方式：

- 有 `resolved_targets` 时（从池配置中解析的目标卡）：概率增量按 `resolved_targets` 中的权重分配给指定卡
- 无 `resolved_targets` 时：概率增量全部加给概率列表中的第一个卡片

### 1.3 软保底概率递增模型

三种 `func_type`：

| func_type | 公式 | 行为 |
|-----------|------|------|
| `linear` | `progress = (counter - start) / (end - start)` | 线性递增 |
| `exp` | `progress = raw²`（`raw` 为线性进度） | 前慢后快——早抽增益小，接近保底时加速 |
| `step` | `raw < 0.33 → 0; raw < 0.66 → 0.5; raw ≥ 0.66 → 1.0` | 三段阶梯——模拟「软保底线」的实际行为 |

### 1.4 重置条件

| 条件 | 含义 | 适用场景 |
|------|------|---------|
| `any_ssr` | 任何 SSR 卡出卡后重置计数器 | 标准 SSR 保底 |
| `featured`（`featured_ssr`） | 仅限定的 SSR 出卡后重置 | 限定池保底——歪到普池 SSR 不重置 |
| `never` | 永不重置 | 跨池累计保底（如「天井」累计兑换点数） |

## 二、多保底管道的优先级问题（缺陷 11）

### 2.1 问题描述

保底按 `pity.txt` 中的定义顺序依次调整概率分布。当多个保底适用同一池且目标集不同时，后处理的行为可能覆盖前者的调整。无冲突解决机制。

**示例场景**：池子同时配置了「SSR 软保底」（高亮所有 SSR）和「限定硬保底」（高亮特定限定卡）。当两个保底都触发时，硬保底将概率 100% 分配给特定限定卡，覆盖软保底对所有 SSR 的概率提升。

### 2.2 理论立场

管道顺序语义是**有意为之的设计**而非 bug——它允许配置者通过排列保底顺序来控制优先级。后应用的保底覆盖前者的调整是管道模式的预期行为。

需要的是**文档化**而非修改代码逻辑：
1. 在 `pity.py` 模块 docstring 中说明管道顺序语义
2. 在 `PityEngine.__init__()` 中增加冲突检测——若两个保底适用同一池且目标集部分重叠，发出 `warnings.warn()`
3. 长期方向：支持显式 `priority` 字段

### 2.3 与 P16 的关系

P16 修复了计数器和归一化的代码 bug（隐式副作用、不正确的归一化），缺陷 11 处理的是保底交互的**语义定义**。两者互补——P16 保证代码正确性，缺陷 11 保证设计可理解性。

## 三、软保底概率调整的数学分析

### SoftPityBehavior._apply_targeted()

给定进度 `progress ∈ [0, 1]` 和当前概率分布 `p_i`：

```
target_prob = Σ_{i∈target} p_i
other_prob = 1 - target_prob
new_target_prob = min(target_prob + progress × other_prob, 1.0)
new_other_prob = 1 - new_target_prob
scale = new_other_prob / other_prob

p'_i = {
    new_target_prob × (w_i / Σw_j)    if i ∈ target
    p_i × scale                       if i ∉ target
}
```

**数学性质**：
- 归一化保持：`Σ p'_i = 1`（当 `new_target_prob < 1` 时）
- 相对权重保持：目标卡之间的相对比例不变，非目标卡之间的相对比例不变
- `progress = 0 →` 不调整；`progress = 1 →` 目标卡概率达到 1.0

### HardPityBehavior.apply()

当 `counter >= threshold`：目标卡概率 = 按 `resolved_targets` 权重分配 1.0，其他卡概率 = 0.0。
