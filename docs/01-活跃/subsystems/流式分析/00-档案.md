## 业务问题

批量模拟产生大量数据——10,000 次模拟 × 每次 200 抽 = 200 万条逐抽记录。如果将所有原始数据保留在内存中，内存占用与模拟次数 N 呈 O(N) 关系，在大规模模拟中迅速耗尽可用内存。

同时，许多分析面板只需要聚合统计量（如成功率、GDR 分布、事件模式频率），不需要保留所有原始数据。传统的「先存储全部结果、再批量分析」模式造成：
1. 内存浪费——原始数据在分析完成前一直占用内存
2. 延迟——必须等全部模拟完成后才开始分析
3. 扩展性瓶颈——N 越大越严重

需要一个**流式处理架构**，在每次模拟完成后立即提取关键指标，然后丢弃原始数据。

## 设计意图

### 核心抽象

- **StreamingAnalyzer（ABC）**：流式分析器抽象基类。`on_result(compact)` — 处理单次模拟结果；`get_result()` — 返回累积分析结果。
- **SharedResultCollector**：流式结果收集器——边模拟边提取边丢弃。内存占用与模拟次数 N 无关（O(1)）。注册多个 extractor 函数，每次 `on_result(compact)` 时并行执行所有 extractor，结果积累在各自的列表中。
- **extract 函数系列**：独立的数据提取函数，从单个 `CompactResult` 字典中提取特定维度的信息。包括 `extract_aggregate`（聚合统计）、`extract_process`（过程事件）、以及 `WorkerLocalExtractor`（Worker 内执行的综合提取器）。

### 流式管道

```
batch_simulator 分发任务到 Worker 进程
  → Worker 返回 CompactResult
  → SharedResultCollector.on_result(compact)
    → extractor_1(compact) → acc_1.append(...)
    → extractor_2(compact) → acc_2.append(...)
    → ...
  → compact 被丢弃（Python GC）
  → 下一轮模拟
```

分析结束时，只有 extractor 的累积结果保留在内存中——通常是轻量级的统计量（成功率计数、分布直方图数据、事件频率表），而非原始序列。

### 单线程使用约束

`SharedResultCollector` 设计为单线程使用。`add_extractor` 应在所有 `on_result` 调用之前完成注册；`on_result` 不可并发调用。`n_results += 1` 非原子操作，多线程并发调用会产生数据竞争。

## 技术实现

### 调用链

```
面板 Worker._run_impl()
  → run_batch_parallel(params, initializer, ...)
    → 每个子进程: GachaService.run_simulation() → CompactResult
    → 主进程: collector.on_result(compact)
    → collector.on_result(compact) 调用每个 extractor
  → 面板读取 collector.get_extracted(name) 获取聚合数据
```

### 核心数据结构

| 类 | 职责 |
|----|------|
| `StreamingAnalyzer (ABC)` | 流式分析器抽象——`on_result` + `get_result` |
| `StreamingSuccessCounter` | 流式成功率计数器——边模拟边累加 `success/total`，O(1) 内存 |
| `SharedResultCollector` | 通用流式收集器——注册多 extractor，结果分 name 累积 |

### extract 函数一览

| 函数 | 提取内容 | 输出类型 |
|------|---------|---------|
| `extract_aggregate(compact)` | 聚合统计（card_counts、pool_draw_counts、total_draws、final_resources、保底计数器最大值等） | `dict` |
| `extract_process(compact, ...)` | 过程分析数据（池事件类型、成功判定、GDR 值） | `dict` |

### WorkerLocalExtractor

位于 Worker 进程内的综合提取器。每次 `process(compact)` 返回轻量提取包（包含聚合数据 + 可选的前 N 个序列），由主进程负责合并。

**设计原因**：避免跨进程状态同步的复杂性——Worker 只负责提取，主进程负责累积和合并。与 `SharedResultCollector` 的分工：前者在 Worker 内执行（每次返回提取包），后者在主进程执行（累积所有提取包）。

### 实施历史

- **v1.8**：引入 `SharedResultCollector` + `StreamingSuccessCounter`，替换旧的「先存储全部结果再分析」模式
- **v1.9**：新增 `WorkerLocalExtractor`，将部分提取工作下沉到 Worker 进程
- **当前状态**：流式架构完成，所有面板使用流式模式。内存占用已验证为 O(1)。无已知活跃问题。
