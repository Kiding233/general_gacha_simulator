# 流式分析 -- 理论

> 创建日期：2026-06-02（首次建档）
> 状态：已完成子系统，无已知理论缺陷

---

## 一、核心假设

### 1.1 提取的独立性假设

每个 extractor 对单个 compact 的提取结果是独立的——extractor A 的输出不依赖于 extractor B 的输出。这保证：
1. extractor 可以任意添加/删除，不影响其他 extractor 的结果
2. extractor 的执行顺序不影响最终结果
3. 单个 extractor 的异常不会破坏其他 extractor 的累积数据

### 1.2 累积结果的可合并性假设

extractor 的累积结果（`acc` 列表）支持在下游做聚合。与 extractor 配套的是面板层的合并逻辑——例如：
- `StreamingSuccessCounter`：累加 `success` 和 `total` 计数
- 直方图数据：合并多个子模拟批次的 bin 计数
- 事件频率表：合并多个批次的频率字典

合并逻辑的数学正确性依赖于提取结果的**可加性**——合并两个批次的提取结果在数学上等价于一次处理所有结果。

### 1.3 内存 O(1) 的边界条件

「O(1)」是相对于 N（模拟次数）的——提取器的累积列表大小随 N 增加但有其上限（如直方图的 bin 数固定为 `n_heatmap_bins`，事件频率表的最大维度为 7×N_pools）。严格意义上，提取器累积数据的大小为 O(1)（不随 N 增长）。

**例外**：`extract_aggregate` 的累积列表大小 = N（每个结果一条记录），不满足 O(1)。但 `extract_aggregate` 通常只用于小规模模拟或调试，大规模模拟使用专用提取器。

### 1.4 单线程约束

`SharedResultCollector` 设计为单线程使用，因为：
- `dict` 和 `list` 的并发写入在 CPython 中受 GIL 保护但不保证原子性
- `n_results += 1` 非原子操作
- 无锁设计换取简单性和零同步开销

在并行模拟场景中，`collector.on_result()` 在主进程的回调中串行调用（`Pool.apply_async` 的回调），满足单线程约束。

## 二、流式处理的正确性

### 2.1 结果顺序无关性

模拟结果的到达顺序可能不同（不同 Worker 的完成时间不同），但 `SharedResultCollector` 的累积结果对到达顺序不敏感——因为每个 extractor 的输出被追加到列表，下游聚合（如 `mean`、`histogram`）对顺序无关。

### 2.2 部分完成的鲁棒性

流式架构天然支持「部分完成」场景——如果批量模拟在中途被取消，已经累积的提取结果仍然有效（成功率 = success_so_far / total_so_far）。这与「先存储全部结果」模式不同——中途取消时已存储的原始数据仍然需要完整分析才能产生有意义的结果。

### 2.3 增量更新

面板 Worker 可以通过检查 `collector.n_results` 在模拟运行期间周期性更新 UI（如每 100 次模拟刷新一次进度和当前统计量）。流式架构使「运行中查看阶段性结果」成为可能。
