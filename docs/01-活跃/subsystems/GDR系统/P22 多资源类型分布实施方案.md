## 一、动机与目标

### 1.1 问题描述

当前所有资源相关的 GDR、图表、统计指标都**硬编码**使用 `draw_resource`（抽卡资源）。系统当前配置中定义了两种资源类型（`draw_resource` 抽卡资源、`exchange_currency` 兑换货币）作为示例，且 `CompactResult.total_consumed`、`final_resources`、`pool_end_resources` 等字段已经是 `Dict[str, float]` 结构——意味着底层数据模型天然支持任意多种资源，但分析层尚未利用这一能力：

1. **GDR 层**：`resource_remaining`、`resource_consumed`、`resource_per_card`、`resource_efficiency` 等指标硬编码 `'draw_resource'`
2. **分析面板**：资源分布直方图、CDF、箱线图等固定使用 `draw_resource`，X 轴标签写死"抽卡资源"；`_DRAW_UNIT_GDR_KEYS` 按资源类 GDR 批量换算
3. **其他面板**：`plan_search_panel`、`retreat_panel`、`strategy_panel` 均硬编码
4. **不抽卡参考线**：仅计算 `draw_resource` 的基线值

用户无法查看「兑换货币」的分布、消耗、效率，也无法在不同资源类型之间切换分析视角。未来若新增第三种资源（如体力），同样无法分析——问题本质是「硬编码单一 key」，而非「当前恰好有两种」。

### 1.2 目标

1. **GDR 层参数化**（v2 方案）：在 `UNIFIED_GDR_REGISTRY` 构建时，资源类 GDR 按 `resource_defs` 展开为 `gdr_key:resource_id` 格式的独立条目（如 `resource_remaining:exchange_currency`），底层 compute 函数通过 `resource_id` 参数支持多资源类型读取
   - ~~v1 方案：GDR 函数签名增加 `resource_id` 参数，通过 UI 下拉选择器全局切换~~ → 已由 v2 替代
2. **分析面板适配**：GDR 下拉列表自动包含展开后的资源类 GDR 条目（通过 `display_name` 区分），原计划的「资源类型全局下拉框」取消——资源维度已融入 GDR 条目本身
3. **图表层联动**：X 轴标签、标题随 GDR `display_name` 自动适配（已含资源显示名），`cost_per_draw` 按 GDR key 中解析的 `resource_id` 取值
4. **其他面板同步支持**：方案搜索/退路分析/策略比较面板的资源图表均支持多资源类型切换

5. **跨资源条件GDR**：支持以资源A的GDR值（如兑换货币剩余量）为条件，筛选后计算资源B的GDR分布（如抽卡资源剩余量），复用现有 `JointSamples` 条件分布机制
6. **多资源并排查看**：在GDR分布多选列表中，同一资源类GDR按资源类型展开为多个条目，用户可同时勾选查看并排对比

### 1.3 非目标

- **不改变** `exchange_currency` 的获取/消耗机制——兑换池逻辑不变，仅增加**事后分析**维度的资源类型切换
- **不新增** GDR 指标——在现有指标基础上参数化，不引入新的计算逻辑
- **不修改** `CompactResult` 数据结构——`Dict[str, float]` 已满足多资源类型需求

## 三、设计方案

### 3.1 GDR 层：资源类型参数化

#### 3.1.1 核心方案：注册表不动，UI 层展开

**设计决策**：v2 方案的正确内核——把 `resource_id` 编码进 GDR key 字符串（`:` 分隔）——保留。它的错误——把展开后的条目写回全局注册表——废弃。

**架构**：

```
UNIFIED_GDR_REGISTRY (17条) ──── 永不改变
    │                              │
    │                              │
    ├─ compute_gdr_from_compact    ├─ get_expanded_gdr_entries(resource_defs)
    │  "消费端"                     │  "生产端" —— 仅 UI 层调用
    │                              │
    │  1. parse_gdr_key(key)     │  纯函数，输入 resource_defs
    │     → (base_key, rid)       │  返回 [(key, display, lower_is_better, threshold)]
    │  2. resolve_gdr_definition  │
    │     (两步查找定义)            │   资源类GDR → 每个资源类型生成 :qualified 条目
    │  3. 注入 resource_id=rid     │   非资源类GDR → 原样输出
    │     执行底层 compute          │
    │                              │
    ├─ 公共辅助函数（两侧共用）       ├─ populate_gdr_combo()
    │  - resolve_gdr_definition   │  - process_analysis GDR下拉框
    │    (仅返回定义，不计算)        │  - plan_search GDR下拉框
    │  - is_resource_gdr(key)     │  - retreat GDR下拉框
    │    (资源类 GDR 判断)          │  - worst_impact GDR下拉框
    │  - parse_gdr_key(key)       │  - strategy_panel GDR下拉框
    │    (key 字符串解析)           │
    │                              ├─ get_expanded_gdr_entries() 直接消费方
    ├─ 所有计算路径                 │  - analysis_panel _gdr_names
    │  - analysis_panel           │    (不走 populate_gdr_combo，直接调用)
    │  - process_analysis         │  - comparison_analysis L4散点轴
    │  - comparison_analysis      │    (不走 populate_gdr_combo，直接遍历注册表)
    │  - plan_search              │
    │  - retreat                  │
```

**`:qualified` key 的完整生命周期**：

```
get_expanded_gdr_entries()
  ↓  产出 'resource_remaining:exchange_currency'
  ↓
UI 下拉框 → 用户选择 → 存入 Worker 参数
  ↓
compute_gdr_from_compact(gdr_key='resource_remaining:exchange_currency')
  ↓  parse_gdr_key → ('resource_remaining', 'exchange_currency')
  ↓  resolve_gdr_definition(gdr_key) → 命中原始定义（两步查找：完整key → base_key）
  ↓  注入 resource_id='exchange_currency' → 执行底层 compute
```

**关键设计约束**：
- `UNIFIED_GDR_REGISTRY` 仍为 17 条，不新增 `parent_gdr` 字段，不调用任何展开函数
- `:` 的语义只存在于 **2 个函数**：`get_expanded_gdr_entries()`（生产 qualified key）和 `compute_gdr_from_compact()`（消费 qualified key）
- 其余所有代码看到的是不透明的 key 字符串，不解析 `:`、不做过滤、不做分组
- UI 条目由 `get_expanded_gdr_entries()` 一次生成，列表干净无冗余——原始 `resource_remaining` 不出现，仅保留 `resource_remaining:draw_resource` 和 `resource_remaining:exchange_currency`

**需要展开的资源类 GDR**（4 个）：`resource_remaining`、`resource_consumed`、`resource_per_card`、`resource_efficiency`。

**不展开**：`draw_conversion_efficiency`（语义天然绑定 `draw_resource`，原因见 §3.1.5）。

**display_name 格式**：`{原显示名} ({资源显示名})`，如「资源剩余 (抽卡资源)」。

> **N 资源兼容性**：本方案不硬编码资源类型数量。`resource_defs: dict[str, str]` 按 `ConfigStore.resources` 传入——用户配置中定义 N 种资源，`get_expanded_gdr_entries()` 即自动产出 17 + 4×N 条 UI 条目。`parse_gdr_key()` 不校验 resource_id 是否在预定义集合中，任意字符串均接受。底层 compute 函数仅做 `dict.get(resource_id, 0)`，与资源数量无关。新增资源类型（如在配置中加 `stamina | 体力`）**零代码改动**即可在全部分析面板中可用。

**向下兼容途径**：

旧代码调用路径 | 行为
---|---
`compute_gdr_from_compact(c, ts, 'resource_remaining')` | `parse_gdr_key` → `('resource_remaining', 'draw_resource')` → 注册表命中原始 key → 行为不变
`populate_gdr_combo(combo)` 未传 `resource_defs` | 按旧行为遍历原始注册表，17 条原样输出
`populate_gdr_combo(combo, resource_defs=self._store.resource_defs)` | 调用 `get_expanded_gdr_entries()`，21 条无冗余

#### 3.1.2 新增函数：`parse_gdr_key()` + `get_expanded_gdr_entries()`

**S1.1** `parse_gdr_key(gdr_key: str) -> tuple[str, str]`：

解析 `:` 分隔的 qualified GDR key，返回 `(base_key, resource_id)`。无 `:` 时默认 `resource_id='draw_resource'`。含 doctest。

```python
# core/gdr.py 模块级常量（UNIFIED_GDR_REGISTRY 下方，~第 55 行）

                                  'resource_per_card', 'resource_efficiency'}

def get_expanded_gdr_entries(resource_defs=None):
    entries = []

                    f'{defn.display_name} ({rname})',

                    defn.default_threshold,

        else:
            entries.append((
                key, defn.display_name,
                defn.lower_is_better, defn.default_threshold,
            ))
    return entries
```

##### C 类：新增公共函数 `resolve_gdr_definition()`

B 类全部 12 个调用点 + A9（`compute_gdr_from_cumulative`）= 13 处需要「先试完整 key → 回退 base_key」的两步查找。提取公共函数消除重复：

```python
def resolve_gdr_definition(gdr_key: str) -> Optional[GDRDefinition]:
    """两步查找 GDR 定义：先试完整 key，回退解析 base_key。
    
    用于所有需要从 gdr_key 获取 GDRDefinition 的消费方。
    与 compute_gdr_from_compact() 的区别：此函数仅返回定义，不执行计算。
    """
    defn = UNIFIED_GDR_REGISTRY.get(gdr_key)
    if defn is not None:
        return defn
    base_key, _ = parse_gdr_key(gdr_key)
    return UNIFIED_GDR_REGISTRY.get(base_key)
```

之后 `compute_gdr_from_compact()` 内部的两步查找也改用此函数（减少内联重复）。

##### 统一规则：两个公共函数覆盖所有消费模式

`:qualified` key 的消费者只有两种需求，各由一个公共函数覆盖：

```
需求：「给我 GDRDefinition 的元数据」
  → resolve_gdr_definition(key)
  → 适用：per_pool_analysis、retreat_search、GDRCalculator.__init__

需求：「计算这次模拟的 GDR 值」
  → compute_gdr_from_compact(compact, specs, key, ...)
  → 适用：analysis_panel、comparison_analyzer、process_analysis、plan_search
```

**禁止模式**（v6 之后应不再出现）：

```python

# ❌ 绕过包装函数直接调 compute——resource_id 不会被注入
defn.compute_from_compact(compact, target_specs=specs)
```

#### 3.1.5 `draw_conversion_efficiency` 保持硬编码

该 GDR 的公式为：

```
抽数转化效率 = (角色/武器池的实际抽数 × cost_per_draw) / 抽卡资源总消耗
```

**含义**：衡量抽卡资源的「有效转化率」。如果玩家只抽卡不兑换，消耗 16000 资源抽了 100 抽，值 = (100×160)/16000 = 1.0。如果部分资源用于兑换（消耗资源但不产生抽数），值会跌破 1.0，反映资源被「分流」到非抽卡途径的程度。

**不展开的理由**：
- 分子（实际抽数 × 每抽成本）和分母（资源消耗）处于**同一个资源系统**——都是 `draw_resource` 的流入流出。这个 GDR 衡量的是单一资源体系内部的转化效率，不是跨资源对比
- 兑换货币有完全不同的获取/消耗路径（兑换池独立运作），不存在「抽数」与「兑换货币消耗」之间的转化关系，公式本身对兑换货币无意义
- 其他 4 个资源类 GDR（剩余/消耗/每卡效率/出卡效率）只是简单地从 `Dict[str, float]` 中读取一个值，切换到不同 key 语义不变；但 `draw_conversion_efficiency` 涉及跨字段运算（`pool_draw_counts` × `cost_per_draw` / `total_consumed`），这三者必须同属一个资源维度才有解释力

**结论**：`draw_conversion_efficiency` 不做多资源展开，不纳入 `get_expanded_gdr_entries()` 的展开范围（即不在 `_RESOURCE_GDR_KEYS` 中），保持现有硬编码行为。

### 3.3 跨模块：cost_per_draw 按资源类型解析

当前 `cost_per_draw` 从池的 `cost` 字段解析（格式为 `draw_resource:160`）。多资源类型化后：

def _extract_cost_per_draw_by_resource(self) -> dict[str, float]:

```python
# 核心层统一入口（位置：core/retreat_search.py）
def extract_cost_per_draw_by_resource(pool_entries) -> dict[str, float]:
    """从池配置解析每种资源类型的每抽成本 {resource_id: cost}。
    
    遍历所有池的 cost 字段（格式 'draw_resource:160|exchange_currency:5'），
    按 resource_id 分类收集。同资源类型多池冲突取首个 + logging.warning。
    默认回退 {'draw_resource': 160.0}。
    """
    costs: dict[str, float] = {}
    for pe in pool_entries:
        cost_str = getattr(pe, 'cost', '')
        if not cost_str:
            continue
        for part in cost_str.split('|'):
            part = part.strip()
            if ':' not in part:
                continue
            rid, _, val_str = part.partition(':')
            rid = rid.strip()
            if rid not in costs:
                try:
                    costs[rid] = float(val_str.strip())
                except ValueError:
                    pass
    if not costs:
        costs['draw_resource'] = 160.0
    return costs
```
