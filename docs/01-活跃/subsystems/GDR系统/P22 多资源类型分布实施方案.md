## 一、动机与目标

### 1.1 问题描述

当前所有资源相关的 GDR、图表、统计指标都**硬编码**使用 `draw_resource`（抽卡资源）。系统虽然定义了两种资源类型（`draw_resource` 抽卡资源、`exchange_currency` 兑换货币），且 `CompactResult.total_consumed`、`final_resources`、`pool_end_resources` 等字段已经是 `Dict[str, float]` 结构，但：

1. **GDR 层**：`resource_remaining`、`resource_consumed`、`resource_per_card`、`resource_efficiency` 等指标硬编码 `'draw_resource'`
2. **分析面板**：资源分布直方图、CDF、箱线图等固定使用 `draw_resource`，X 轴标签写死"抽卡资源"；`_DRAW_UNIT_GDR_KEYS` 按资源类 GDR 批量换算
3. **其他面板**：`resource_search_panel`、`retreat_search_panel`、`retreat_panel`、`strategy_panel` 均硬编码
4. **不抽卡参考线**：仅计算 `draw_resource` 的基线值

用户无法查看「兑换货币」的分布、消耗、效率，也无法在两种资源类型之间切换分析视角。

### 1.2 目标

1. **GDR 层参数化**（v2 方案）：在 `UNIFIED_GDR_REGISTRY` 构建时，资源类 GDR 按 `resource_defs` 展开为 `gdr_key:resource_id` 格式的独立条目（如 `resource_remaining:exchange_currency`），底层 compute 函数通过 `resource_id` 参数支持多资源类型读取
   - ~~v1 方案：GDR 函数签名增加 `resource_id` 参数，通过 UI 下拉选择器全局切换~~ → 已由 v2 替代
2. **分析面板适配**：GDR 下拉列表自动包含展开后的资源类 GDR 条目（通过 `display_name` 区分），原计划的「资源类型全局下拉框」取消——资源维度已融入 GDR 条目本身
3. **图表层联动**：X 轴标签、标题随 GDR `display_name` 自动适配（已含资源显示名），`cost_per_draw` 按 GDR key 中解析的 `resource_id` 取值
4. **其他面板同步支持**：资源搜索/退路搜索/退路分析/策略面板的资源图表均支持多资源类型切换

5. **跨资源条件GDR**：支持以资源A的GDR值（如兑换货币剩余量）为条件，筛选后计算资源B的GDR分布（如抽卡资源剩余量），复用现有 `JointSamples` 条件分布机制
6. **多资源并排查看**：在GDR分布多选列表中，同一资源类GDR按资源类型展开为多个条目，用户可同时勾选查看并排对比

### 1.3 非目标

- **不改变** `exchange_currency` 的获取/消耗机制——兑换池逻辑不变，仅增加**事后分析**维度的资源类型切换
- **不新增** GDR 指标——在现有指标基础上参数化，不引入新的计算逻辑
- **不修改** `CompactResult` 数据结构——`Dict[str, float]` 已满足多资源类型需求

## 三、设计方案

### 3.1 GDR 层：资源类型参数化

#### 3.1.1 核心方案：GDR key 参数化（注册时展开）

**设计决策**：将 `resource_id` 从「运行时传参」提升为「GDR 标识符层面的维度」，使每种资源类型对应一个独立的虚拟 GDR 条目。

在构建 `UNIFIED_GDR_REGISTRY` 时，资源类 GDR 按 `ConfigStore.resource_defs` 中定义的资源类型自动展开：

```python
# 概念示例：resource_remaining 展开为多个注册条目
UNIFIED_GDR_REGISTRY = {
    ...
    'resource_remaining:draw_resource': GDRDefinition(
        key='resource_remaining:draw_resource',
        display_name='资源剩余(抽卡资源)',
        default_threshold=0.0,
        compute_from_compact=_make_resource_gdr(
            _gdr_resource_remaining, resource_id='draw_resource'
        ),
        category='resource',
        parent_gdr='resource_remaining',  # 标记来源，便于 UI 分组
    ),
    'resource_remaining:exchange_currency': GDRDefinition(
        key='resource_remaining:exchange_currency',
        display_name='资源剩余(兑换货币)',
        default_threshold=0.0,
        compute_from_compact=_make_resource_gdr(
            _gdr_resource_remaining, resource_id='exchange_currency'
        ),
        category='resource',
        parent_gdr='resource_remaining',
    ),
}
```

**展开规则**：
- `:` 作为分隔符，格式为 `{gdr_key}:{resource_id}`
- 需要展开的资源类 GDR：`resource_remaining`、`resource_consumed`、`resource_per_card`、`resource_efficiency`（共 4 个）
- **不展开** `draw_conversion_efficiency`——该 GDR 的语义（「抽数 / (资源消耗 / 每抽成本)」）天然绑定 `draw_resource`，兑换货币不参与抽卡动作，展开无分析价值。保持现有硬编码
- 非资源类 GDR（如 `target_achievement`、`all_targets`）保持原 key 不变
- `display_name` 格式：`{原显示名}({资源显示名})`
- 展开后的注册表同时保留原始 key（如 `resource_remaining`，默认指向 `draw_resource`）以向下兼容

**关键优势**：
1. **条件分布直接复用**：条件/目标下拉框中出现 `resource_remaining:draw_resource` 和 `resource_remaining:exchange_currency` 两个独立条目，现有 `JointSamples` + `conditional_second` 机制无需任何改动
2. **多选并排查看**：GDR 分布多选列表中同时勾选两个条目，图表自动并排或叠加
3. **向下兼容**：不带 `:` 的旧 key（如 `resource_remaining`）默认解析为 `resource_remaining:draw_resource`

#### 3.1.2 底层实现：compute 函数保留 `resource_id` 参数 + GDR key 解析

展开后的虚拟 GDR 仍通过 `**kw` 将 `resource_id` 传递给底层 compute 函数（与原方案一致，只是参数来源从「UI 全局选择器」变为「GDR key 解析」）：

```python
def _parse_gdr_key(gdr_key: str) -> tuple[str, str]:
    """从参数化 GDR key 中解析基础 key 和 resource_id。
    
    >>> _parse_gdr_key('resource_remaining:exchange_currency')
    ('resource_remaining', 'exchange_currency')
    >>> _parse_gdr_key('target_achievement')
    ('target_achievement', 'draw_resource')
    """
    if ':' in gdr_key:
        base_key, resource_id = gdr_key.split(':', 1)
        return base_key, resource_id
    return gdr_key, 'draw_resource'  # 默认


def _make_resource_gdr(compute_fn, resource_id: str):
    """工厂函数：为指定资源类型创建闭包 compute 函数。
    
    用于展开注册表时注入 resource_id，避免每个 GDRDefinition 手写 lambda。
    """
    import functools
    @functools.wraps(compute_fn)
    def wrapper(compact, target_specs, **kw):
        return compute_fn(compact, target_specs, resource_id=resource_id, **kw)
    return wrapper


def compute_gdr_from_compact(compact, target_specs, gdr_key='target_achievement', ...):
    base_key, resource_id = _parse_gdr_key(gdr_key)
    defn = UNIFIED_GDR_REGISTRY.get(gdr_key)  # 优先用完整 key 查找
    if defn is None:
        defn = UNIFIED_GDR_REGISTRY.get(base_key)  # 回退基础 key
    if defn is None:
        return 0.0
    return defn.compute_from_compact(compact, ..., resource_id=resource_id)
```

**受影响的底层 compute 函数（7 个）**：函数签名增加 `resource_id` 参数，默认 `'draw_resource'`。
- `resource_remaining` — `final_resources.get(resource_id, 0)`
- `resource_consumed` — `total_consumed.get(resource_id, 0)`
- `resource_per_card` — 同上，分子改为 `resource_id`
- `resource_efficiency` — 同上
- `_total_resource_consumed(history, resource)` — 已有参数，无需改
- `_total_resource_gained(history, resource)` — 同上
- `draw_conversion_efficiency` — **不展开**，保持硬编码 `draw_resource`，不受 `resource_id` 参数影响

#### 3.1.3 `draw_conversion_efficiency` 保持硬编码

该 GDR 的公式为：

```
抽数转化效率 = (角色/武器池的实际抽数 × cost_per_draw) / 抽卡资源总消耗
```

**含义**：衡量抽卡资源的「有效转化率」。如果玩家只抽卡不兑换，消耗 16000 资源抽了 100 抽，值 = (100×160)/16000 = 1.0。如果部分资源用于兑换（消耗资源但不产生抽数），值会跌破 1.0，反映资源被「分流」到非抽卡途径的程度。

**不展开的理由**：
- 分子（实际抽数 × 每抽成本）和分母（资源消耗）处于**同一个资源系统**——都是 `draw_resource` 的流入流出。这个 GDR 衡量的是单一资源体系内部的转化效率，不是跨资源对比
- 兑换货币有完全不同的获取/消耗路径（兑换池独立运作），不存在「抽数」与「兑换货币消耗」之间的转化关系，公式本身对兑换货币无意义
- 其他 4 个资源类 GDR（剩余/消耗/每卡效率/出卡效率）只是简单地从 `Dict[str, float]` 中读取一个值，切换到不同 key 语义不变；但 `draw_conversion_efficiency` 涉及跨字段运算（`pool_draw_counts` × `cost_per_draw` / `total_consumed`），这三者必须同属一个资源维度才有解释力

**结论**：`draw_conversion_efficiency` 不做多资源展开，不纳入 `expand_resource_gdrs()` 的展开范围，保持现有硬编码行为。

#### 3.2.4 条件分布：跨资源条件 GDR

参数化 GDR key 使得「以资源A的GDR为条件，计算资源B的GDR分布」成为可能——这是原方案（全局资源切换）无法实现的场景。

用户在分析面板的「条件分布」模块中：

1. **条件指标下拉框**选择 `resource_remaining:exchange_currency`（显示为「资源剩余(兑换货币)」）
2. **目标指标下拉框**选择 `resource_remaining:draw_resource`（显示为「资源剩余(抽卡资源)」）
3. 设置条件阈值（如「兑换货币剩余 ≥ 100」）

Worker 在 `_run_impl()` 中的现有代码路径**无需任何改动**——GDR key 参数化后，条件指标和目标指标天然可以是两个独立的 key：

```python
# 现有代码路径（analysis_panel.py:648-665），GDR key 参数化后自动支持跨资源
cond_values = [compute_gdr_from_compact(agg, target_specs, 
                'resource_remaining:exchange_currency') 
               for agg in aggregate_data]
target_values = [compute_gdr_from_compact(agg, target_specs, 
                 'resource_remaining:draw_resource') 
                for agg in aggregate_data]

joint = JointSamples(list(zip(cond_values, target_values)))
cond_dist = joint.conditional_second(lambda f: f >= threshold)
# → cond_dist 即为「兑换货币 ≥ threshold」条件下，抽卡资源剩余的分布
```

**设计要点**：
- 不需要新增 UI 控件——条件指标和目标指标各有一个下拉框，每个下拉框中的条目来自 `UNIFIED_GDR_REGISTRY`，展开后自然包含不同资源类型的条目
- `JointSamples` 不关心两个 GDR key 是否同源（同一个基础 GDR 的不同资源类型），它只处理两组 float 值的配对
- 跨资源条件分析的使用场景举例：

  | 条件指标 | 阈值 | 目标指标 | 分析问题 |
  |---------|------|---------|---------|
  | 资源剩余(兑换货币) | ≥ 500 | 资源剩余(抽卡资源) | 兑换货币充裕时，抽卡资源还剩多少？ |
  | 资源消耗(抽卡资源) | ≤ 20000 | 资源剩余(兑换货币) | 抽卡资源消耗少时，兑换货币的剩余分布？ |
  | 资源剩余(抽卡资源) | ≥ 10000 | 目标达成率 | 抽卡资源充裕时，目标达成率如何？ |

#### 3.2.5 多资源并排查看

GDR 分布多选列表（`_get_gdr_dist_selections()`）中，资源类 GDR 按资源类型展开为多个条目：

```
☑ 资源剩余(抽卡资源)      [直方图] [CDF]
☑ 资源剩余(兑换货币)      [直方图] [CDF]
☐ 资源消耗(抽卡资源)      [直方图] [CDF]
☐ 资源消耗(兑换货币)      [直方图] [CDF]
☑ 目标达成率              [直方图] [CDF]
```

**叠加渲染**：当用户同时勾选同一基础 GDR 的多个资源类型条目（如「资源剩余(抽卡资源)」+「资源剩余(兑换货币)」），图表渲染为**叠加直方图**（同一坐标系内两组半透明柱状条，不同颜色 + 图例区分）。

**实现方式**：
- `ChartSpec` 支持 `overlay_histograms: list[HistogramData]` 配置项，每个 `HistogramData` 绑定独立的 `label`（资源显示名）和 `color`
- 在 GDR 分布渲染阶段（`_run_impl()` 的对应步骤），检测用户选中的条目中是否存在共享同一 `parent_gdr` 的多组数据，若是则合并为一个叠加图表
- 对于 CDF，同样支持叠加曲线

**注意**：不同资源类型的数值量级可能差异很大（如抽卡资源以万计，兑换货币以百计），叠加直方图时需使用**双 X 轴**或**归一化**选项避免某一方被压缩为一条线。默认行为：如果两组数据的量级差异 > 10 倍，自动切换为上下并排（two-panel）而非叠加。

### 3.3 跨模块：cost_per_draw 按资源类型解析

当前 `cost_per_draw` 从池的 `cost` 字段解析（`resource_search_panel.py:93-107`），格式为 `draw_resource:160`。多资源类型化后：

1. `cost_per_draw` 改为 `Dict[str, float]`（resource_id → 每抽成本）
2. 解析逻辑遍历所有池，按 resource_id 分类收集成本值
3. 多个池同资源类型但成本不同时，取第一个（或显示警告）
4. UI 中「以抽数为单位」复选框的 `QSpinBox` 改为按当前选中的资源类型显示对应成本值

```python
def _extract_cost_per_draw_by_resource(self) -> dict[str, float]:
    """从池配置解析每种资源类型的每抽成本 {resource_id: cost}"""
    costs = {}
    for pool_entry in self.pool_entries:
        cost_str = getattr(pool_entry, 'cost', 'draw_resource:160')
        for part in cost_str.split('|'):
            part = part.strip()
            if ':' in part:
                rid, val = part.split(':', 1)
                if rid not in costs:
                    try:
                        costs[rid] = float(val)
                    except ValueError:
                        pass
    return costs
```

- [ ] **S1.5** 实现注册表展开函数 `expand_resource_gdrs(resource_defs)`

  在 `core/gdr.py` 中新增。接收 `resource_defs: dict[str, str]`（如 `{'draw_resource': '抽卡资源', 'exchange_currency': '兑换货币'}`），为每个资源类 GDR 生成 `{gdr_key}:{resource_id}` 条目。

  展开逻辑：
  ```python
  RESOURCE_GDR_KEYS = {'resource_remaining', 'resource_consumed', 'resource_per_card', 
                        'resource_efficiency'}
  # 注：draw_conversion_efficiency 不在此集合中，保持硬编码 draw_resource
  
  def expand_resource_gdrs(resource_defs: dict[str, str]) -> dict[str, GDRDefinition]:
      expanded = {}
      for gdr_key, defn in UNIFIED_GDR_REGISTRY.items():
          expanded[gdr_key] = defn  # 保留原始条目（向下兼容）
          if gdr_key in RESOURCE_GDR_KEYS:
              for rid, rname in resource_defs.items():
                  new_key = f'{gdr_key}:{rid}'
                  new_defn = GDRDefinition(
                      key=new_key,
                      display_name=f'{defn.display_name}({rname})',
                      default_threshold=defn.default_threshold,
                      compute_from_compact=_make_resource_gdr(
                          defn.compute_from_compact, resource_id=rid
                      ),
                      lower_is_better=defn.lower_is_better,
                      category=defn.category,
                      parent_gdr=gdr_key,  # 新增字段，用于 UI 分组和叠加检测
                  )
                  expanded[new_key] = new_defn
      return expanded
  ```

  > **修订说明（2026-05-28）**：原方案在 UNIFIED_GDR_REGISTRY 构建阶段静态展开。考虑到 `resource_defs` 来自 `ConfigStore`（运行时加载），展开应在 `ConfigStore` 加载后、GDR 面板初始化前调用。或者改为惰性展开：`get_expanded_registry(resource_defs)` 缓存结果。

- [ ] **S1.6** 修改 `compute_gdr_from_compact()` 集成 `_parse_gdr_key`
