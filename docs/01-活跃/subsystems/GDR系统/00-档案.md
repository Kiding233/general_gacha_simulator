## 业务问题

抽卡模拟的结果是多维度的——玩家关心的不只是「抽到了几张目标卡」，还包括资源消耗、保底触发频率、抽卡效率、收集完成度等。传统的单一指标（如「总抽数」或「目标卡数量」）无法全面衡量一次抽卡结果的优劣，更无法在不同策略之间进行公平比较。

需要一个**统一的指标体系**，将多维度的抽卡结果映射到可比较的标量值，使得：

1. 不同策略的效果可以在同一尺度下对比
2. 用户可以自定义「什么是好的抽卡结果」（通过 GDR 阈值、权重配置）
3. 所有分析面板共享同一套计算逻辑，避免重复实现和口径不一致

## 设计意图

### 核心抽象

- **GDR（广义出率，Generalized Drop Rate）**：将一次完整模拟的 `CompactResult` 映射为一个标量值（float）的函数。每个 GDR 捕捉抽卡结果的一个特定维度。
- **GDRContext**：封装 GDR 计算所需的全部静态参数（目标卡规格、SSR ID 集合、初始资源、每日资源收入、收集集合、武器-角色映射表），将「计算什么」与「如何计算」解耦。
- **UNIFIED_GDR_REGISTRY**：所有 GDR 指标的单一注册表（dict），消除散落的硬编码列表。每个条目包含 key、display_name、default_threshold、compute_from_compact、compute_from_history、category、lower_is_better 等完整元数据。
- **SuccessChecker**：统一的成功/失败判定器。对任一 GDR 指标，判断单次模拟的 GDR 值是否达到阈值（支持 `lower_is_better` 反向判定）。被所有分析面板复用。

### 两种计算路径

| 路径 | 函数 | 时间复杂度 | 用途 |
|------|------|-----------|------|
| 紧凑路径（主流） | `compute_from_compact(compact, target_specs, ...)` | O(1) | 从 `CompactResult` 字典直接读取聚合字段，主流路径 |
| 历史路径（兼容） | `compute_from_history(history, ctx: GDRContext)` | O(T) | 逐抽遍历 `InfoVector` 列表，兼容旧代码 |

两种路径对同一指标的计算结果在数学上等价。

### 指标注册表（17 项）

实际代码（`core/gdr.py`）中 `UNIFIED_GDR_REGISTRY` 含 17 个条目（注意：部分历史文档记载为 13 个，因早期版本未包含 `resource_consumed`、`resource_per_card`、`target_card_draws`、`draw_conversion_efficiency`）。

#### 目标达成类（值域 [0,1]）

| key | display_name | default_threshold | 说明 |
|-----|-------------|-------------------|------|
| `target_achievement` | 简单目标达成率 | 1.0 | 加权数量达成率：`Σ min(got_i, target_i) / Σ target_i` |
| `target_collection` | 目标卡收集率 | 1.0 | 种类收集率：`count(card_id: got > 0) / len(target_specs)` |
| `all_targets` | 抽出全部目标卡 | 1.0 | 二元判断：全部目标达到要求数量 → 1.0，否则 0.0 |
| `ssr_collection` | SSR收集率 | 0.05 | SSR 种类收集率：`collected_ssr / total_ssr` |

#### 资源效率类

| key | display_name | default_threshold | lower_is_better | 说明 |
|-----|-------------|-------------------|-----------------|------|
| `resource_remaining` | 资源剩余 | 0.0 | 否 | 最终资源剩余量：`initial + gained - consumed` |
| `resource_consumed` | 资源消耗 | 0.0 | **是** | 总资源消耗量 |
| `resource_efficiency` | 目标卡出卡效率 | 0.01 | 否 | 每消耗 1 单位资源获得的目标卡数：`achieved / consumed` |
| `resource_per_card` | 每目标卡资源消耗 | 0.0 | **是** | 获得一张目标卡的平均资源消耗：`consumed / obtained`（obtained=0 时返回 inf） |
| `extra_target` | 额外目标卡 | 0.0 | 否 | 超出目标数量的额外获得数 |
| `non_pity_draws` | 非保底抽卡数 | 0.0 | 否 | 非保底触发的抽卡次数 |
| `pity_draws` | 保底抽卡数 | 0.0 | 否 | 保底触发的抽卡次数 |

#### 加权综合类（需用户配置权重）

| key | display_name | default_threshold | needs_weights | 说明 |
|-----|-------------|-------------------|---------------|------|
| `weighted_satisfaction` | 加权满意度 | 0.0 | desire+miss_cost | `Σ got×desire - missed×miss_cost`（可加拟线性效用） |
| `total_card_value` | 总出卡价值 | 0.0 | card_value | `Σ got_i × card_value_weight(card_id)`（非目标卡默认权重 1.0） |
| `per_pool_draw_rate` | 每池下池出卡率 | 0.0 | — | 每个有抽卡的池中目标卡出数均值 |
| `weapon_character_ratio` | 专武角色比 | 0.0 | needs_weapon_map | 专武获得数 / 对应角色获得数（无配置入口时始终返回 0.0） |

#### 辅助/内部指标

| key | display_name | default_threshold | 说明 |
|-----|-------------|-------------------|------|
| `target_card_draws` | 目标卡出数 | 1.0 | 目标卡总出现次数（不去重），用于 per_pool_target 判定 |
| `draw_conversion_efficiency` | 抽数转化效率 | 0.0 | 实际抽数 × cost / 资源消耗。排除兑换池和资源池的抽数，仅计入抽卡池 |

## 技术实现

### 调用链

```
面板 Worker._run_impl()
  → compute_gdr_from_compact(compact_or_aggregate, target_specs, gdr_key, ...)
    → UNIFIED_GDR_REGISTRY[gdr_key].compute_from_compact(compact, target_specs=..., **extra_kwargs)
      → _gdr_xxx(compact, target_specs, ...)  # 各指标的具体实现
  → SuccessChecker(target_specs, gdr_key, gdr_threshold, ...)
    → SuccessChecker.is_success(compact)
      → compute_gdr_from_compact(...) >= threshold (或 <= 若 lower_is_better)
```

### 核心数据结构

| 类 | 职责 |
|----|------|
| `GDRContext` | 不可变 dataclass，封装 GDR 计算的静态参数：`target_specs`、`ssr_ids`、`all_drawable_ids`、`initial_resources`、`resource_gain_per_day`、`collection_sets`、`weapon_character_map` |
| `GDRDefinition` | NamedTuple，描述单个 GDR 指标：`key`、`display_name`、`default_threshold`、`compute_from_compact`、`compute_from_history`、`category`、`lower_is_better`、`needs_ssr_ids`、`needs_weapon_map`、`needs_weights` |
| `WeightedGDRConfig` | 权重配置包装器：`weight_config` + `scheme_map` → `get_weight_function(gdr_name)` |
| `SuccessChecker` | 成功判定器：封装 `gdr_key`、`gdr_threshold`、`lower_is_better` 和所有权重参数。提供 `is_success(compact)`、`check_batch(results)`、`from_registry()` 工厂方法 |

### 核心函数

| 函数 | 用途 |
|------|------|
| `compute_gdr_from_compact(compact, target_specs, gdr_key, ...)` | 紧凑路径入口，按 key 从注册表查找定义并调用 compute |
| `compute_all_gdr(history, ctx)` | 历史路径批量计算所有 GDR |
| `compute_weighted_gdr(history, ctx, weight_config, miss_cost_func)` | 历史路径加权 GDR（加权目标达成率、加权额外卡、遗憾指数、总出卡价值、加权满意度） |
| `compute_success_probability(histories, target_specs, gdr_key, ...)` | 批量计算成功率（成功次数 / 总次数） |
| `populate_gdr_combo(combo)` | 填充 GUI 下拉框：遍历注册表，按 `lower_is_better` 添加 `(-)` 前缀 |
| `get_default_threshold(gdr_key)` | 查询指标的默认阈值 |
| `register_gdr(definition, overwrite=False)` | 动态注册新 GDR 指标（支持扩展） |
| `compute_gdr_from_cumulative(cum_snapshot, ...)` | 从累积快照构造伪 compact 并计算 GDR（用于逐池/逐时间分析） |
