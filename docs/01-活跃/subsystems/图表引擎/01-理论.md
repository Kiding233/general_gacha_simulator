# 图表引擎 -- 理论

> 创建日期：2026-06-02（首次建档）
> 状态：P11 迁移已完成，无已知理论缺陷

---

## 一、核心假设

### 1.1 数据与渲染分离

`ChartSpec` 作为中间表示（Intermediate Representation, IR），不依赖任何绑图库（`matplotlib`、`plotly`、`seaborn` 均不 import）。面板层只负责将分析数据打包为 `ChartSpec`，渲染层负责将 `ChartSpec` 转换为具体绑图库的输出。

**好处**：
- 面板代码完全不接触 Plotly API，降低耦合
- 更换渲染后端（如从 Plotly 换到 ECharts）只需替换 `PlotlyRenderer`，面板代码无需修改
- `ChartSpec` 可序列化为 JSON，支持进程间传输（未来可能）

### 1.2 单渲染后端假设

当前假设只存在一个渲染后端（Plotly）。`PlotlyRenderer` 是 `ChartSpec → go.Figure` 的**唯一转换点**。全项目只有 `plotly_charts.py` 一个文件 `import plotly`。

此假设简化了架构——不需要渲染后端的注册/发现机制。如果未来需要多种后端（如 Plotly + ECharts + D3），需要引入渲染后端注册表。

### 1.3 WebView 生命周期假设

`ChartWebView` 假设：
1. 每个面板持有自己的 WebView 实例（而非全局共享）
2. WebView 在面板 `__init__` 时创建，在面板销毁时自动回收
3. 预加载的骨架页在整个面板生命周期内有效，不会因页面跳转失效

由于 PyQt6 WebEngine 的进程模型，多个 WebView 实例共享后台的 Chromium 渲染进程，不会产生 N 个独立浏览器进程的开销。

## 二、中间表示设计的理论基础

### 2.1 数据类型化

每种图表类型对应一个专属的 `*Data` dataclass（如 `HistogramData`、`CDFData`），而非使用通用的 `Dict[str, Any]`。好处：

- **类型安全**：`PlotlyRenderer.build_figure()` 通过 `isinstance` 或模式匹配处理各类型，不会把直方图数据误传给 CDF 渲染函数
- **IDE 支持**：自动补全和类型检查
- **文档自描述**：dataclass 字段和类型注解即文档

### 2.2 注解系统独立于数据

`ChartAnnotation` 和 `ShadedRegion` 独立于 `ChartData`，存储在 `ChartSpec` 层面。这意味着：

- 同一数据可以搭配不同的注解（如不同阈值下的参考线）
- 注解不侵入数据类型定义——`HistogramData` 不需要知道「是否有一条阈值线」

### 2.3 layout_hints 的开放性

`ChartSpec.layout_hints: dict` 提供非结构化的渲染提示（如 `figsize`、`nbins`、`color`、`bin_edges`），作为类型化 `*Data` 的补充。这平衡了「严格类型化」与「灵活性」——常见需求走类型化字段，小众需求走 `layout_hints`。

## 三、图表切换的零重载策略

传统多图表管理方案（WebView 每次 `load()` 新 HTML）有 ~200-500ms 延迟和视觉闪烁。当前方案：

1. **单 HTML 文件**：所有图表数据内嵌在同一个 HTML 中
2. **Tab 栏 + 面板**：每个图表在一个 `<div class="chart-panel">` 中
3. **JS 切换**：`display: block/none`——浏览器 DOM 操作，<1ms 延迟，无重绘
4. **增量更新**：`runJavaScript('rebuildChart("tab_name", figureJson)')`——只更新单个图表的数据和 DOM，不触动其他图表

**与 QTabWidget 方案的对比**：

| | QTabWidget | 当前方案 |
|---|---|---|
| 切换延迟 | ~50-100ms（Widget 显示/隐藏） | <1ms（DOM display） |
| 并发图表数 | 所有 Tab 同时渲染 | 仅可见图表渲染（`display:none` 的图表 Plotly 不渲染） |
| 增量更新 | 需重新生成 Widget | `runJavaScript` 原地更新 |

## 四、与外部理论的对照

| 概念 | 文献 | 一致性 |
|------|------|--------|
| 中间表示 (IR) | 编译器设计中的 IR（如 LLVM IR） | ✅ 数据与渲染完全解耦 |
| 数据驱动 UI | React/Vue 单向数据流 | ✅ ChartSpec → Figure → DOM 单向管道 |
| 渐进增强 | Web 设计原则 | ✅ 骨架页预加载 → JS 填充数据 |
| 单一职责 | SOLID 原则 | ✅ ChartSpec（数据）、PlotlyRenderer（转换）、ChartWebView（显示）各司其职 |
| 开放-封闭 | SOLID 原则 | ✅ 新增图表类型：扩展 `ChartData` union + switch case，无需修改现有类型 |
