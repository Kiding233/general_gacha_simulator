## 业务问题

抽卡模拟的核心决策问题：在每个时间步，给定当前状态（资源、已获得卡片、保底计数器、可用池子），应该执行什么行动（抽哪个池、兑换什么卡、还是等待）？

不同的玩家有不同的偏好和约束——有的追求「抽到目标卡就收手」，有的希望「最大化资源效率」，有的采用「固定预算分配」。需要一个**可组合的策略系统**，允许用户选择不同风格的决策规则，并评估它们在各种场景下的效果。

## 设计意图

### 核心抽象

- **Strategy（抽象基类）**：所有策略的统一接口 `select_action(self, ctx: StrategyContext) -> Action`。策略不维护可变运行时状态——所有状态由 `StrategyContext` 传入。
- **StrategyContext（dataclass）**：封装策略所需的全部信息快照。包含 `state`（当前资源/时间）、`current_pools`（当前可用池）、`all_pools`（全部池）、`future_schedules`（未来池时间表）、`target_cards`（目标卡集合）、`stop_condition`（停止条件）、保底引擎/状态（`_pity_engine`/`_pity_state`）、运行时历史（`acquired`、`pool_draw_counts`、`total_draws`、`ssr_ids`）、保底概率缓存（`_pity_cache`）。
- **STRATEGY_REGISTRY**：所有策略的单一点注册表（dict），每个条目包含 `display_name`、`class`、`params`、`description`。GUI 策略下拉列表从此注册表自动生成。
- **Action**：统一行动表示——`DRAW(pool_id)`、`EXCHANGE(pool_id, item_id, quantity)`、`WAIT`。

### 7 种注册策略

| key | display_name | 决策逻辑 | 特征 |
|-----|-------------|---------|------|
| `smart` | 按需追卡 | 优先兑换 → 按目标追卡 → 等待 | 综合启发式，含兑换优先、资源检查、目标导向 |
| `pool_quota` | 指定池配额 | 每池指定抽卡上限后切换 | 预算分配型，参数为 `quotas: Dict[pool_id, max_draws]` |
| `pity_reserve` | 保底预留 | 只在保底概率 ≥ 阈值时抽卡 | 风险规避型，唯一使用 `get_pity_probabilities()` 的策略 |
| `stop_on_target` | 目标即停 | 抽到 up/目标卡后停止 | 简短型，参数为 `stop_on_up: bool` |
| `target_hunting` | 指定池追卡 | 只在指定池子抽卡 | 专注型，参数为 `target_pool_ids: List[str]` |
| `fixed_count` | 固定次数 | 抽满固定次数后等待 | 简单型，参数为 `max_draws: int` |
| `draw_target` | 指定目标抽卡 | 从指定池抽卡直到获得目标 | 内部策略（`internal: True`），不在 GUI 策略列表显示 |

### 工厂函数

| 函数 | 用途 |
|------|------|
| `create_strategy(name, params)` | 从注册表构建策略实例（参数自动填充默认值） |
| `strategy_type_to_key(display_name)` | 中文显示名 → 注册表 key（用于 GUI 下拉选择） |
| `strategy_key_to_type(key)` | 注册表 key → 中文显示名（用于展示） |

## 技术实现

### 调用链

```
GachaService.run_simulation()
  → strategy.select_action(ctx)
    → StrategyContext 传入当前状态快照
    → 策略返回 Action(DRAW/EXCHANGE/WAIT)
  → GachaService 执行 Action → 更新状态 → 下一轮
```

### 核心数据结构

| 类 | 职责 |
|----|------|
| `StrategyContext` | 不可变快照 dataclass，封装策略决策所需全部信息。含惰性 `get_pity_probabilities(pool_id)` 方法，仅 `PityReserveStrategy` 调用 |
| `Strategy (ABC)` | 抽象基类，定义 `select_action(ctx) -> Action` 接口 |
| `Action` | 简单值类型（`DRAW`/`EXCHANGE`/`WAIT`），由 `core/action.py` 定义 |
| `TargetCardSet` | 目标卡规格集合，由 `core/target_card.py` 定义 |

### 关键设计决策

1. **策略无状态**：`acquired`、`pool_draw_counts`、`total_draws` 等运行时历史由 `StrategyContext` 传入，策略本身不维护 mutable 状态。保证策略可被多个模拟实例复用。
2. **保底引擎与策略解耦**：保底概率查询通过 `StrategyContext.get_pity_probabilities()` 间接访问，策略不直接持有 `PityEngine` 引用。
3. **`future_schedules` 已传入但未被任何策略使用**（缺陷 14）：当前所有策略仅做一阶前瞻，不评估未来池的价值。

### 实施历史

- **v1.0-v1.7**：策略逐步增加，从硬编码 if-else 演进到注册表模式
- **v1.8**：引入 `StrategyContext` dataclass 统一参数传递，消除策略对全局变量的依赖
- **v1.9**：`draw_target` 策略从 `target_hunting` 分拆，标记为 internal
- **当前状态**：7 种策略全部可用，注册表机制完善。已知理论缺陷 3 项（P30——启发式文档化 + 统一效用框架 + 信息完备性）
