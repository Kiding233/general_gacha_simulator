## 三、StrategyContext 详析

### 字段一览

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `state` | `GachaState` | `GachaService` 当前状态 | 资源、时间 |
| `current_pools` | `List[Pool]` | 排期管理器 | 当前开放池列表 |
| `all_pools` | `List[Pool]` | 全局配置 | 全部池（含历史/未来） |
| `future_schedules` | `List[PoolSchedule]` | 排期管理器 | 未来池时间表（**当前无策略使用**） |
| `target_cards` | `TargetCardSet` | 目标配置 | 目标卡 id→quantity |
| `stop_condition` | `StopCondition` | 配置 | 停止条件实例 |
| `_pity_engine` | `Optional[PityEngine]` | Worker 全局变量 | 保底引擎（repr=False） |
| `_pity_state` | `Optional[PityState]` | Worker 全局变量 | 保底计数器状态（repr=False） |
| `acquired` | `Dict[str, int]` | `GachaService` 累加 | 已获得的有效卡（不含 `_no_card`） |
| `pool_draw_counts` | `Dict[str, int]` | `GachaService` 累加 | 每池抽卡次数 |
| `total_draws` | `int` | `GachaService` 累加 | 总抽卡次数 |
| `last_draw_pity_triggered` | `bool` | 上一抽结果 | 上一抽是否触发保底 |
| `ssr_ids` | `Set[str]` | 配置 | SSR 卡片 ID 集合 |
| `_pity_cache` | `Dict[str, Dict[str, float]]` | 惰性计算 | 保底概率缓存（repr=False） |

### 惰性方法

`get_pity_probabilities(pool_id) -> Dict[str, float]`：
- 首次调用时从 `_pity_engine` 计算并缓存
- 后续调用直接返回缓存值
- 仅 `PityReserveStrategy` 调用此方法
- 兑换池返回空 dict

## 四、策略决策流程详解

### SmartStrategy（按需追卡）——最复杂的策略

```
1. 如有目标卡可在兑换池使用兑换货币兑换 → EXCHANGE
2. 遍历 current_pools:
   a. 跳过兑换池、已关闭池
   b. 检查池中是否包含未完成的目标卡
   c. 检查是否有足够的抽卡资源
   d. 选择第一个满足条件的池 → DRAW
3. 无可抽池 → WAIT
```

### PityReserveStrategy（保底预留）——唯一使用保底概率的策略

```
1. 对 target_pool_ids 中每个当前可用的池:
   a. ctx.get_pity_probabilities(pool_id) 获取调整后概率
   b. 检查目标卡概率是否 ≥ threshold
   c. 如是 → 选择该池 → DRAW
2. 无池满足概率阈值 → WAIT
```

### 其他策略（简化逻辑）

| 策略 | 核心决策 |
|------|---------|
| `PoolQuotaStrategy` | `pool_draw_counts[pid] < quotas[pid]` → DRAW；否则跳过 |
| `StopOnTargetStrategy` | 当前池含 up/目标卡 → DRAW；所有池不含 → WAIT |
| `TargetHuntingStrategy` | 按 `target_pool_ids` 顺序遍历，第一个可用且有资源的池 → DRAW |
| `FixedCountStrategy` | `total_draws < max_draws` → 按第一个可用池 DRAW；否则 WAIT |
| `DrawTargetStrategy` | 在 `target_pool_id` 中抽卡直到获得 `target_card_id` |

## 五、扩展机制

### 添加新策略

1. 在 `core/strategy.py` 实现策略类（继承 `Strategy`，实现 `select_action(ctx)`）
2. 在 `STRATEGY_REGISTRY` 添加条目（`display_name`、`class`、`params`、`description`）
3. GUI 策略下拉列表自动包含新策略（通过 `populate_strategy_combo()` 遍历注册表）

### 约束

- 策略不得维护可变实例状态——所有运行时历史必须从 `ctx` 读取
- 策略不得直接访问全局变量（`_wk_*`）——所有信息通过 `StrategyContext` 传入
- 策略返回的 `Action` 必须有效（池 ID 存在、资源充足等）——`GachaService` 会验证
