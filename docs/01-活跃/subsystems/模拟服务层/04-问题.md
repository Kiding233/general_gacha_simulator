## P1 — 高优先级

### P12 Phase 2：Numba 数值化内核

- **来源**：`模拟性能优化方案.md`，Phase 2
- **状态**：未开始
- **目标**：将主模拟循环转为 `@njit` 编译的 numpy 数组操作，预期 10-25x 加速
- **关键挑战**：策略决策需回调 Python 层、ID 映射系统需重写、逐抽数组需在 Numba 内核中填充
- **依赖**：Phase 1/1.5（已完成）
- **预计工期**：1-2 周

### P12 Phase 3：共享内存替代 Pickle

- **来源**：`模拟性能优化方案.md`，Phase 3
- **状态**：未开始
- **目标**：使用 `multiprocessing.shared_memory` 将 CompactResult 写入共享缓冲区，避免 pickle 序列化开销（当前占 40-60% 总耗时）
- **关键挑战**：Windows 上共享内存需显式 `close()` + `unlink()`，子进程崩溃时泄漏风险
- **依赖**：Phase 1.5（已完成，精简传回已减少 93% 传输量）
- **预计工期**：3-5 天

### P4 Task 1：自适应停止机制

- **来源**：`自适应模拟与方差缩减计划.md`，Task 1
- **状态**：未开始（设计完成，含 TDD 测试用例）
- **目标**：实时 RSE 监控 → 自动停止，替代手动猜测模拟次数
- **关键组件**：`AdaptiveSimController`（新建）→ `run_batch_parallel` 新增 `stop_condition` 回调 → `GachaPanel` 自适应精度 UI
- **核心缓解**：连续两次检查均达标才停止（`_consecutive_passes >= 2`），缓解 RSE 停止偏误
- **依赖**：P23 scipy（已完成）、P3 Bootstrap（核心完成）
- **预计工期**：2-3 天

## P2 — 低优先级

### P4 Task 2：对偶变量法

- **来源**：`自适应模拟与方差缩减计划.md`，Task 2
- **状态**：暂缓
- **目标**：通过单次模拟内 `(u, 1-u)` 配对降低方差（1.2-2x）
- **暂缓原因**：需改造 `Pool.draw()` 随机数注入管线（侵入性大），自适应停止后边际收益降低
- **复评条件**：自适应停止上线后，若用户仍要求更高精度且不接受增加模拟次数

### 单进程路径全局变量污染

- **来源**：`01-理论.md`，已知局限性 #2
- **状态**：已识别，未修复
- **触发条件**：`max_workers <= 1` 时，`_wk_init()` 在主进程中覆盖模块级全局变量
- **影响范围**：CLI 模式、单次 GUI 模拟
- **建议修复**：单进程路径在 `_wk_init()` 前保存旧值，完成后恢复

### 多面板 CPU 争抢

- **来源**：`01-理论.md`，已知局限性 #3
- **状态**：已识别，未设计解决方案
- **影响**：多个面板同时运行时无全局调度
- **可能方案**：全局 `WorkerPoolManager` 单例，共享进程池 + 任务队列
