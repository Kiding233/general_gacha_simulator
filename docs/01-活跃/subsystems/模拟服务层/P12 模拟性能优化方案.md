# P12 抽卡模拟核心性能优化方案

## 背景

当前模拟引擎（Python + multiprocessing）：
- **CLI 模式**：1000 样本耗时 ~1.6 秒（4 核）
- **GUI 模式**：1000 样本耗时 **10-15 秒**（18 核）——其中模拟本身仅 ~0.6s，剩余 10-14s 被主进程串行 `DrawSequenceExtractor` 消耗

用户计划扩展到 10万-1000万 样本量级，需要 100-1000x 级别的加速。

**约束条件**：
- 仅本地 Windows 运行（~20 核），但需兼容不同核心数
- 可引入编译型扩展（Numba）但必须 `pip install` 自动安装，不能要求用户手动配置编译工具链
- 保持 GUI、策略系统、停止条件的外部接口不变

## 热路径分析摘要

经过对 `gacha_service.py`、`pool.py`、`pity.py`、`strategy.py`、`batch_simulator.py` 的深入分析：

| 瓶颈 | 详情 |
|------|------|
| Python 对象模型 | 每次迭代 20+ 次 dict 操作，5-10 次对象方法调用 |
| StrategyContext 重建 | 每迭代 `dict(acquired)` + `dict(pool_draw_counts)` 两次浅拷贝 |
| 保底引擎 dict 拷贝 | `before_draw()` 中级联 2-3 次 dict copy |
| per-simulation 重复构建 | `card_def_map`、`targets`、策略对象每次模拟重新创建 |
| 多进程序列化 | `CompactResult` pickle 开销占总耗时 40-60% |
| PityReserve 副作用 bug | `select_action` 中调用 `get_pity_probabilities()` 会污染保底计数器 |
| **Extractor 串行瓶颈**（2026-05-25 新增） | `DrawSequenceExtractor` 在主进程中串行处理每条结果（~10-15ms/条），18 个 worker 的并行优势被主进程的串行 extractor 完全抵消 |

## GUI vs CLI 性能差异分析（2026-05-25）

## Phase 1附加：脆弱性分析向量化（纯numpy替代statsmodels，半天）✅ 已完成（2026-05-25）

### 瓶颈定位

`vulnerability.py` 第 72-88 行 `_fit_local_polynomial()` 是脆弱性分析的计算瓶颈。它对每个池子调用两次 `statsmodels.nonparametric.KernelReg.fit()`，statsmodels 的核回归实现并非纯向量化，内部逐点循环，1000 数据点时每池耗时 2-5 秒。**但更严重的是 statsmodels 默认使用交叉验证（`cv_ls`）选择带宽，消耗时间远超拟合本身**——实测每池 14 秒。

**这不是模拟嵌套问题**——脆弱性分析本身零模拟调用，瓶颈纯粹在 statsmodels 的 Python 循环 + CV 带宽选择。实测 5 次 `KernelReg` 构造耗时 70s。

### 优化方案：局部线性回归闭式解 + numpy 矩阵化

局部线性回归有解析闭式解。对评估点 x₀，解加权最小二乘：

\[\min_{\beta_0, \beta_1} \sum w_i(y_i - \beta_0 - \beta_1(x_i - x_0))^2, \quad w_i = e^{-\frac{1}{2}\left(\frac{x_i-x_0}{h}\right)^2}\]

矩阵形式 β = (XᵀWX)⁻¹XᵀWy 展开得：

\[\beta_0 = \frac{S_2 \sum w_i y_i - S_1 \sum w_i(x_i-x_0)y_i}{S_0 S_2 - S_1^2}, \quad S_k = \sum w_i(x_i-x_0)^k\]

关键洞察：构建核权重矩阵 **K** = [exp(-½((x_i - r_j)/h)²)]，形状 (n_grid, n_data)，然后所有 S_k 和加权和可通过矩阵乘法一次算出——200 个网格点 × 1000 数据点一次矩阵运算完成，而非逐点循环。

**数学严谨性**：此闭式解与 statsmodels `KernelReg(reg_type='ll')` 内部使用的公式完全等价。验证：相同带宽下，新实现与 statsmodels 输出的 lp_curve 逐元素差异 < 5e-12。

**带宽选择更正**：计划初版误认为 statsmodels 默认使用 Silverman 规则（`'normal_reference'`）。实际上 statsmodels 默认使用交叉验证（`'cv_ls'`），这会消耗大量时间且对二值响应变量（failure=0/1）产生过大的带宽（~2.87e8），将条件概率平滑为接近常数，丧失脆弱性检测能力。新实现使用 **Silverman 自适应带宽** `h = 1.06·σ·n^(-1/5)`，既快速又能保留局部变异，更有利于脆弱性区间检测。

### 1.2 Extractor 移入 Worker 进程 → `batch_simulator.py` + `streaming.py`

**核心思路**：将 `DrawSequenceExtractor` 和 `extract_aggregate` 的执行从主进程移入 worker 进程。Worker 完成模拟后立即在本地执行提取，只将**聚合后的提取结果**传回主进程。主进程仅负责合并各 worker 的局部结果。

#### 1.2.1 架构变更

**当前**：
```
Worker: 模拟 → CompactResult → pickle → 主进程 → extractor(compact) → 存入列表
```

**优化后**：
```
Worker: 模拟 → CompactResult → extractor(compact) → WorkerLocalResult → pickle → 主进程 → merge()
```

#### 1.2.2 具体改动

**`streaming.py`** — 新增 `WorkerLocalExtractor`：

```python
class WorkerLocalExtractor:
    """在 worker 进程内执行的提取器——与 SharedResultCollector 功能等价但无需跨进程通信。"""
    
    def __init__(self, pool_end_times, target_ids, ssr_ids, target_specs,
                 initial_resources, n_heatmap_bins=50, max_keep=200):
        # 预计算不变数据（bin edges 等）
        self._pool_end_times = pool_end_times
        self._target_ids = target_ids
        self._ssr_ids = ssr_ids
        self._target_specs = target_specs
        self._initial_resources = initial_resources
        self._n_heatmap_bins = n_heatmap_bins
        self._max_keep = max_keep
        # 初始化局部累加器
        self._heatmap_ach = np.zeros((n_heatmap_bins,), dtype=np.int32)
        self._heatmap_res = np.zeros((n_heatmap_bins,), dtype=np.int32)
        self._cumulative = {}  # pool_id → list of snapshots
        self._transitions = []
        self._kept = []
        self._aggregates = []
        self._n = 0
        # 预计算 bin edges（所有 worker 使用相同的 edges）
        max_resource = max(initial_resources.get('draw_resource', 0), 1.0) * 2
        self._ach_bins = np.linspace(0, 1.05, n_heatmap_bins + 1)
        self._res_bins = np.linspace(0, max_resource, n_heatmap_bins + 1)
    
    def process(self, compact: dict) -> None:
        """在 worker 进程中调用，处理单次模拟结果。"""
        if compact is None:
            return
        self._n += 1
        # 1. extract_aggregate（纯计算，无副作用）
        agg = extract_aggregate(compact)
        self._aggregates.append(agg)
        # 2. 热力图更新
        self._update_heatmap(compact)
        # 3. 累积快照更新
        self._update_cumulative(compact)
        # 4. 转变标记更新
        self._update_transition(compact)
        # 5. 保留序列
        if len(self._kept) < self._max_keep:
            self._kept.append(_extract_sequence_subset(compact))
    
    def to_result(self) -> dict:
        """序列化为可通过 pickle 传回主进程的结果。"""
        return {
            'aggregates': self._aggregates,
            'heatmap_ach': self._heatmap_ach,
            'heatmap_res': self._heatmap_res,
            'cumulative': self._cumulative,
            'transitions': self._transitions,
            'kept': self._kept,
            'n': self._n,
        }
```

**主进程合并逻辑**（新增 `merge_worker_results`）：

```python
def merge_worker_results(worker_results: List[dict], heatmap_bins: dict) -> dict:
    """合并多个 worker 的局部提取结果。"""
    merged_aggregates = []
    merged_heatmap_ach = None
    merged_heatmap_res = None
    merged_cumulative = {}
    merged_transitions = []
    merged_kept = []
    total_n = 0
    
    for wr in worker_results:
        merged_aggregates.extend(wr['aggregates'])
        merged_transitions.extend(wr['transitions'])
        merged_kept.extend(wr['kept'])
        total_n += wr['n']
        
        if merged_heatmap_ach is None:
            merged_heatmap_ach = wr['heatmap_ach'].copy()
            merged_heatmap_res = wr['heatmap_res'].copy()
        else:
            merged_heatmap_ach += wr['heatmap_ach']
            merged_heatmap_res += wr['heatmap_res']
        
        for pool_id, snapshots in wr['cumulative'].items():
            merged_cumulative.setdefault(pool_id, []).extend(snapshots)
    
    # 截断保留序列
    if len(merged_kept) > 200:
        merged_kept = merged_kept[:200]
    
    return {
        'aggregates': merged_aggregates,
        'heatmap': {'achievement': merged_heatmap_ach, 'resource': merged_heatmap_res},
        'cumulative': merged_cumulative,
        'transitions': merged_transitions,
        'kept': merged_kept,
        'n': total_n,
    }
```

### 1.3 精简 `_update_cumulative`：单次遍历替代多池遍历 → `streaming.py`

**问题**：`_update_cumulative()` 对每个池子独立遍历全部抽卡记录，总迭代次数 = `N_pools × N_draws`（典型：8 × 300 = 2400 次/结果）。

**优化**：改为单次遍历，在遍历过程中同时维护所有池子的累积状态。按时间排序的池子可转化为时间区间 `(prev_end, cur_end]`，一次遍历中识别当前抽卡属于哪个区间。

```python
def _update_cumulative_single_pass(self, compact):
    """单次遍历构建所有池子的累积快照。"""
    card_ids = compact.get('draw_card_ids', [])
    times = compact.get('draw_times', [])
    # ... 其他数组 ...
    
    sorted_pools = sorted(self._pool_end_times.items(), key=lambda x: x[1])
    pool_idx = 0
    # 每个池子的当前累积状态
    snapshots = []  # (pool_id, 累积状态) 的列表
    
    for i in range(len(card_ids)):
        t = times[i] if i < len(times) else float('inf')
        # 检查是否越过了某个池子的结束时间
        while pool_idx < len(sorted_pools) and t > sorted_pools[pool_idx][1]:
            pid = sorted_pools[pool_idx][0]
            # 保存当前池子的快照
            self._cumulative_snapshots.setdefault(pid, []).append(current_snapshot)
            pool_idx += 1
        # 更新当前累积状态
        # ...
```

**预期**：`_update_cumulative` 耗时从 ~8ms/结果 降至 ~2ms/结果（4x 加速），与 1.2 组合后进一步降低 worker 内开销。

**风险**：逻辑从「每池独立遍历」变为「单次遍历 + 池间切换」，需仔细处理边界（抽卡恰好发生在池子切换时间点）。可通过对比测试验证新旧实现输出一致。

### 1.4 按需加载 DrawSequenceExtractor → `gacha_panel.py`

**问题**：`GachaPanel` 无条件创建完整的 `DrawSequenceExtractor`（热力图 + 累积快照 + 转变标记），即使当前用户只想看「快速结果预览」表格中的均值/中位数。这些数据仅在「过程分析」面板中使用。

**优化**：延迟创建 extractor，仅当用户切换到需要详细数据的分析 Tab 时才启用。

```python
# gacha_panel.py
class GachaPanel(QWidget):

        self._enable_sequence_extraction = False  # 默认关闭
    
    def set_sequence_extraction_enabled(self, enabled: bool):
        """由 MainWindow 调用，根据用户是否进入过程分析面板决定。"""
        self._enable_sequence_extraction = enabled
```

`MainWindow` 监听 Tab 切换事件，仅当切换到「过程分析」Tab 时启用：

```python
def _on_tab_changed(self, index):
```

**预期**：

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 只看快速预览 | 10-15s（含 extractor） | **1-2s**（无 extractor 开销） |
| 切换到过程分析 | — | 需重新模拟或缓存已有结果 |

**风险**：如果用户先跑模拟再看过程分析，需要重新模拟。缓解：在 `result_bundle` 中保留原始 compact 列表（当前已保留前 200 条），过程分析面板可基于这些数据重新计算。

### 1.6 CompactResult 传输体积优化 → `batch_simulator.py`

**问题**：每个 `CompactResult` 包含全部逐抽数组（`draw_card_ids`、`draw_pool_ids`、`draw_times` 等），每条 ~300 个元素 × 6 个数组 = ~1800 个值。1000 条结果 → ~180 万个值通过 pickle 从 worker 传到主进程。这是 Phase 3.1（共享内存）的动机，但在 Phase 1.5 中可以先做轻量优化。

**优化**：当 extractor 已在 worker 内执行（1.2），且调用方不需要逐抽序列时，worker 可**不传回完整的逐抽数组**，只传回聚合字段：

```python
# Worker 内执行 extractor 后，返回精简版结果
def _wk_run_single_lightweight(args):
    # ... 模拟 ...
    compact = service.run_simulation_compact(state)
    if _wk_extractor is not None:
        _wk_extractor.process(compact)
        # 只返回聚合字段（远小于完整 compact）
        return {
            'card_counts': compact['card_counts'],
            'total_draws': compact['total_draws'],
            'total_consumed': compact['total_consumed'],
            'final_resources': compact['final_resources'],
            'pity_triggers': compact['pity_triggers'],
            'final_time': compact['final_time'],
            'pool_draw_counts': compact['pool_draw_counts'],
            'pool_end_resources': compact['pool_end_resources'],
        }
    return compact  # 回退：返回完整 compact
```

**预期**：每个结果的 pickle 大小从 ~15KB 降至 ~1KB，减少 ~93% 的传输开销。

## Phase 2：Numba 数值化内核（主力加速，1-2 周）

### 核心理念

将模拟的内部状态（概率、计数器、资源、时间）转为 numpy 数组，核心循环用 `@njit` 编译。策略决策保留在 Python 层（通过轻量回调）。

### 2.1 ID 映射系统

新建 `gacha_simulator/core/id_mapping.py`：

```python
@dataclass
class IdMapping:
    pool_ids: List[str]    # pool_id → int index
    card_ids: List[str]    # card_id → int index
    resource_ids: List[str]  # resource_id → int index
    pity_ids: List[str]    # pity_name → int index
    # + 反向 dict 映射
```

### 2.2 数值化数据结构

新建 `gacha_simulator/core/numba_types.py`，定义结构化 numpy dtype：

- `PoolData`: 静态池子信息（可用时间、奖励偏移、成本）
- `RewardData`: 静态奖励信息（卡牌索引、概率、累积概率）
- `PityData`: 保底配置（类型、阈值、目标分布偏移）
- `RuntimeState`: 可变状态（时间、计数器、资源、获得计数）

### 2.3 Numba 内核

新建 `gacha_simulator/core/numba_kernel.py`：

核心函数 `_nk_run_simulation(seed, static_data, runtime, output_buf) -> int`：
- 单次模拟的完整循环，全部 JIT 编译
- 内部：spend、pity_before、draw (二分查找)、pity_after、gain_resource
- 外部：策略决策通过回调返回 `(action_type, pool_idx, wait_duration)`

**随机数**：使用 `np.random` (PCG64)，每个模拟独立 seed。

### 2.4 策略适配

在 `strategy.py` 中新增 `NumericalStrategy` 基类：

```python
class NumericalStrategy(ABC):
    @abstractmethod
    def select_action(self, ctx: NumericalContext) -> Tuple[int, int, float]:
        """返回 (action_type, pool_idx, wait_duration)"""
```

策略兼容性：

| 策略 | 可数值化 | 备注 |
|------|---------|------|
| SmartStrategy | 是 | 基于资源+目标+可用池 |
| PoolQuotaStrategy | 是 | 基于配额+计数 |
| TargetHuntingStrategy | 是 | 基于目标池过滤 |
| FixedCountStrategy | 是 | 基于 total_draws |
| StopOnTargetStrategy | 是 | 基于 acquired_counts |
| PityReserveStrategy | 部分 | 需预计算保底概率 |
| CompositeStrategy | 是 | 组合子策略 |

### 2.5 双路径回退

在 `GachaService.run_simulation()` 中：

```python
    return self._run_numba_simulation(state)
else:
    return self._run_python_simulation(state)  # 原路径
```

Numba 不可用或策略不兼容时自动回退。

### 2.7 兼容性分析

#### 2.7.1 抽卡资源奖励（reward.resources_gained）

**现状**：`Reward.resources_gained` 是配置加载时确定的静态属性，每次抽到该卡**无条件**累加到资源中。不存在「首次抽到额外奖励」机制。

**Numba 内核兼容**：`RewardData` dtype 已包含 `resource_gain_res_idx` 和 `resource_gain_amount` 字段，每次 draw 后无条件累加即可。如果未来添加首次获得奖励，需在 Numba 内核中检查 `acquired_counts[card_idx] == 0` 后应用额外奖励——改动量小。

#### 2.7.2 下游分析系统兼容

**关键结论**：所有分析系统直接从 `CompactResult` 提取数据，**不需要重跑模拟**。Numba 内核只要输出格式匹配的 CompactResult，下游完全无感。

**CompactResult 字段依赖矩阵**：

| 下游组件 | 依赖的字段 | 缺失后果 |
|---------|-----------|---------|
| GDR 指标（全部 13 种） | `card_counts`、`total_draws`、`pity_triggers`、`final_resources`、`pool_draw_counts`、`total_consumed` | 无影响（仅用聚合字段） |
| 过程分析（AA/BB/AB/BA） | `draw_pool_ids`、`draw_card_ids`、`draw_pity`、`draw_pity_names`、`draw_pity_counter_max` | **回退到聚合推断模式**：`_infer_from_aggregate()` 可降级运行，但 `early_hit` 的精确抽数丢失 |
| 抽取序列可视化 | `draw_card_ids`、`draw_pool_ids`、`draw_times`、`draw_pity` | **时序图/瀑布图/热力图无数据** |
| 转变分析 | `draw_times`、`draw_pool_ids` | **累积快照缺失，转变分析静默跳过**（已知问题 #6） |
| 最差影响/退路分析 | `card_counts`、`pool_draw_counts`、`final_resources` | 无影响（仅用聚合字段） |
| 流式分析 | 全部字段 | 无影响（直接透传 compact dict） |

**设计要求**：Numba 内核**必须填充逐抽数组**（`draw_pool_ids`、`draw_card_ids`、`draw_times`、`draw_pity`、`draw_pity_names`、`draw_pity_counter_max`、`draw_resources_consumed`、`draw_resources_gained`），否则过程分析和抽取序列可视化将降级或失效。这些数组写入是纯 append 操作，在 Numba 中实现开销很低（预分配固定大小缓冲区 + 索引递增）。

#### 2.7.3 InfoVector 旧路径

`InfoVectorCollector` 路径仅在 `collector=None` 时作为默认值触发，当前 GUI 流程全部走 `CompactCollector`。Numba 内核不需要支持 InfoVector 输出。

## Phase 3：并行架构优化（3-5 天）

### 3.1 共享内存替代 pickle 传输

`batch_simulator.py`：使用 `multiprocessing.shared_memory` 将 CompactResult 写入共享缓冲区，避免 pickle 序列化开销。

- 每个子进程分配固定输出槽位
- 完成后通过 `Queue` 通知主进程槽位索引
- Windows 特殊处理（显式 close + unlink）

### 3.2 改进 chunksize 策略

基于单次模拟估算耗时动态计算 chunksize，减少负载不均。

### 3.3 流式聚合（超大规模）

当 `num_simulations > 100000` 时，自动启用 `SharedResultCollector` 流式聚合模式——边模拟边提取统计量，不保留完整序列。

### Phase 3 验证

- 多核扩展性测试：1/4/8/16/20 核
- 内存监控：100 万样本下 < 2GB
- 长时间稳定性：连续运行无内存泄漏
