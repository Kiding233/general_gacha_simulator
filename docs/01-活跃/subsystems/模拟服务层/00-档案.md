## 业务问题

GachaSimulator 的核心模拟能力需要跨越 GUI 层和核心引擎层之间的鸿沟。GUI 面板需要发起批量并行模拟、收集结果、提取分析数据，而核心引擎（`core/pool.py`、`core/pity.py`、`core/strategy.py` 等）只提供单次模拟的原子操作。

模拟服务层的角色是**桥梁**：
- 将 `ConfigStore`（GUI 层配置）转换为 `SimulationEnv`（核心层可消耗的运行时对象）
- 管理多进程并行模拟的生命周期（启动 → 分发任务 → 收集结果 → 合并提取数据）
- 封装主模拟循环（`GachaService.run_simulation()`），协调策略决策、保底引擎、资源管理、停止条件

## 设计意图

1. **解耦 GUI 与核心引擎**：GUI 面板不直接依赖 `Pool`、`PityEngine`、`Strategy` 等核心类，通过 `SimulationEnvBuilder` 和 `run_batch_parallel()` 间接使用
2. **透明并行**：调用方只需指定 `max_workers`，无需关心进程池创建/销毁、任务分发、结果收集等并发细节
3. **内存高效**：支持两种模拟模式——`CompactCollector`（紧凑，主路径）和 `InfoVectorCollector`（完整，兼容旧代码），通过 `StreamingAnalyzer` 实现边模拟边提取边丢弃，O(1) 内存
4. **双路径回退**：单进程路径（`max_workers <= 1`）直接在主进程调用，避免不必要的进程创建开销；多进程路径通过 `multiprocessing.Pool(initializer=...)` 注入运行环境

## 技术实现

### 调用链

```
GUI 面板 SimulationThread.run()
  → SimulationEnvBuilder.from_config_store(config_store) → SimulationEnv
  → run_batch_parallel(n, target_specs, initial_resources, max_workers, seed,
                       on_result=collector.on_result,
                       stop_condition=controller.should_stop)
    → [多进程] multiprocessing.Pool(initializer=_wk_init, initargs=(env, target_specs))
        → _wk_run_single(seed, initial_resources)
          → GachaService(pools, strategy, stop_cond, ...).run_simulation_compact(state)
          → CompactResult
    → [单进程] 直接在主进程调用 _wk_run_single
  → BatchResult (向后兼容 list 接口 + .extraction 聚合数据)
  → 各面板从 BatchResult 读取
```

### 核心类

| 类 | 文件 | 职责 |
|----|------|------|
| `SimulationEnv` | `service/batch_simulator.py` | 18 字段 dataclass，封装一次批量模拟所需的所有运行时对象 |
| `SimulationEnvBuilder` | `service/batch_simulator.py` | 将 `ConfigStore` 转换为 `SimulationEnv` 的工厂类 |
| `GachaService` | `service/gacha_service.py` | 主模拟循环，协调抽卡、策略决策、资源管理、停止条件 |
| `BatchResult` | `service/batch_simulator.py` | 一批模拟的结果容器，向后兼容 list 接口 + `.extraction` 聚合数据 |
| `SimulationStats` | `service/gacha_service.py` | 统计量汇总：total_actions、total_draws、card_counts、pity_triggers 等 |

### 两种模拟模式

| 模式 | Collector | 输出 | 用途 |
|------|-----------|------|------|
| 紧凑模式 | `CompactCollector` | `CompactResult`（dict，含逐抽数组 + 汇总统计） | 主流路径，内存高效 |
| 完整模式 | `InfoVectorCollector` | `List[InfoVector]`（每抽一条详细记录） | 需逐抽详细记录时使用，兼容旧代码 |

### Worker Initializer 模式

11 个模块级全局变量通过 `_wk_init()` 注入子进程：

```
_wk_env: SimulationEnv           # 全部运行时对象（含 pools, schedule_mgr, pity_engine 等）
_wk_target_set: TargetCardSet    # 预构建的目标卡集合（不可变）
_wk_extractor: WorkerLocalExtractor  # Worker 内提取器
```

全局变量消除方案（已在 `docs/Worker全局变量消除方案.md` 中设计，P10）将 11 个变量合并为 1 个 `_wk_env: SimulationEnv`。

### 外部依赖

- `multiprocessing.Pool`：多进程并行（`spawn` 模式，Windows 必须）
- `numpy`：Worker 内提取器的直方图聚合
- `core/`（pools、pity、strategy、stop_condition、gdr、streaming）：运行时依赖
