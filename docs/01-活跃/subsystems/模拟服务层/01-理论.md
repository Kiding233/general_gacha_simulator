# 模拟服务层 -- 理论

> 创建日期：2026-06-02（首次建档）

---

## 核心假设

### 1. 独立种子假设

每次模拟通过独立的随机种子（`seed`）启动，保证任意两次模拟之间统计独立。`batch_simulator.py` 中种子生成逻辑：

```python
base_seed = seed  # 用户提供的基础种子
task_args = [(base_seed + i, initial_resources) for i in range(n)]
```

独立性是后续所有统计推断（均值、方差、置信区间、Bootstrap）的前提。违反独立性的场景（如错误共享 RNG 状态）会导致标准误低估，置信区间过窄。

### 2. 进程隔离假设

Windows 上 `multiprocessing` 使用 `spawn` 模式，每个子进程拥有独立的 Python 解释器和地址空间。模块级全局变量（`_wk_env`、`_wk_target_set` 等）通过 `initializer` 函数复制到各子进程，之后互不干扰。

关键推论：**子进程内的全局变量修改不影响主进程或其他子进程**。这意味着：
- 保底引擎状态在各模拟间完全隔离（只要 `GachaService` 每次创建新的 `PityEngine` 实例）
- `PityReserveStrategy` 中 `get_pity_probabilities()` 的副作用不会跨模拟传播
- 单进程路径（`max_workers <= 1`）除外——全局变量在主进程直接修改，需特别小心

### 3. 无共享状态假设

`run_batch_parallel()` 的 `on_result` 回调在主进程中执行，子进程只通过 pickle 返回结果。设计保证：
- 子进程不持有主进程对象的引用
- 所有跨进程通信通过值传递（pickle 序列化/反序列化）
- 流式分析的 `SharedResultCollector` 在主进程中维护聚合状态，子进程不可见

## 已知局限性

### 1. 11 个 Worker 全局变量（P10）

当前 `batch_simulator.py` 使用 11 个模块级全局变量（`_wk_pools`、`_wk_schedule_mgr`、`_wk_end_time`、`_wk_pity_engine` 等），通过 `_wk_init()` 注入。问题：
- 变量管理分散，新增一个依赖需要新增一个全局变量
- 单进程路径直接修改这些全局变量，可能意外污染后续调用
- 消除方案（P10）：将 11 个变量合并为 1 个 `_wk_env: SimulationEnv`

### 2. 单进程路径污染

当 `max_workers <= 1` 时，`run_batch_parallel()` 直接在主进程调用 `_wk_init()` 和 `_wk_run_single()`。这意味着：
- `_wk_init()` 覆盖了模块级全局变量
- 如果主进程中其他代码依赖这些全局变量的旧值，会产生意外行为
- CLI 模式（`python -m gacha_simulator.cli`）默认走单进程路径

### 3. 多面板 CPU 争抢

每个 GUI 面板的 Worker 线程独立创建 `multiprocessing.Pool`，多个面板同时运行时：
- 无全局调度——各面板争抢 CPU 核心
- 每个 Pool 独立创建/销毁，进程启动开销重复支付
- 线程间无协调机制（当前依赖用户手动避免多面板同时运行）

### 4. Pickle 序列化开销

`CompactResult` 通过 pickle 在子进程和主进程之间传输。每条结果 ~15KB（含逐抽数组），1000 条结果 → ~15MB pickle 数据。Phase 1.5（精简传回）已将传回体积降至 ~1KB（仅聚合字段），但需要逐抽序列的场景（过程分析）仍需完整传输。

### 5. 伪随机数质量

当前使用 Python 内置 `random` 模块 + NumPy `np.random.default_rng()`（PCG64）。PCG64 的周期长度（2^128）和统计质量满足当前需求（最多百万级模拟），但 Numba 内核（Phase 2）需使用 Numba 自带的 Xoroshiro128+ 或回退到 Python RNG。

### 6. 无 GPU 加速路径

当前架构纯 CPU 并行。经评估（见 `P12 模拟性能优化方案.md`），GPU 加速在此场景大概率是负优化——gacha 模拟的决策树分支密集、每步依赖上一步状态（强序列依赖），GPU SIMD 模式无法利用。不计划添加 GPU 路径。
