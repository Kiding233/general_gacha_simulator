## 一、核心假设

### 1.1 `sys.frozen` 作为环境标识

PyInstaller 在打包后的 Python 解释器中设置 `sys.frozen = True`（同时设置 `sys._MEIPASS` 指向 `_internal/` 目录）。`paths.py` 依赖此标识区分开发/打包环境：

```python
def get_base_dir() -> str:
    if getattr(sys, 'frozen', False):
        return sys._MEIPASS       # 打包 → _internal/
    return os.path.dirname(os.path.abspath(__file__))  # 开发 → 项目根
```

**假设约束**：
- `sys.frozen` 在 PyInstaller 打包中稳定为 `True`——验证来源：PyInstaller 官方文档 `runtime-information`
- 未来若切换到 Nuitka，`sys.frozen` 仍为 `True`（Nuitka 兼容 PyInstaller 的 `frozen` 约定）
- 若切换到 Briefcase/cx_Freeze，需验证该标识是否一致——这构成 PyInstaller 的**锁定因子**（见 3.2 节）

### 1.2 `sys._MEIPASS` 目录布局

PyInstaller 的 onedir 模式将 `sys._MEIPASS` 指向 `_internal/`。所有通过 `--add-data` 注入的文件和包模块均位于此目录下。

```
sys._MEIPASS/
├── config/           ← --add-data "gacha_simulator/config;config"
├── resources/        ← --add-data "gacha_simulator/resources;resources"
├── gacha_simulator/  ← Python 包（自动收集）
├── PyQt6/ Qt6/       ← Qt 库（hook 自动收集）
└── numpy/ scipy/ ... ← 依赖库（自动收集）
```

**假设约束**：
- `--add-data` 的源路径使用相对于 `.spec` 文件的路径（项目根目录）
- 目标路径保持 `config/` 和 `resources/` 的扁平结构，与 `paths.py` 中的 `get_config_dir()` / `get_resource()` 一致
- QtWebEngine 的 `QtWebEngineProcess.exe` 和 Chromium 资源文件由 PyInstaller 的 `PyQt6.QtWebEngineWidgets` hook 自动收集到 `_internal/PyQt6/Qt6/` 路径下

### 1.3 multiprocessing spawn 模式在打包后的正确性

Windows 上 Python 的默认 `multiprocessing` 启动模式为 `spawn`——每个子进程重新执行主模块。打包后，主模块变为 `GachaStat.exe`，子进程通过 `sys.executable` 重新启动自身。

**必要条件**：
1. `main.py` 的 `if __name__ == '__main__':` 护卫内首行调用 `multiprocessing.freeze_support()`
2. Worker initializer（`_wk_init`）所需的全局变量注入方式不依赖主模块的顶层代码——当前已满足（通过 `initializer` + `initargs` 参数传递 `SimulationEnv`）

**验证逻辑**：`freeze_support()` 在 PyInstaller 打包环境中模拟了 `python -m gacha_simulator.main` 的模块执行语义，确保子进程能正确解析 `sys.argv` 并重新导入 `gacha_simulator` 包。

### 1.4 Plotly.js 路径可用性

`chart_webview.py` 中通过 `plotly.__file__` 定位 `plotly.min.js`：

```python
path = os.path.join(os.path.dirname(plotly.__file__), "package_data", "plotly.min.js")
```

**假设**：PyInstaller 将 `plotly` 包完整收集（包括 `package_data/plotly.min.js`），且 `plotly.__file__` 在打包后指向 `_internal/plotly/__init__.pyc`——因此 `os.path.dirname` 结果正确。

此假设在 PyInstaller 对标准 Python 包的收集机制下成立，无需额外配置。

## 二、路径适配的正确性

### 2.1 开发→打包 路径等价性

`paths.py` 保证 `get_config_dir()` 和 `get_resource()` 在两种环境下返回相同语义的路径：

| 环境 | `get_config_dir()` | `get_resource('app_icon.png')` |
|------|--------------------|-------------------------------|
| 开发 | `<project_root>/config/` | `<project_root>/resources/app_icon.png` |
| 打包 | `<sys._MEIPASS>/config/` | `<sys._MEIPASS>/resources/app_icon.png` |

两个路径在打包后指向 `_internal/config/` 和 `_internal/resources/`——通过 `--add-data` 注入的目录。

### 2.2 不变性：不影响现有代码的路径语义

`paths.py` 只新增两个函数（`get_config_dir`、`get_resource`），不改写任何现有路径逻辑。受影响的两个文件（`main.py`、`main_window.py`）的改动仅限于将硬编码的相对路径替换为 `paths` 调用——语义等价，风险可控。

`chart_webview.py` 的 `_get_plotly_js_url()` 不受影响——其通过 `plotly.__file__` 动态定位，打包后自动指向正确位置。

### 2.3 配置目录的回退兼容

`main_window.py` 的 `import_config()` 方法（`QFileDialog.getExistingDirectory`）允许用户导入任意目录的配置文件——此路径直接来自文件对话框，不依赖 `paths`，打包后仍正常工作。`_load_default_config()` 改用 `paths.get_config_dir()` 后，默认加载打包内置配置。

### 2.4 用户数据目录的可写性

打包后，捆绑在 `_internal/` 下的所有文件为只读。应用运行时的输出（分析结果导出、用户自定义配置保存等）必须写入用户数据目录：

| 环境 | `get_user_data_dir()` 解析 | 保证 |
|------|--------------------------|------|
| 开发 | `gacha_simulator/` 下的对应子目录 | 与源码结构一致，方便调试 |
| Windows 打包 | `%APPDATA%/GachaStat/` | 用户应用数据标准位置，始终可写 |
| Linux（未验证） | `$XDG_DATA_HOME/GachaStat/` 或 `~/.local/share/GachaStat/` | XDG 规范 |
| macOS（未验证） | `~/Library/Application Support/GachaStat/` | macOS 规范 |

`os.makedirs(target, exist_ok=True)` 确保路径在首次写入前已存在，调用方无需手动创建。

### 2.5 只读 vs 可写路径的设计分离

| 接口 | 环境 | 性质 | 用途 |
|------|------|------|------|
| `get_config_dir()` | `get_base_dir()` → `_internal/config/` | 只读 | 捆绑的默认配置 |
| `get_resource()` | `get_base_dir()` → `_internal/resources/` | 只读 | 图标等静态资源 |
| `get_user_data_dir()` | `%APPDATA%/GachaStat/` | 可写 | 输出文件、用户配置、分析结果 |

调用方不应绕过 `paths` 直接读写文件——所有 I/O 路径必须通过以上三个接口之一获取。

## 三、已知局限与锁定因子

### 3.1 仅验证 Windows 平台

首版打包仅在 Windows 11 上验证。macOS 和 Linux 的理论兼容性基于 PyInstaller 的跨平台支持，但以下差异未验证：

| 平台 | 潜在差异 |
|------|---------|
| macOS | `.app` Bundle 结构 vs 文件夹分发；QtWebEngine 沙箱权限；签名公证要求 |
| Linux | AppImage/flatpak 需求；Qt 平台插件 `libqxcb.so` 依赖；Wayland vs X11 |

**锁定因子**：如果未来需支持 macOS/Linux，`paths.py` 的 `get_base_dir()` 接口无需修改，但 `.spec` 文件可能需要平台特定的 `binaries` 和 `datas` 配置。

### 3.2 PyInstaller 锁定

当前方案深度依赖 PyInstaller 的运行时约定（`sys.frozen`、`sys._MEIPASS`、hook 机制）。切换到其他打包工具需：

1. 验证 `sys.frozen` 等价性
2. 重新配置数据文件注入（`.spec` → Nuitka `--include-data-dir` / Briefcase `setup.py`）
3. 重新验证 QtWebEngine 资源收集

`paths.py` 接口（`get_base_dir` / `get_config_dir` / `get_resource`）屏蔽了底层差异——切换打包工具时只需修改 `paths.py` 实现，调用方无需改动。

### 3.3 QtWebEngine 体积

Chromium 内核（约 120MB）是打包产物的最大单一贡献者——占总体的 50-65%。这是 QtWebEngine 的固有特征，PyInstaller 无法缩减。如果未来需要减小体积：

- **选项 A**：从 QtWebEngine 迁移到 `matplotlib.backends.backend_qtagg`（原生 Qt 图表）——需重写所有 Plotly 图表逻辑
- **选项 B**：使用 `PyQt6.QtWebChannel` + 系统浏览器——打破自包含假设，引入外部依赖
- **选项 C**：提供「精简版」（CLI only，不含 GUI/WebEngine）——体积可降至 ~80MB

当前不采纳以上方案，因牺牲用户体验或自包含性。

### 3.4 杀软误报

Python 打包的 `.exe`（尤其包含嵌入式解释器 + 数十个 DLL）可能触发杀软启发式扫描。文件夹模式比单文件模式误报率低，但非零。缓解策略：

- 文件夹分发（dll 分散可见，不如单文件「可疑」）
- 可选的代码签名（需 EV 证书，年费 ~$300-500）
- 提交误报到各杀软厂商白名单（按需，非自动化）
