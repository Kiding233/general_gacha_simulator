> 创建日期：2026-06-02（首次建档）
> 无关联主题计划（稳定子系统）
> 关联模块：`core/stop_condition.py`
> 关联子系统：策略系统（策略的决策受停止条件约束——当停止条件满足时，`GachaService` 终止模拟循环）、模拟服务层（`GachaService.run_simulation()` 每步检查停止条件）

---

## 业务问题

模拟需要知道何时停止。不同的使用场景有不同的终止条件：

- 全周期模拟：模拟所有池子从开放到结束的完整过程
- 预算约束模拟：「我有 100 抽，能抽到什么？」
- 目标达成模拟：「要抽多少才能拿到目标卡？」
- 时间限制模拟：「30 天活动期内能达成目标吗？」
- 资源耗尽模拟：「石头用完后是否已拿到想要的卡？」

需要一个**可组合的停止条件系统**，允许用户组合多种条件（AND/OR），并在配置面板中自由选择和参数化。

## 设计意图

### 核心抽象

- **StopCondition（ABC）**：所有停止条件的统一接口。`check(state, history, stats) -> bool`（判断是否应停止）和 `description() -> str`（人类可读的描述）。
- **STOP_CONDITION_REGISTRY**：所有停止条件的单一点注册表（dict），每个条目包含 `display_name`、`class`、`params`、`description`。GUI 停止条件下拉列表从此注册表生成。
- **CompositeStopCondition**：组合条件——`any` 模式（任一满足即停止）或 `all` 模式（全部满足才停止）。

### 7 种注册条件

| key | display_name | 判定逻辑 | internal |
|-----|-------------|---------|----------|
| `all_pools_end` | 所有池结束 | `state.real_time >= end_time`（所有池结束时间） | 否 |
| `fixed_action_count` | 固定次数 | `stats.total_actions >= max_actions`（或 `len(history) >= max_actions`） | 否 |
| `resource_threshold` | 资源阈值 | `state.resources[resource] <=/>=/== threshold` | 否 |
| `target_acquired` | 目标获得 | `stats.card_counts[target_id] >= quantity` | 否 |
| `last_draw_card` | 抽到即停 | `stats.last_draw_card_id == card_id`（或 `history[-1].card_id == card_id`） | 否 |
| `time_limit` | 时间限制 | `state.real_time >= max_time` | 否 |
| `consecutive_pool_target` | 连续池目标 | 资源耗尽、超时、或连续池目标未达成 | **是** |

`consecutive_pool_target` 标记为 `internal: True`，不在 GUI 的停止条件列表中显示，仅用于内部自动化场景。

### 工厂函数

| 函数 | 用途 |
|------|------|
| `create_stop_condition(name, params)` | 从注册表构建停止条件实例（参数自动填充默认值、类型转换） |
| `stop_condition_type_to_key(display_name)` | 中文显示名 → 注册表 key |
| `stop_condition_key_to_type(key)` | 注册表 key → 中文显示名 |

## 技术实现

### 调用链

```
GachaService.run_simulation():
  while True:
    action = strategy.select_action(ctx)
    if action is WAIT: break   # 策略主动停止
    执行 action → 更新 state
    if stop_condition.check(state, history, stats):  # 每步检查
      break
```

### 核心数据结构

| 类 | 职责 |
|----|------|
| `StopCondition (ABC)` | 抽象基类，定义 `check()` 和 `description()` |
| `FixedActionCountCondition` | 固定操作次数限制 |
| `ResourceThresholdCondition` | 资源阈值（支持 `<=`/`>=`/`==` 三种比较符） |
| `TargetAcquiredCondition` | 目标卡获得数量 |
| `LastDrawCardCondition` | 上一抽为指定卡 |
| `TimeLimitCondition` | 现实时间限制 |
| `AllPoolsEndCondition` | 所有池结束时间 |
| `ConsecutivePoolTargetCondition` | 连续池目标未达成（内部用） |
| `CompositeStopCondition` | 组合条件（`any`/`all` 模式） |

### 实施历史

- **v1.0-v1.5**：仅有 `fixed_action_count` 和 `all_pools_end` 两种条件
- **v1.6**：引入注册表模式和 `CompositeStopCondition`
- **v1.7**：新增 `target_acquired`、`last_draw_card`、`time_limit`、`resource_threshold`
- **v1.8**：新增 `consecutive_pool_target`（内部用）
- **当前状态**：7 种条件全部可用，注册表机制完善。稳定子系统，无已知活跃问题。
