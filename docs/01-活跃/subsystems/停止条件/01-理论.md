# 停止条件 -- 理论

> 创建日期：2026-06-02（首次建档）
> 状态：稳定子系统，无已知理论缺陷

---

## 一、核心假设

### 1.1 停止判定在每步之后发生

`GachaService` 在每次执行完 Action 之后立即调用 `stop_condition.check()`。这意味着：

- 如果第 N 步满足了停止条件，模拟的总步数为 N（含第 N 步）。
- 第 N 步的结果（获得的卡、消耗的资源）会被计入最终结果。
- 不存在「提前预见停止」的机制——策略在第 N-1 步做出决策时不知道第 N 步会触发停止。

### 1.2 条件判定使用两种数据源

`check()` 方法接收 `stats: Optional[SimulationStats]` 参数：

- **有 `stats` 时**：使用 `stats`（聚合统计对象）中的预计算值，O(1) 判定
- **无 `stats` 时**：遍历 `history: List[InfoVector]`，O(T) 判定

这意味着停止条件的计算复杂度可能依赖于传入参数的形式——紧凑模式（有 `stats`）更快。

### 1.3 组合条件的短路求值

`CompositeStopCondition`：
- `any` 模式：从左到右短路求值——遇到第一个 `True` 立即返回，后续条件不评估
- `all` 模式：从左到右短路求值——遇到第一个 `False` 立即返回，后续条件不评估

短路求值意味着在 `any` 模式下，条件评估顺序（即 `conditions` 列表的顺序）影响性能但不影响结果。不依赖副作用的评估顺序是安全的。

## 二、各条件的理论性质

### 2.1 终止性（Termination）

| 条件 | 保证终止？ | 条件 |
|------|----------|------|
| `all_pools_end` | 是 | 只要 `end_time` 有限 |
| `fixed_action_count` | 是 | 操作计数单调递增 |
| `resource_threshold` | 取决于操作符 | `<=` 在资源递减时保证终止；`>=` 在资源递增时保证终止 |
| `target_acquired` | 否 | 目标卡可能永远抽不到（概率 > 0 但可能需无限步） |
| `last_draw_card` | 否 | 特定卡可能永远抽不到 |
| `time_limit` | 是 | 时间单调递增 |
| `consecutive_pool_target` | 是 | 资源耗尽或超时保证终止 |

**非终止风险**：`target_acquired` 和 `last_draw_card` 理论上可能无限运行。实际使用中通常与 `fixed_action_count` 或 `time_limit` 组合（通过 `CompositeStopCondition` 的 `any` 模式）来保证终止。

### 2.2 单调性

所有停止条件都是**单调的**——一旦 `check()` 返回 `True`，之后的任何 `check()` 调用也都返回 `True`（因为状态指标单调变化：行动计数增加、资源通常递减、时间递增）。

这是模拟循环安全性的关键保证——不存在「停止后又恢复」的振荡。

## 三、局限性

### 3.1 无概率化停止条件

当前所有条件都是确定性的（二值 True/False）。不支持「以概率 p 停止」或「GDR 值达到阈值后停止」等概率化/指标化条件。

概率化停止可作为 GDR 系统的职责——在 `GachaService` 层面而非 `StopCondition` 层面处理。

### 3.2 无时间-资源联合约束

`resource_threshold` 和 `time_limit` 各自独立工作。不支持「在时间 T 内资源 R 不低于 X」的联合约束。此类需求需通过 `CompositeStopCondition(all)` 近似实现（但语义不完全等价——all 要求两者同时满足才停止，而非联合约束）。
