## 〇、合并范围：三个面板 vs 两个面板

### 当前涉及的三个页面

| Tab 标签 | 面板类 | 文件 | 行数 | 起点 | 搜索模式 | 权重来源 |
|---------|--------|------|------|------|---------|---------|
|「最多目标卡」| `StrategyPanel` | `strategy_panel.py` | 673 | 完整时间线（day 0） | 前进法 / 后退法 | **自有权重表**（desire + miss_cost） |
|「最少资源」| `ResourceSearchPanel` | `resource_search_panel.py` | 648 | 完整时间线（day 0） | 二分搜索最少资源 | config_panel |
|「退路分析」→「方案搜索」| `RetreatSearchPanel` | `retreat_search_panel.py` | 546 | 退路点（截断时间线，需脆弱性分析） | 最少资源 / 后退法 / Pareto | **自有权重表**（仅 miss_cost） |

三个面板合计 **1,867 行**，各自包含独立的 Worker + 模拟调用 + 进度报告 + 结果展示。

### 两个合并方案

| | 方案 A（大三合一） | 方案 B（小二合一） |
|---|---|---|
| **合并范围** | StrategyPanel + ResourceSearchPanel + RetreatSearchPanel | StrategyPanel + ResourceSearchPanel |
| **RetreatSearchPanel** | 并入统一面板 | **保持独立**（退路点起点差异大） |
| **新增能力** | 前进法（退路点）+ 完整时间线 Pareto + 退路点 Pareto 已有 | 完整时间线 Pareto |
| **权重处理** | 三表合一：面板内一张统一可编辑权重表（desire+miss_cost 双列），替代 StrategyPanel + RetreatSearchPanel 各自权重表 | 两表合一：面板内一张统一可编辑权重表，替代 StrategyPanel 权重表 |
| **UI 复杂度** | 1 起始池下拉 × 3 模式（前进/后退为子选项）= 灰显逻辑控制 | 单起点 × 3 模式 = 布局简单 |
| **回归风险** | 高（三个面板全部替换） | 中（两个面板替换） |
| **代码减少** | ~1867 → ~800（净减 ~1000 行） | ~1321 → ~600（净减 ~700 行） |
| **Tab 变更** | 移除 3 个旧 Tab → 新增 2 个一级 Tab（方案搜索 + 脆弱性分析） | 移除 2 个旧 Tab → 新增 1 个 Tab（方案搜索），脆弱性分析保留为一级 Tab |

### 推荐：方案 A（大三合一）

理由：

1. **三个面板做的是同一件事**：迭代运行模拟 → 搜索最优方案。它们的搜索算法（前进/后退/二分/Pareto）、模拟调用（`run_batch_parallel` + `compute_success_probability`）、进度报告模式完全一致。差异仅在于起点（完整 vs 截断时间线）和搜索维度（资源 vs 目标卡），这些是参数而非架构差异。

2. **权重表重复但用途独立**：StrategyPanel 和 RetreatSearchPanel 各自有独立权重表，且搜索面板的权重用途（排序决策）与配置页权重（GDR 计算）不同。合并后保留面板内独立可编辑权重表，三表合一（desire + miss_cost 双列），消除两处重复的同时保持编辑独立性。

3. **「最多目标卡」Tab 名误导**：StrategyPanel 的 Tab 名是「最多目标卡」，但 RetreatSearchPanel 中也有「最多目标卡」模式——用户看到两个同名功能会困惑。合并后搜索模式统一命名，不再重复。

4. **前进法天然跨起点**：StrategyPanel 的前进法（完整时间线）和 RetreatSearchPanel 只缺的前进法（退路点）共享同一算法，合并后自然补全退路点前进法。

## 一、信息流设计（方案 A）

### 1.1 起始池切换 × 三种搜索模式

起点不由独立 RadioButton 选择，而是通过「起始池」QComboBox 隐式决定：

- **首项 `(从头开始)`**：`from_pool_id=None` → 完整时间线，使用配置的初始资源与保底初值
- **其他项（来自脆弱性分析）**：`from_pool_id=str` → 截断时间线，基准资源/保底从 VulnerabilityResult 填充

```
                      ┌─────────────────────────┐
                      │  ConfigStore（配置页）     │ ← 权重、目标卡、策略、GDR
                      └────────┬────────────────┘
                               │
                               ▼
                    ┌──────────────────────┐
                    │  PlanSearchPanel     │
                    │                      │
                    │  起始池 QComboBox:    │
                    │  ├ (从头开始)         │  ← from_pool_id=None
                    │  ├ pool_1 (脆弱性)    │  ← from_pool_id="pool_1"
                    │  └ pool_2 (脆弱性)    │  ← from_pool_id="pool_2"
                    └──────────┬───────────┘
                               │
                               ▼
                    ┌──────────────────────┐
                    │  PlanSearchEngine    │
                    │  from_pool_id=None   │  ← 完整时间线
                    │  from_pool_id=str    │  ← 截断时间线
                    └──────────┬───────────┘
                               ▼
          ┌──────────┬─────────┼──────────┬──────────┐
          ▼          ▼         ▼          ▼          ▼
    最少资源    最多目标卡  最多目标卡   Pareto
    (二分搜索)  (前进法)    (后退法)    前沿
```

### 1.2 起始 × 模式适用矩阵

| 起始池选择 \ 模式 | 最少资源（二分） | 最多目标卡·前进法 | 最多目标卡·后退法 | Pareto 前沿 |
|------------------|----------------|------------------|------------------|------------|
| **(从头开始)** | ✅ StrategyPanel + ResourceSearchPanel 现有 | ✅ StrategyPanel 现有 | ✅ StrategyPanel 现有 | ⭐ **新增** |
| **脆弱性池子** | ✅ RetreatSearchPanel 现有 | ⭐ **新增**（前进法天然跨起点） | ✅ RetreatSearchPanel 现有 | ✅ RetreatSearchPanel 现有 |

> 合并后补齐两个空白格：完整时间线 Pareto + 退路点前进法。

### 1.3 面板间数据流

```
ConfigPanel ──config_changed──→ PlanSearchPanel（更新策略、权重、目标卡列表）

GachaPanel ──simulation_finished──→ RetreatPanel（触发脆弱性分析）

RetreatPanel ──vulnerability_finished──→ PlanSearchPanel（传入退路起点数据）
                                         （信号携带 VulnerabilityResult）
```

PlanSearchPanel 内部状态机：

```
状态1: 无脆弱性数据
  └─ 起始池下拉仅含「(从头开始)」一项 → 完整时间线，资源/保底编辑区灰显

状态2: 收到 VulnerabilityResult
  └─ 起始池下拉追加剧中池子 → 用户可选池子，资源/保底自动填充并可用

状态3: 搜索中 → 仅停止按钮可用

状态4: 搜索完成 → 恢复配置控件，结果区展示，快照自动 push
```

### 1.4 权重方案：面板内独立可编辑

**为什么不从 config_panel 读取**：搜索面板的权重用途（排序决策——「哪张卡更不重要/更想要」）与配置页权重（GDR 计算——「每张卡的价值/代价」）语义不同。用户在搜索时可能临时调整排序优先级而不想改动全局 GDR 权重。两者应保持独立。

**三表合一**：StrategyPanel 有 desire+miss_cost 双列表，RetreatSearchPanel 有 miss_cost 单列表。合并后面板内只保留**一张统一权重表**（卡ID | 抽取意愿 | 错失代价），替代原来的两张独立表，消除维护重复。

**权重使用**：

| 搜索模式 | 使用列 | 排序方向 | 含义 |
|---------|--------|---------|------|
| 前进法 | 抽取意愿（desire） | 降序 | 高渴望的卡优先加入目标集 |
| 后退法 | 错失代价（miss_cost） | 升序 | 低代价的卡优先从目标集移除 |
| 最少资源 | —（不使用） | — | 目标集固定，无需排序 |
| Pareto | 错失代价（miss_cost） | 升序 | 按代价升序逐步缩减目标集 |

**card_value_weights**：GDR 计算需要，从 config_panel 获取（GDR 权重属于全局配置，不应在搜索面板重复编辑）。

### 1.5 脆弱性分析（RetreatPanel）的处理

**结论：不合并，升级为一级 Tab。**

当前 `RetreatPanel` 和 `RetreatSearchPanel` 嵌套在「退路分析」Tab 的子Tab 中——这是整个 GUI 中**唯一使用子Tab嵌套的地方**，和其他 9 个一级 Tab 风格不一致。

合并后的 Tab 结构：

```
当前（合并前）                              合并后（方案 A）
─────────────                              ────────────────
Tab「最多目标卡」  StrategyPanel           Tab「方案搜索」  PlanSearchPanel（三合一）
Tab「最少资源」    ResourceSearchPanel     Tab「脆弱性分析」 RetreatPanel（独立一级Tab）
Tab「退路分析」                            （Tab「退路分析」移除）
  ├ 子「脆弱性分析」RetreatPanel
  └ 子「方案搜索」  RetreatSearchPanel
```

**效果**：3 个旧 Tab → 2 个一级 Tab，消除子Tab嵌套，所有 Tab 层级统一。

**数据流**：`RetreatPanel.vulnerability_finished` 信号 → `PlanSearchPanel.set_vulnerability_result()`。两个独立一级 Tab 之间通过信号通信，和当前子Tab间通信方式一致，仅改接收方类名。

**为什么不合并脆弱性分析**：
1. **性质不同**：脆弱性分析是诊断工具（分析过去），方案搜索是决策工具（规划未来）
2. **输入不同**：脆弱性分析消费 `List[CompactResult]`（批量模拟产出），方案搜索消费 `ConfigStore` + 可选 `VulnerabilityResult`
3. **脆弱性分析代码不在此次范围内**：核心在 `core/vulnerability.py`（~800 行算法），面板只是薄壳（322 行配置+触发），合并它不会简化代码

## 二、UI 设计（v8 终版，经 8 轮视觉伴侣迭代确认）

### 2.1 设计原则

- **灰显替代消失**：不相关控件 `setEnabled(False)` 而非 `setVisible(False)`，布局稳定无跳动，控件值保留
- **QGroupBox + QRadioButton**：所有选项组使用 QGroupBox 包裹 + QRadioButton 横向排列，与现有 `retreat_search_panel.py` 风格一致
- **单一起始状态**：起始池 QComboBox 首项为 `(从头开始)`，其余项来自脆弱性分析结果；选择不同项切换完整时间线 / 退路点模式
- **统一权重表**：一张双列可编辑表（抽取意愿 + 错失代价），所有模式共用

### 2.2 状态机

```
                    ┌──────────────┐
                    │  无退路数据    │ ← 初始状态，或配置变更后
                    └───┬──────────┘

                        ▼
                    ┌──────────────┐
           ┌───────│    就绪      │←──────────────┐

           │           │ 用户点击「开始搜索」        │
```

快照自动命名规则：`{搜索模式}·{搜索方向} @{HH:MM}`。例如：
- `最少资源 @17:42`
- `最多目标卡·前进 @17:48`
- `Pareto·后退 @17:55`

快照行为逻辑：

| 属性 | 规则 |
|------|------|
| **存储** | 内存列表，非持久化（关闭面板即清空） |
| **上限** | 20 条，超出时移除最旧快照 |
| **触发** | 每次搜索正常完成时自动 push；用户停止/出错不产生快照 |
| **跨模式** | 三种搜索模式的快照共存于同一列表，按时间排序 |
| **导航** | `<` `>` 切换前后快照，结果区立即刷新（图表 + 表格） |
| **删除** | `×` 删除当前快照，删除后自动切到相邻快照 |
| **交互限制** | 翻看历史快照时搜索参数区保持当前值，不可编辑（灰显）；点击「开始搜索」退出翻看模式 |

### 2.6 错误处理统一

合并后 PlanSearchWorker 统一三种面板的错误处理模式：

| 行为 | 策略 |
|------|------|
| **停止机制** | `engine.stop()`——引擎级统一停止（当前 Strategy/Resource 用标志位轮询，Retreat 用引擎方法） |
| **异常包装** | `DetailedError(msg)` 含完整 traceback（沿用 StrategyPanel 方案，Retreat 当前未包装） |
| **结果空检查** | `result is None` → 视为用户停止，状态栏显示「搜索已停止」 |
| **traceback 输出** | 统一 `traceback.print_exc()`（当前 Resource 缺失此行为） |
| **结果类型校验** | 不强制 `assert isinstance`——由 `PlanSearchResult.search_mode` 字段区分类型

```python
            ir = dict(env.initial_resources)
            ir['draw_resource'] = resource_value
            env.initial_resources = ir
            return env
        else:
            # 截断时间线：通过 RetreatConfigBuilder
            ...

    # --- 四种搜索模式 ---
    def search_min_resource(self, target_specs) -> ResourceSearchResult: ...
    def search_max_targets_forward(self, candidate_specs) -> ForwardResult: ...
    def search_max_targets_backward(self, target_specs) -> BackwardResult: ...
```

## 六、风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| **回归风险**：破坏三种现有搜索功能 | 高 | 先跑测试覆盖现有行为；旧文件标记 deprecated 不立即删除；分阶段提交 |
| **权重数据迁移**：StrategyPanel 双列 + RetreatSearchPanel 单列 → 统一双列表 | 低 | 合并后面板仅一张双列权重表（1.4 节已确认）；初始化时从 StrategyPanel 取值，RetreatSearchPanel 的 miss_cost 列补充缺值 |
| **UI 状态爆炸**：2 起始池 × 3 模式 = 6 种有效组合 | 低 | 灰显（setEnabled）而非隐藏，布局稳定；仅 3 个 QStackedWidget 页面；所有控件始终在位 |
| **完整时间线 Pareto 性能**：N 张卡 × 每张一次二分搜索（~12 次模拟） | 中 | 显示预估耗时；二分结果缓存复用；提供「快速模式」（减少每步模拟次数） |
| **脆弱性分析依赖**：退路点模式强依赖脆弱性分析 | 低 | 起始池下拉仅含 `(从头开始)` 直到收到 VulnerabilityResult 信号 |

**已确认不构成风险（2026-06-08 讨论）**：

| 事项 | 结论 |
|------|------|
| **P26（退路搜索 10 项缺陷）→ P8 依赖** | 非硬依赖，可并行。P26 修复引擎层算法（`core/retreat_search.py`），P8 合并 UI 层，代码变更区域不重叠。建议不同日合入避免冲突。 |
| **错误处理模式统一** | 2.6 节已明确方案：引擎级 `stop()` + `DetailedError` 包装 + 统一 `traceback.print_exc()`。 |
| **`cost_per_draw` 多池问题** | 理论局限，非可修的 bug——多池成本不同时「资源↔抽数」本身就是模糊概念，不存在正确值。二分搜索在资源空间运行，cost 仅影响 epsilon 精度（值越小越精确）和 UI 换算展示，不影响正确性。处理：四处重复代码归并入引擎层（阶段 E），spinbox tooltip 注明近似性质。 |
