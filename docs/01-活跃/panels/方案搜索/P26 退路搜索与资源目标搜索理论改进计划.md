### 1.2 关键实证发现（集成实验综合结论）

在进入各缺陷详解之前，先陈述五轮集成实验（实验 9 quick/full、11 expanded、14、15）确立的四个核心事实——它们构成后续所有修改建议的实证基础。

**事实一：p(r) 为三级阶梯函数。** 实际抽卡系统的成功率曲线不是平滑 logistic 曲线，而是三个离散跳跃组成的阶梯（3.5×SE、17.6×SE、35.0×SE）。默认配置仅有**一个软保底**（`type=soft, start=80, end=90, reset=featured_ssr`），三级阶梯来自该保底在跨池共享计数器 + 出 SSR 重置的机制下，为获取 6 张目标卡需要 6 个完整保底周期。第三跳跃（35.0×SE）将 p 从 0.395 直接拉至 1.000，完全跳过临界区 [0.40, 0.99]。[实验 11 expanded]

**事实一（补充——因果验证，2026-06-10）：** 三级阶梯的充要条件经四项保底因果实验确认：

| 条件 | 作用 | 实证 |
|------|------|------|
| 保底存在 | 必要条件——移除保底后 p(r) 在 r≤200,000 范围内恒 ≤0.03，无任何跳跃 | [实验 A] |
| `reset=featured_ssr` | 创造多级结构——改为 `reset=never` 后阶梯数从 3 降为 1（计数器永不重置，首个保底周期后所有抽卡 100% SSR） | [实验 B] |
| 软保底 80–90 | 决定每跳位置——改为硬保底 `threshold=90` 后跳跃更尖锐（63.6×SE vs 35.0×SE）但交叉点更高（82,019 vs 72,295），跳数从 3 降为 2 | [实验 D] |
| 6 张目标卡 | 决定所需保底周期数——收敛资源随目标卡数严格单调递增（2 卡→21,544, 3 卡→39,764, 4 卡→54,021, 5 卡→73,390, 6 卡→73,390） | [实验 C] |

结论：三级阶梯**高度配置特定**——改变保底重置条件、保底类型或目标卡数中任意一个，阶梯数量和位置即发生变化。但「p(r) 呈现阶梯函数而非平滑曲线」这一核心属性是普遍的：只要系统中存在保底阈值，跳跃不连续就必然存在。实验脚本：[scripts/verify_p26_pity_causality.py](../../../scripts/verify_p26_pity_causality.py)，结果：[output/p26_pity_causality_20260610_104716.json](../../../output/p26_pity_causality_20260610_104716.json)（快速模式）+ [output/p26_pity_causality_20260610_105640.json](../../../output/p26_pity_causality_20260610_105640.json)（全量模式）。

**事实二：std=0 跨 110 次独立搜索确证。** 跨 4 种配置（1 卡、2 卡、6 卡 × N=200/300）、两种 GDR 类型（二值 `all_targets`、连续 `target_achievement`），每一次独立搜索的每一步 Bernoulli 决策均为 100% 正确。阶跃函数是抽卡系统的本质属性——硬保底的确定性触发是根本原因，不是多卡交互的统计假象，也不是二值 GDR 的特例。[实验 9 quick + 9 full + 15]

**事实三：`resource_efficiency` 确认不兼容。** p(r) 在 r ≤ 150,000 范围内恒为零，二分搜索翻倍至 153,600,000 资源仍未收敛。N9 第三类退化的直接实证。[实验 14]

**事实四：N=200 已足够。** 在阶跃 p(r) 下，二分搜索的 Bernoulli 决策完全确定——N=200 和 N=300 给出完全相同的结果。默认 N=500 为 2.5× 浪费。建议默认 N 降至 200–300。[实验 9 full]

这些事实对实施路线的影响贯穿全文，各缺陷的「修改建议」中不再重复引用，仅在首次涉及处标注实验来源。

**实施进展（2026-06-10）**：基于上述事实，8 项修改已实施——引擎层 Q4（首卡失败标记）、Q15（pareto_points 删除）、Q14（种子独立性，此前已完成）、Q1/Q8（GDR 兼容性守卫——`_check_gdr_compatibility()` 统一入口替代 §1.6 独立守卫，三类退化全覆盖，含 A4 注册表字段）、Q6（翻倍上界增强）、Q12（降级为 docstring 注释）。GUI 层 Q1/Q8（按钮预判禁用 + GDR 兼容性提示 label）、Q5（Pareto 术语修正）。详见 [P26 零风险即时实施清单](P26 零风险即时实施清单.md)。

## 二、缺陷详解

### A 类：算法正确性（产生错误答案）

#### Q1: `lower_is_better` GDR 方向反转 ⚔ 严重

**原始编号**：N1

**现状**（`compute_success_probability()`，`gdr.py:814-828`）：
- 对 `lower_is_better=True` 的 GDR（`resource_consumed`、`resource_per_card`），成功判定逻辑为 `val <= gdr_threshold`
- p(r) 因此呈**单调递减**：更多资源 → 更高 GDR 值 → P(GDR ≤ threshold) 下降
- 二分搜索的决策逻辑（`prob >= threshold → r_hi = r_mid`）**假设单调递增**——方向完全反转

**实证确认**（[实验 8]，200 次重复）：
- 构造递减 p(r) = 1 − logistic(r)，运行标准二分搜索
- 估计均值 49,951（接近上界），正确根 4,111，偏离 45,840 单位
- **100% 的搜索收敛到上界而非正确的最小资源值**

**为何不能通过「翻转比较方向」修复**：
- 对 `resource_consumed`：零资源时消耗为 0 → P(0 ≤ threshold) = 1.0 → 「最少资源」的答案是 0（平凡解）
- 对 `resource_per_card`：零资源时无法抽卡 → 返回值 NaN → 不被计为成功 → p(0) = 0，但 p(r) 递减
- 两个 GDR 的「最少资源」问题在语义上均为退化：用户真正想问的是「最多能花多少资源而不超过消耗上限」（资源上限搜索），而非「最少需要多少资源」

**修改建议**：
1. **引擎层直接禁用**——`search_min_resource()` 入口检查 `lower_is_better`，若为 True 立即返回 `result.success = False` 并报告错误。✅ **已实施（2026-06-10）**：[`retreat_search.py:184-191`](gacha_simulator/core/retreat_search.py#L184-L191)
2. **GUI 层禁用按钮**——选择不兼容 GDR 时，禁用「最少资源」和「Pareto」模式按钮，显示红色说明文字（✅ 已实施——见 P26 零风险即时实施清单 §2.2）
3. 长期：若需支持此类 GDR，应实现独立的「资源上限搜索」模式——在 `PlanSearchEngine` 中新增 `search_max_resource()` 方法

#### Q2: 保底跳跃点的几何锚定效应 ⚔ 中等

**原始编号**：N8

**现状**：
- 实际 p(r) 含三级阶梯跳跃（[事实一]），由**软保底**（`start=80, end=90, func=linear`）的跨池共享计数器 + `reset=featured_ssr` 机制产生——软保底末端（counter=90, 100% 概率）等效于硬保底的确定性触发
- [实验 D] 确认：若将软保底替换为硬保底（`threshold=90`），跳跃从 3 个降为 2 个，最终跳跃更尖锐（63.6×SE vs 35.0×SE）但收敛点更高（82,019 vs 72,295）
- 跳跃不连续处，二分搜索的「区间确定性减半」性质退化为「区间在跳跃点处被不对称截断」

**实证确认**（[实验 4b]，跳跃/SE 比值扫描，200 次/档）：

| 跳跃/SE | 收敛右侧% | 正确决策% | 行为 |
|----------|----------|----------|------|
| 1–4 | 99.5–100 | 90.4–92.1 | 正确收敛到跳跃点上方 |
| 6 | 78.0 | 94.1 | 混合行为开始 |
| 8 | 11.5 | 98.3 | 多数收敛到下方 |
| 12 | 0.0 | 100.0 | **全部收敛到错误侧** |

**机制**：与 P26 原始担忧方向相反——小跳跃（1–4×SE）安全（二分搜索跨跳跃点两侧搜索），大跳跃（≥6×SE）危险。原因是**几何锚定**而非噪声混淆：
1. 当某 r_mid 恰好落在跳跃点且 p ≥ θ 时，r_hi 被锚定在跳跃点
2. 此后所有 r_mid < 跳跃点 → p < θ（远离跳跃点）→ r_lo 从下方逼近
3. 最终估计 = (r_lo + 跳跃点) / 2 < 跳跃点 → 收敛到错误侧
4. 跳跃越大 → p(跳跃点) 越接近 1.0 → 锚定越确定

收敛方向对**初始边界敏感**——不同的初始上界可能产生不同的中点序列，有的恰好击中跳跃点，有的从未击中。

**对实际引擎的影响**：
- 实际 p(r) 的第三跳跃为 35.0×SE，远超 6×SE 的危险阈值——理论上存在几何锚定风险
- 但 std=0 的实证（[事实二]）表明 110 次搜索中每次 Bernoulli 决策均正确——当前默认上界 55,000 产生的中点序列未击中跳跃点
- 风险存在于**边界条件变化时**——若用户修改上界、精度或目标配置，中点序列可能发生变化

**修改建议**：
1. 二分搜索结束后进行**双向跳跃检测**——大跳跃（≥6×SE）触发「几何锚定」警告；小跳跃（1–4×SE）仅记录不警告
2. 若检测到跳跃且二分结果在跳跃点 ±1×SE 范围内，在 progress_callback 中通知用户
3. 阶段二可考虑「跳跃感知二分」——在跳跃点两侧分别搜索，取更保守的结果

#### Q3: 二分搜索噪声决策错误 ⚔ 中等

**原始编号**：26, R1

**现状**（`PlanSearchEngine.search_min_resource()`，`retreat_search.py:175-309`）：
- 每步固定运行 `num_simulations`（默认 500）次模拟估计 P(success | r_mid)
- 根据 `prob >= success_threshold` 决定二分方向
- 当 p_true 恰好等于阈值时，决策错误率趋近 50%（硬币抛掷）
- 当 p_true 偏离阈值 ≥ 2×SE 时，错误率 < 2.5%

**实证发现（解析函数实验）**：

| 场景 | 发现 |
|------|------|
| p_true = θ = 0.95, N=500 | 错误率 44.6%（趋近 50%）[实验 1] |
| p_true ≤ 0.90 或 p_true ≥ 0.99 | 错误率 0.0% [实验 1] |
| 一维 logistic 函数, 200 次重复 | 偏差 −48（< 0.2×SE，不显著），标准差 291 [实验 2] |
| 区间长度衰减 | 确定性减半——与噪声无关 [实验 2] |

**关键认识修正**：P26 v1/v2 声称「噪声将二分搜索的对数收敛率退化为线性」——此为**错误断言**。区间长度的对数衰减是二分搜索的结构性质，噪声影响的是**决策正确性**（收敛到哪个区间）而非**收敛率**（区间多快缩小）。Waeber, Frazier & Henderson (2013) 发表于 *SIAM Journal on Control and Optimization*（非 Operations Research），论文核心结论是 PBA 以几何速率收敛——与 P26 退化叙事方向相反。

**但 Q3 在当前配置上的实际严重度已大幅降低**。[事实一] 表明 p(r) 为三级阶梯函数，第三跳跃（35.0×SE）将 p 从 0.395 直拉至 1.000——所有评估点 p 值均远离阈值（0.95），每一步 Bernoulli 决策 100% 正确。**std=0 意味着 Q3 的噪声决策问题在阶跃 p(r) 下不存在。**

此结论的适用范围限于当前已测试的配置（`all_targets` + `target_achievement`，θ=0.95/0.80）。其他 GDR/θ 组合的 p(r) 形状未经测量——若某组合的 p(r) 表现为平滑 logistic（存在临界区），Q3 的噪声决策问题将重新出现。

**修改建议**：
1. **默认 N 值削减**（原 A1）：从 1000（GUI）/ 500（引擎）→ 统一 200（[事实四]）。在阶跃 p(r) 下，N=200 已消除全部决策噪声。⚠ 此前估计为 500→200，实际 GUI 默认是 1000（5× 浪费）
2. **Wald SPRT 优先级降低但保留**：当前配置下 SPRT 的早停空间被阶跃函数压缩至接近零。但此结论仅覆盖 2/13 GDR——`resource_efficiency` 已证明不同 GDR 的 p(r) 行为截然不同。应在更广泛的 GDR/θ/配置组合上完成**SPRT 集成检验**（参照 [实验 7] 的方法，但使用真实 `GachaService`）后再决定是否实施或搁置
3. **阶跃函数监控**：在二分搜索中增加「决策确定性」监控——若连续 N 步 Bernoulli 决策的 p 值均远离临界区，报告「p(r) 表现为阶跃函数，N 值可酌减」

**SPRT 技术方案（供集成检验参考）**：

Wald SPRT 对 Bernoulli 试验使用似然比检验：
```
LR = (p1^k × (1-p1)^(n-k)) / (p0^k × (1-p0)^(n-k))
其中 p1 = θ + δ, p0 = θ - δ, k = 成功次数, n = 模拟次数
决策：LR > (1-β)/α → 接受 H₁；LR < β/(1-α) → 接受 H₀
```

[实验 7] 的实证校准（解析 logistic 函数，200 次重复/档）：

| 方案 | 偏差 | 标准差 | 决策正确率 | 平均总模拟次数 |
|------|------|--------|----------|-------------|
| 固定 N=500 | −65 | 286 | 87.3% | 4,500 |
| SPRT δ=0.02 | +129 | 418 | 86.2% | 1,494 |
| SPRT δ=0.05 | +1,874 | 1,862 | 75.8% | 179 |

**若未来实施 SPRT，必须硬编码 δ_min = 0.02 下限保护**——δ=0.05 导致偏差 28× 恶化。

#### Q5: 贪心前沿 ≠ 真正 Pareto 前沿 ⚔ 严重

**现状**（`PlanSearchEngine.search_pareto()`，`retreat_search.py:513-613`）：
- 沿单一贪心路径探索子集：后退法从全量目标集逐一移除，前进法从最高 desire 单卡逐一添加

| 7 | 128 | ~16.9 h | ~63 min |
| 10 | 1,024 | ~135 h（不可行） | ~87 min |
| 15 | 32,768 | ~4,324 h（不可行） | ~127 min |

全枚举在 N≥10 时不可行（> 5 天），这验证了贪心近似的必要性——但也要求我们诚实地声明其局限性。

**修改建议（两阶段）**：

**阶段一（低风险，立即执行）——UI 降级声称 + 支配性过滤（可视化增强）**：

2. **支配性过滤**（暂时撤回，留待阶段二）：单方向贪心路径上任意两点互不支配（目标集⊂、资源↗同向变化），过滤为死代码。阶段二双向合并搜索时前进+后退点集合并，支配关系自然出现，届时再引入。

#### Q6: 翻倍上界搜索可能失败 ⚔ 中等

**原始编号**：25, R3

**现状**（`retreat_search.py:243-265`）：
- 翻倍 10 次：从 `max(upper_bound, cost_per_draw)` 开始逐次翻倍直到 P ≥ θ 或 10 次上限

**实证发现**：
- [事实一] 中 6 目标 SSR 配置的阈值交叉在 ~72,000 资源——GUI 默认 55,000 不足，但翻倍机制正常工作（1 次翻倍 → 110,000 找到上界）
- [事实三] 中 `resource_efficiency` 翻倍至 153,600,000 仍未收敛——确认翻倍机制在退化 GDR 上会耗尽

**修改建议**（✅ 已实施 2026-06-10——提取 `_find_upper_bound()` 方法 + 翻倍 10→15 + 上界不足失败标记）：
1. 翻倍次数从 10 提升至 15（最大 ≈ 8,000 × 2^15 ≈ 262M 资源单位）——与旧 `resource_search_panel.py` 的 15 次行为对齐
2. 提取 `_find_upper_bound(initial_guess, cost_per_draw, max_iter=15)` 独立方法
3. 上界不足时设置 `result.success = False`，在 progress_callback 中报告「搜索上界不足——实际所需资源可能超过当前估计范围」
4. 基于保底参数计算解析上界作为安全网：`max_draws = n_cards × max(各池硬保底阈值, default=90)` → 见下方第 5 条（原 A2，v3.3 公式改为解析导出）
5. **默认上界动态估计**（原 A2，v3.3 公式改为解析导出）：原 A2 使用 `max(55000, n×80×cost)` 基于 [实验 C] 的 6 配置收敛点测量（"实测 77 抽/卡"）。**此论证不成立**——单配置（`start=80, end=90, reset=featured_ssr, cost=160`）无法泛化到任意保底/池子设置。实验 B 证明单改 `reset=never` 即改变跳跃数；实验 D 证明单改保底类型即偏移收敛点 +10,000。替代方案——基于保底参数的解析最坏情况：

   ```python
   def _estimate_upper_bound(self, n_cards, cost_per_draw):
       """基于保底参数计算解析最坏情况上界。不依赖任何实测数据。"""
       max_pity = max(
           (p.threshold for pool in self._pools 
            for p in pool.pity_configs if p.type == 'hard'),
           default=90
       )
       max_draws = n_cards * max_pity       # 最坏情况：每卡一个完整保底周期
       max_resource = max_draws * cost_per_draw
       return max(55000, int(max_resource * 1.5))  # 1.5× 安全余量

   # 对默认配置（软保底 80–90, 6 卡, cost=160）：
   #   解析值 = 6 × 90 × 160 × 1.5 = 129,600
   #   实测收敛 ≈ 73,390（1.77× 过度——故意保守的）
   #   对 reset=never（实验 B）：解析值同样保守（90+5=95 抽, n×90=540 抽）
   ```

   1.5× 安全余量覆盖软保底末端的非确定性。此公式总是保守的——上界搜索只需保守估计，不需要精确。

#### Q10: 双权重系统的基数解释缺失 ⚔ 中等

**原始编号**：R5

**现状**（`PlanSearchPanel` 双列权重表 + `PlanSearchEngine` 两组权重参数）：

| 权重 | 参数 | 排序方向 | 使用的搜索路径 |
|------|------|---------|--------------|
| 抽取意愿（desire） | `desire_weights: Dict[str, float]` | 降序——最想要的优先加入 | `search_max_targets_forward()`、`search_pareto(direction='forward')` |

3. 排序不一致提示：当 desire 和 miss_cost 排序不同时，UI 显示「两种权重给出的卡牌优先级不同，前进法和后退法可能探索不同的子集路径」

#### Q12: `all_targets` 在高成功率区间辨别力退化 ⚔ 低等（⬇️ 2026-06-10 重评估降级）

**原始编号**：N5

**现状**：
- `all_targets` GDR 是二值的——成功 = 1.0（全部目标卡获取），失败 = 0.0（缺失任意一张）
- 理论推演：若资源充裕到几乎所有模拟都返回 1.0，`compute_success_probability()` 对任何高于饱和点的资源值均返回 ~0.99——二分搜索失去辨别力

**实证检验**（[实验 6]，解析饱和函数，200 次重复）：
- 饱和函数 p(r) = 1−0.5·exp(−0.001·(r−5000)) 与对照 logistic 对比
- 标准差差异不显著（172 vs 201），「过早停止」比例仅 2.5%（5/200）
- **解析函数上未观察到明显的辨别力退化**

此结果不排除实际抽卡模拟器中出现饱和效应的可能性——解析函数的 Bernoulli 噪声无法模拟多池交互、保底链式触发等复杂噪声结构。但 [事实一]（三级阶梯函数）实际上从另一个方向解决了此问题：p(r) 在第三跳跃直接从 0.395 跳到 1.000，不存在 p ≈ 0.99 的「饱和区」——饱和担忧在阶跃 p(r) 下不成立。

**修改建议（⬇️ 2026-06-10 重评估降级）**：

原 v3.1 建议为运行时 `_emit` 预防性警告（`if final_prob > 0.99`）。重评估结论——**降级为代码注释**。理由：

1. **阶跃 p(r) 使运行时警告为死代码（或假阳性）**：[实验 C] 6 卡数据中确有一个 p=0.995 点——`prob > 0.99` **技术上会触发**。但此触发是**假阳性**：p=0.995（200 次模拟中 199 次成功）在 Bernoulli 决策（阈值 0.95）时 100% 确定——二分搜索未失去辨别力。真正的饱和退化需要 p(r) 在连续区间内 ≈ 0.99 且存在采样噪声，阶跃函数中不存在这样的区间。
2. **阈值 0.99 缺乏依据**：任意选择的阈值——无理论或实证支撑为什么是 0.99 而非 0.999 或 0.95。
3. **维护负担**：每个 `search_min_resource()` 调用执行此检查但永不触发——增加代码路径却无实际效用。

替代方案——在 `search_min_resource()` 方法 docstring 中添加说明：

> 对于二值 GDR（如 all_targets），当 p(r) 非阶跃函数时，可能在高资源区间出现辨别力退化（p ≈ 1.0 时二分搜索失去方向）。当前默认配置的 p(r) 为阶跃函数，不存在此问题。若未来引入平滑 p(r) 的 GDR，建议切换到连续型 GDR（如 target_achievement）。

**触发条件**：仅当有用户报告实际饱和问题 + 附带该配置的实证数据确定触发阈值时，再升级为运行时警告。

### D 类：工程实现（影响效率、健壮性或可维护性）

#### Q13: epsilon 精度设置缺乏统计依据 ⚔ 低等

**原始编号**：R6

**现状**（`retreat_search.py:216`）：
- `epsilon = cost_per_draw × precision_draws`，默认 `precision_draws = 1`
- 对于 160 成本的单抽，epsilon = 160 资源单位
- 当前 epsilon 与噪声水平的关系未经过形式化论证——既不是噪声的倍数，也不保证二分搜索在噪声水平下收敛

**修改建议**：
1. 将 `precision_draws` 默认值从 1 提升至 5（epsilon ≈ 800 资源单位）
2. 基线检测后利用 SE 估计计算建议精度值——`epsilon = max(cost_per_draw, z × SE × resource_range)`
3. UI 参数区显示「当前精度 ≈ XXX 资源；估计噪声 ≈ XXX 资源」

### E 类：运行时监控（信息性增强，不修复缺陷）

#### Q16: 阶跃函数监控 ⚔ 低等

**原始编号**：A3（原改进建议，v3.3 独立为 Q16）

**现状**：
- [事实一] 和 [事实二] 确认当前默认配置的 p(r) 为三级阶梯函数——所有 Bernoulli 决策的 p 值均远离临界区 [θ−0.10, θ+0.10]
- 但此发现目前仅作为计划文件中的文档记述——未在运行时的进度消息中向用户传递
- 用户没有机会知道「当前 p(r) 是阶跃函数」——无法据此调整 N 值以加速搜索

**修改建议**：

在二分搜索中跟踪连续 Bernoulli 决策的 p 值分布：

```python
# 在 search_min_resource() 中添加
_eval_p_vals = []  # 收集所有 Bernoulli 决策点的 p 值

# 每次 _simulate_with_resource 调用后
_eval_p_vals.append({'r': r_mid, 'prob': prob, 'n': self.num_simulations})

# 二分搜索结束后分析
far_from_threshold = sum(
    1 for pt in _eval_p_vals
    if pt['prob'] <= self.success_threshold - 0.10 
    or pt['prob'] >= self.success_threshold + 0.10
)
if far_from_threshold == len(_eval_p_vals) and len(_eval_p_vals) >= 3:
    _emit(f"所有 {len(_eval_p_vals)} 步 Bernoulli 决策的 p 值均远离阈值 "
          f"({self.success_threshold}±0.10)——p(r) 表现为阶跃函数。"
          f"可考虑降低模拟次数以加速搜索。", 20)
```

**为何不合并到 Q2**：Q2 关注的是「p(r) 存在跳跃→二分搜索可能被锚定」——这是算法正确性问题。Q16 关注的是「p(r) 远离临界区→N 值可削减」——这是计算效率的运行时提示。两者使用同一个 `_eval_points` 收集机制，但监测目标和消息内容不同。

**优先级**：低——纯信息性增强，不影响搜索结果正确性。
