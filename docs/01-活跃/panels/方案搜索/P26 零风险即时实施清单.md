### 1.1 Q13：epsilon 精度调整

> 对应理论计划 §2 Q13 + 阶段一 S1-5 | 优先级：最高（零依赖，3 行改动）

**位置**：[`PlanSearchEngine.__init__()` L86](gacha_simulator/core/retreat_search.py#L86)、[`PlanSearchWorker.__init__()` L130](gacha_simulator/gui/plan_search_panel.py#L130)、GUI `precision_spin`

**问题**：`precision_draws` 默认值为 1（epsilon = 160 资源单位）。在阶跃 p(r) 下不影响正确性，但导致二分搜索多跑 2–3 轮无谓迭代后才终止。

**噪声分析**（2026-06-10 补充）：

| N | SE(p̂) @ p=0.5 | SE(r) @ 中等斜率* | epsilon=160 | epsilon=800 |
|---|---------------|-------------------|-------------|-------------|
| 200 | 0.035 | ~71 资源 | 2.3× SE | 11× SE |
| 500 | 0.022 | ~45 资源 | 3.6× SE | 18× SE |
| 1000 | 0.016 | ~32 资源 | 5× SE | 25× SE |
| 10000 | 0.005 | ~10 资源 | 16× SE | **80× SE** ← 明显浪费 |

> *斜率假设：p(r) 在 ~2000 资源区间内从 0→1

- **阶跃 p(r)**：epsilon 任意值均不影响正确性——p(r) 在跳跃点从不连续，`p(r_mid) ≥ θ` 的 Bernoulli 决策 100% 确定
- **平滑 p(r)**：SE(r) ∝ 1/√N，大 N 下精细 epsilon 有意义。`1→5` 是保守调整，对当前 N=500–1000 合理；若未来 N≥5000，需 Q13 建议 2（自适应 epsilon）替代硬编码
- **Q3 建议 1 联动**：若 N→200，噪声增大，epsilon=800 更合理（11× SE）

**修改**（三处同步）：

```python
# PlanSearchEngine.__init__ (L86)
precision_draws: int = 5  # 原: 1

# PlanSearchWorker.__init__ (L130)
precision_draws: int = 5  # 原: 1

# GUI QSpinBox 默认值 (plan_search_panel.py)
self.precision_spin.setValue(5)  # 原: 1
```

**验证**：同一配置下运行搜索，确认精度值在结果页正确显示（`±800` 而非 `±160`）。

### 1.2 Q2 + Q16：保底跳跃双向检测 + 阶跃函数监控

#### 步骤二：Q2 跳跃检测

**问题**：[事实一] 确认实际 p(r) 含三级阶梯跳跃（3.5×SE、17.6×SE、35.0×SE），第三跳跃远超几何锚定危险阈值（≥6×SE）。二分搜索在跳跃点附近可能被「锚定」到跳跃边缘而非真实阈值穿越点。

**修改**（二分搜索结束后执行）：

```python
import math
if len(_eval_points) >= 3:
    sorted_pts = sorted(_eval_points, key=lambda p: p['r'])
    for i in range(len(sorted_pts) - 1):
        r_a, p_a, n = sorted_pts[i]['r'], sorted_pts[i]['prob'], sorted_pts[i]['n']
        r_b, p_b = sorted_pts[i+1]['r'], sorted_pts[i+1]['prob']
        delta = abs(p_b - p_a)
        if delta < 0.05:
            continue  # 忽略微小差异
        se_avg = max((math.sqrt(p_a*(1-p_a)/n) + math.sqrt(p_b*(1-p_b)/n)) / 2, 1e-6)
        ratio = delta / se_avg
        if ratio >= 6.0:
            _emit(f"⚠ 检测到大跳跃: r [{r_a:.0f}, {r_b:.0f}] "
                  f"p {p_a:.3f}→{p_b:.3f} ({ratio:.1f}×SE)。"
                  f"二分搜索结果可能受几何锚定影响——建议验证。", 85)
        elif ratio >= 3.0:
            _emit(f"检测到小跳跃: r [{r_a:.0f}, {r_b:.0f}] "
                  f"p {p_a:.3f}→{p_b:.3f} ({ratio:.1f}×SE)。", 30)
```

**验证**：6 目标卡配置下运行搜索，确认收到跳跃提示（预期：1 个小跳 + 2 个大跳警告）。

#### 步骤三：Q16 阶跃函数监控

**问题**：[事实一] 和 [事实二] 确认 p(r) 为阶跃函数——所有 Bernoulli 决策的 p 值均远离临界区。但用户无从知晓——无法据此削减 N 值加速搜索。

**与 Q2 的区别**：Q2 监测「p(r) 跳跃→二分搜索可能被锚定」（正确性风险）；Q16 监测「p(r) 远离临界区→N 值可削减」（效率提示）。共用 `_eval_points` 但监测目标不同。

**修改**（在 Q2 跳跃检测之后）：

```python
far_from_threshold = sum(
    1 for pt in _eval_points
    if pt['prob'] <= self.success_threshold - 0.10
    or pt['prob'] >= self.success_threshold + 0.10
)
if far_from_threshold == len(_eval_points) and len(_eval_points) >= 3:
    _emit(f"所有 {len(_eval_points)} 步 Bernoulli 决策的 p 值均远离阈值 "
          f"({self.success_threshold}±0.10)——p(r) 表现为阶跃函数。"
```

**验证**：默认配置下运行搜索，确认收到阶跃函数提示（预期触发——默认 p(r) 确为阶跃）。

### 1.4 Q3 建议 1：N 值削减

> 对应理论计划 §2 Q3 修改建议第 1 条 | 优先级：中（无代码前置，需决策确认）

**问题**：[事实一] 和 [事实二] 确认 p(r) 为三级阶梯函数——p 值仅取 {0, 0.015, 0.395, 1.000}，所有 Bernoulli 决策 100% 确定（p 值远离临界区 0.95 ± 0.05）。当前 N=1000（GUI）/ 500（引擎）是 **5× 浪费**。

**修改**（三处同步）：

**安全性论证**：
- 阶跃 p(r) 下 Bernoulli 噪声不影响决策正确性——p=0.395 和 p=1.000 距离阈值 0.95 的差距远超 N=200 时的 SE（≈0.035），不存在误判风险

**与 Q3 完整 SPRT 方案的关系**：SPRT（序贯概率比检验）是比简单 N 削减更根本的效率改进——每步自适应决定所需模拟次数。但因跨 GDR 集成检验未完成（当前仅覆盖 2/13 GDR），SPRT 留在阶段二。

### 1.5 Q6 建议 5：解析上界公式

> 对应理论计划 §2 Q6 修改建议第 5 条 | 优先级：中（独立）

**问题**：当前上界初始值硬编码 `upper_bound=8000`（引擎）/ `55000`（GUI）。原 A2 建议用实测数据拟合——已被拒绝（单配置无法泛化，见附录 A）。替代方案：从保底参数直接解析导出。

**位置**：[`PlanSearchEngine.__init__()`](gacha_simulator/core/retreat_search.py#L89) 和 [`_find_upper_bound()` 初始值](gacha_simulator/core/retreat_search.py#L219)

**修改**：

```python
def _estimate_upper_bound(self, n_cards: int, cost_per_draw: float) -> float:
    """基于保底参数计算解析最坏情况上界。不依赖任何实测数据。"""
    max_pity = max(
        (p.threshold for pool in self._pools
         for p in pool.pity_configs if hasattr(p, 'threshold')),
        default=90
    )
    max_draws = n_cards * max_pity       # 最坏情况：每卡一个完整保底周期
    max_resource = max_draws * cost_per_draw
    return max(55000, int(max_resource * 1.5))  # 1.5× 安全余量
```

在 `__init__` 或首次 `search_min_resource()` 调用时计算并替换 `self.upper_bound`。

**与现有 `_find_upper_bound()` 的关系**：解析公式提供更好的初始猜测，减少翻倍迭代次数。`_find_upper_bound()` 的翻倍逻辑保留作为 fallback——若解析值不足，翻倍搜索仍会继续。

**默认配置参考值**：6 卡 × 90 保底 × 160 成本 × 1.5 = 129,600（vs 实测 73,390——1.77× 过度，故意保守）。

### 为何 A2 公式改为解析导出

**重评估结论**：降级为代码注释。理由：
1. 阶跃 p(r) 使警告为假阳性——p=0.995（200 次中 199 次成功）技术上触发但 Bernoulli 决策（阈值 0.95）100% 确定

### B.5 Q5：支配性过滤 —— 撤回（2026-06-10）

**结论**：支配性过滤在单方向搜索中是**空转**。

贪心路径的单调性保证了一条路径上的任意两点互不支配：前进法目标集递增、资源也递增；后退法目标集递减、资源也递减。支配要求「目标 ⊇ 且资源 ≤」——两者同向变化，永不满足。

支配关系仅在双向合并时可能出现（前进法的 `{A,B,C}` 可能支配后退法的 `{A,B}`）——但当前 `search_pareto()` 一次只走一个方向，两个方向的点永不共存。

**撤回内容**：`RetreatSearchPoint.dominated` 字段 + `search_pareto()` 中 13 行 O(n²) 遍历。阶段一实际完成项仅 UI 术语修正（Pareto 按钮文本）。
