## P1 — 高优先级缺陷

### P1-01：面板独立王国——无法与其他面板联动

- **来源**：P19 问题诊断第 3 条
- **严重度**：高（架构层面）
- **现状**：`StrategyComparisonPanel` 自己启动模拟，与其他面板零交互。比较结果无法溯源到统计分析/过程分析/退路分析。用户无法「在策略比较中看到一个异常策略 → 点进去看它的 GDR 分布或过程分析」
- **计划方案**：P19 第一阶段——ResultStore 集中式数据管理 + 数据管理 Tab，使所有面板共享同一批数据集
- **状态**：待实施（P19 第一阶段）

### P1-02：成功率定义与 GDR 框架脱节

- **来源**：代码审查
- **严重度**：高（可能误导排名）
- **现状**：`_compute_summary()` 中 `success_count = sum(1 for a in target_acquired_list if a > 0)` 硬编码为「至少获得 1 张目标卡」。用户在其他面板使用的 GDR 指标（如 `target_achievement=0.8`、`resource_efficiency` 等）和阈值在此完全无效
- **后果**：同一批模拟数据，在策略比较面板显示「策略 A 成功率 75%」，但在统计分析面板可能显示「策略 A GDR 成功率 60%」——两个数字来自不同的成功定义，用户困惑
- **计划方案**：P19 统一使用 GDR 框架判定成功/失败
- **状态**：待实施

### P1-03：无统计检验——肉眼比较不可靠

- **来源**：P19 问题诊断第 3 条 + L1
- **严重度**：高（结论可能被随机噪声主导）
- **现状**：仅展示 5 个点估计（均值），无置信区间、无假设检验、无效应量。当 N=500 时，均值的 95% CI 可能宽达数个抽数——两个策略的平均抽数差 5 抽可能完全在噪声范围内
- **计划方案**：P19 第二阶段——ComparisonAnalyzer 引擎（KS/MWU/置换检验/配对 t 检验/Cohen's d + 多重比较校正）
- **状态**：待实施（P19 第二阶段）

### P1-04：无配对/反事实分析

- **来源**：P19 关键设计决策第 5 条
- **严重度**：高（丢弃了大量信息）
- **现状**：所有策略使用 seed=0 但独立运行——随机数序列因策略决策不同而分叉，无法做配对比较。同一用户可能想知道「在完全相同的运气下，策略 A 比策略 B 多抽了多少次？」
- **计划方案**：P19 配对模式——同种子 + 同 N 时自动启用配对 t 检验和反事实遗憾分析
- **状态**：待实施（P19 第二阶段）

## P2 — 低优先级改进

### P2-01：指标贫乏——仅有 5 个均值

- **来源**：代码审查
- **严重度**：低等（功能缺失）
- **现状**：仅计算 avg_draws / avg_pity_triggers / avg_target_acquired / avg_final_resource / success_rate
- **缺失**：标准差、分位数（中位数/P25/P75）、VaR/CVaR、各池维度细分、GDR 分布
- **计划方案**：P19 ComparisonAnalyzer 使用 PerSimValues 中更丰富的每模拟数据
- **状态**：待实施

### P2-02：无持久化——结果随面板关闭丢失

- **来源**：P19 问题诊断第 1/2 条
- **严重度**：低等（UX 问题）
- **现状**：比较结果仅在内存中，关闭程序后丢失。无法保存、导入、导出、分享
- **计划方案**：P19 ResultStore 的 `save_to_file()` / `load_from_file()` JSON 持久化
- **状态**：待实施（P19 第一阶段）

### P2-03：无法比较「相同策略 + 不同配置」

- **来源**：P19 关键设计决策第 4 条
- **严重度**：低等（功能缺失）
- **现状**：只能比较不同策略，不能比较「smart 策略 + 月卡党初始资源」vs「smart 策略 + 零氪党初始资源」
- **计划方案**：P19 ResultStore 中数据集为任意组合的模拟结果，比较对象不限于策略
- **状态**：待实施

### P2-04：其他策略使用空参数 `sparams = {}`

- **来源**：代码审查（`ComparisonWorker.run()` line 73-74）
- **严重度**：低等（功能不完整）
- **现状**：
  ```python
  if skey == current_strategy_key:
      sparams = current_strategy_params
  else:
      sparams = {}
  ```
  只有正在配置面板中选中的策略能获得参数（如 `pool_quota` 的配额配置），其他策略以空参数运行——可能不反映其最佳表现
- **建议**：要么为每个策略提供独立的参数配置入口，要么在比较结果中标注「未配置参数，使用默认行为」
- **状态**：未计划

### P2-05：停止按钮的优雅性

- **来源**：代码审查
- **严重度**：低等（UX 细节）
- **现状**：`_should_stop` 标志仅在策略间循环间隙检查，运行中的 `run_batch_parallel()` 不会被中断。如果第一个策略耗时最长，点击停止后仍需等待其完成
- **建议**：使用 `multiprocessing.Pool.terminate()` 或更细粒度的中断机制
- **状态**：未计划
