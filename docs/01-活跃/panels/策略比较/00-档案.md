## 一、业务问题

抽卡游戏中有多种决策策略——按需追卡、保底预留、指定池配额、目标即停、固定次数等等。不同策略在相同的游戏环境中可能产生截然不同的结果。玩家和决策者需要回答：

1. **哪个策略的效率最高？**（平均抽卡数、平均目标获取数、成功率）
2. **哪个策略最稳健？**（在运气好/差的极端情况下表现如何）
3. **策略之间的差异是否统计显著？**还是仅仅是随机波动？
4. **在多目标空间中，各策略的帕累托前沿在哪里？**（效率 vs 风险 vs 资源消耗）

### 关键业务场景

- 攻略组发布「不同抽卡策略的横向评测」
- 玩家根据自身风险偏好选择策略
- 游戏设计师评估不同策略对玩家留存/付费的影响
- 版本更新后（保底规则变更）重新评估策略排名变化

## 二、设计意图

### 2.1 当前实现（v1，即将废弃）

当前 `StrategyComparisonPanel` 是一个**独立王国**：

- 拥有自己的 `ComparisonWorker` QThread，自己运行模拟
- 从 `ConfigStore` 直接读取配置，构建 `SimulationEnv`
- 依次为每个选中策略运行 `run_batch_parallel()`
- 仅计算 5 个朴素汇总指标：平均抽卡数、平均保底触发、平均目标获得、成功率、平均剩余资源
- 在 QTableWidget 中展示 7 列表格（策略名 | 模拟次数 | 5 个指标）
- 无图表、无统计检验、无高级比较

**核心缺陷**：

1. 数据孤岛：自己跑自己的模拟，与其他面板零交互。策略比较结果无法溯源到统计分析/过程分析/退路分析
2. 指标贫乏：只有均值，无分布、无置信区间、无假设检验
3. 无配对分析：不同策略使用不同随机种子运行，无法做反事实比较（同一随机种子下策略 A vs 策略 B）
4. 无持久化：结果无法保存/加载/导出
5. 无法比较配置差异：只能比较策略，不能比较「相同策略 + 不同初始资源」的场景

### 2.2 目标架构（v2，P19 计划中）

参见 `比较分析面板重构与结果数据管理层设计.md` (P19)。

核心设计理念：

```
模拟与分析彻底分离
  ↓
ResultStore（集中式数据管理）
  ├─ 数据集 A：策略 smart × 配置 V1 × N=1000
  ├─ 数据集 B：策略 pool_quota × 配置 V1 × N=1000
  └─ 数据集 C：策略 smart × 配置 V2 × N=1000
       ↓
比较分析面板（从 ResultStore 选取 2+ 数据集）
  ├─ 配对模式（同种子+同N）→ 反事实遗憾 + 配对 t 检验
  └─ 分布模式（不同种子）→ KS 检验 + 随机占优 + MWU
```

**比较范围扩展**：不仅比较策略，也支持比较「同一策略 + 不同配置」、「同一配置 + 不同初始资源」等任意数据集组合。

### 2.3 比较分析引擎（ComparisonAnalyzer）

P19 计划在 `core/` 中新增独立模块，从各数据集的 `PerSimValues` 中提取指标进行成对比较：

**两种模式自动检测**：

| 模式 | 条件 | 方法 |
|------|------|------|
| 配对模式 | seed 相同 + N 相同 | 反事实遗憾、配对 t 检验、Wilcoxon 符号秩 |
| 分布模式 | 否则 | KS 检验、Mann-Whitney U、置换检验、随机占优、Cohen's d |

**多重比较校正**：所有假设检验输出原始 p 值 + Holm-Bonferroni（FWER）+ Benjamini-Hochberg（FDR）三组矩阵。

**可视化**：

| 图表 | 内容 |
|------|------|
| CDF 叠加图 | 选定指标的所有数据集 CDF 曲线叠加 |
| 帕累托散点图 | X=期望收益，Y=风险(CVaR)，每个数据集一个点 |
| 效应量热力图 | n×n 矩阵，颜色深浅 = Cohen's d |
| 指标对比明细表 | 每行一个数据集，均值/中位数/标准差/CVaR₅%/成功率 |
| 占优矩阵 | FSD/SSD 随机占优关系 |

## 三、技术实现（v1 当前）

### 3.1 当前文件结构

| 组件 | 文件 | 职责 |
|------|------|------|
| `StrategyComparisonPanel` | `gui/strategy_comparison_panel.py` | GUI 面板：策略选择 + 参数配置 + 结果表格 |
| `ComparisonWorker` | `gui/strategy_comparison_panel.py` | QThread：顺序运行每个选中策略的模拟 |

### 3.2 当前数据流

```
用户选择策略（多选复选框） + 设置 N/workers
       │
       ▼
StrategyComparisonPanel._on_run_clicked()
  │
  └─ 创建 ComparisonWorker(strategy_keys, num_simulations, config_store, max_workers)
       │
       ▼
     ComparisonWorker.run()  ← QThread
       │
       ├─ SimulationEnvBuilder.from_config_store(config_store)
       ├─ 提取 target_specs、current_strategy_key、current_strategy_params
       │
       └─ for each strategy_key:
            ├─ run_batch_parallel(env, ..., strategy_name=key, strategy_params=...)
            ├─ _compute_summary(histories, display_name)
            │    ├─ 遍历每次模拟的 dict/CompactResult
            │    ├─ 计算 total_draws / pity_triggers / target_acquired / final_resource
            │    └─ 返回 {display_name, count, avg_draws, avg_pity, avg_target, avg_resource, success_rate}
            └─ results[key] = summary
            │
            ▼
       finished.emit(results)
            │
            ▼
     StrategyComparisonPanel._on_finished(results)
       ├─ 按成功率降序排序
       ├─ QTableWidget 填充 7 列
       └─ 最优行绿色高亮
```

### 3.3 `_compute_summary()` 的当前逻辑

```python
def _compute_summary(self, histories, display_name):
    # 遍历 histories（List[dict] 或 List[CompactResult]）
    # 提取每次模拟的: total_draws / pity_triggers / draw_resource / target_acquired
    # 计算 5 个均值 + 成功率 = sum(acquired > 0) / N
    # 成功率判定：仅基于「目标卡获得数 > 0」——硬编码，不使用 GDR 框架
```

**问题**：成功率定义硬编码为「至少获得 1 张目标卡」，与面板的 GDR 框架脱节——用户在其他面板使用的 GDR 指标和阈值在此不适用。
