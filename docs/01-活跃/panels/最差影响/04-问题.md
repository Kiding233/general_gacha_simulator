## P1 — 高优先级缺陷

### P1-01：条件分布 → 新模拟的逻辑跳跃（缺陷 27）

- **来源**：P31 第 2.1 节
- **严重度**：中等（方向性偏差——高估连续成功池数）
- **现状**：从清空历史的新模拟比真实失败场景更有利（保底归零、兑换完整）
- **偏差估计**：可能高估连续成功池数 1-3 个
- **计划方案**：短期 UI 警告标注 → 中期条件保底计数器初始化 → 长期接续模拟框架
- **状态**：设计中，未实施

### P1-02：虚拟池同质化假设（缺陷 22 / 28）

- **来源**：P31 第 2.2 节
- **严重度**：中等（系统性建模偏差）
- **现状**：99 个虚拟池共享相同成本/概率/持续时间，消除「高成本池连续」失败模式
- **计划方案**：阶段 1 UI 标注 + 敏感度滑块 → 阶段 2 可配置池参数序列 → 阶段 3 历史数据拟合
- **状态**：设计中，未实施

### P1-03：MAX_POOLS=99 硬截断（缺陷 21）

- **来源**：P31 第 2.3 节
- **严重度**：中等（高储蓄玩家 expected_pools 低估）
- **现状**：成功次数分布在 K=99 处截断
- **计划方案**：动态上限 + 截断检测警告
- **状态**：设计中，未实施（P14 时曾搁置，P17 后需重新评估）

## P2 — 低优先级改进

### P2-01：无资源收入假设（缺陷 23）

- **来源**：P31 第 2.4 节
- **严重度**：低等（保守下界，对「最差影响」是合理默认）
- **现状**：`resource_gain=None`，绝对连续成功池数被低估
- **计划方案**：UI 标注说明假设，不改变默认行为
- **状态**：设计中，未实施

### P2-02：自定义资源模式下无验证

- **来源**：代码审查
- **严重度**：低等
- **现状**：用户在「自定义初始资源」输入框输入任意数值，无范围/合理性检查
- **建议**：添加输入验证（如非负数、合理上限）

### P2-03：单线程嵌套模拟性能

- **来源**：代码审查
- **严重度**：低等（UX 问题，非正确性问题）
- **现状**：虚拟池模拟在面板后台线程中 max_workers=1，1000 次 × 99 池可能耗时数分钟
- **建议**：考虑多进程并行（需解决嵌套 Pool 问题）或增量结果更新（每 100 次刷新一次中间图表）

## 已完成的修复

### P17：最差影响保底匹配修复

- **状态**：**已完成**
- **修复内容**：虚拟池 ID 与保底规则的通配符匹配静默失败问题——`fnmatch.fnmatch(pid, pdef.pools)` 现在正确匹配 `*` 模式
- **影响**：修复后保底正确触发，连续成功池数可能显著增加，放大了 P1-03（MAX_POOLS 截断）和 P1-01（逻辑跳跃）的影响
