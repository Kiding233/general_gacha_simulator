## 一、业务问题

用户在抽卡游戏中面临的核心焦虑：「如果运气很差，我的资源能撑过几个后续池子？」

具体而言：

1. **最差情况资源评估**：给定当前储蓄和一批历史模拟结果，在「运气差到只剩 α 分位资源」时，初始资源为多少？
2. **保底覆盖倍数**：这最差资源量能覆盖多少次「大保底到顶」的抽卡消耗？（即「大保底覆盖倍数」= 最差资源 / 单保底全耗）
3. **连续成功池数**：以此最差资源为初始预算，能连续成功抽取（获得目标限定卡）多少个后续新池子？
4. **条件分解**：分别评估「历史成功组」和「历史失败组」的最差资源水平（条件分布），而非笼统的全样本分位数

### 关键业务场景

- 玩家在版本末期决定是「全押当前池」还是「留资源保下一个池」
- 攻略组评估「月卡党在最差情况下能拿几个限定」
- 风险厌恶型玩家寻求保守的资源规划下界

## 二、设计意图

### 2.1 核心理念：条件分布 + 重置模拟

最差影响分析分两步走：

```
步骤 1：条件资源分位数提取
  批量模拟结果 → 按 GDR 成功/失败分类 → 取条件资源分布的 α 分位数 q_α
  （例：condition=failure, α=0.05 → 「失败路径中，运气最差的 5% 的人还剩多少资源？」）

步骤 2：以 q_α 为初始预算的新模拟
  构建 99 个完全相同的虚拟池 → 从清空历史（保底归零）开始模拟 →
  统计「能连续获取多少个池的限定卡」
```

两步分离的设计源于一个认识：原批量模拟只覆盖了「过去一段时间的池子」，用户关心的是「未来还有多少池子能拿下」。无法直接从原模拟结果中回答，只能构建未来场景的独立模拟。

### 2.2 为什么要条件分布

不条件于成功/失败，全样本的 α 分位数会混合两种截然不同的情况：

- 成功组资源低 → 可能因为「拼尽全力抽到了，所以资源消耗大」
- 失败组资源低 → 「运气差+资源少，双重打击」

用户关心的通常是**失败组**的最差情况——「如果我没抽到目标，资源还剩多少？」这比「无论成败」的笼统分位数更有决策价值。

### 2.3 虚拟池设计

99 个虚拟池（`MAX_POOLS=99`）模拟连轴转的后续池子。每个池：

- 卡牌概率分布复制自当前最后一个池子的配置
- 限定卡 ID 为 `_wi_featured_{i}`（i=0..98），确保每个池成功判定独立
- 时间上首尾相接（每个持续 T 天，T 取自池配置）
- 保底机制复用原始配置，通配符匹配到虚拟池 ID
- 停止条件：连续 pool 目标未达成则终止（`ConsecutivePoolTargetCondition`）

## 三、技术实现

### 3.1 数据流

```
用户配置面板 → 编辑 GDR 指标 + 阈值 + α + 条件 + 模拟次数 + 虚拟池参数
       │
       ▼
WorstImpactPanel._on_run()
  │
  ├─ 构建 WorstImpactAnalyzer(模拟结果, 目标卡, store, GDR配置, 虚拟池配置)
  │
  └─ 启动 WorstImpactWorker
       │
       ▼
     WorstImpactAnalyzer.analyze(condition, alpha, num_simulations)
       │
       ├─ _prepare_pool_info()           ← 读取原始池卡牌分布/保底/成本
       ├─ _build_success_checker()       ← 用 GDR key+threshold 判定成功
       ├─ ConditionalResourceDistribution ← 条件资源分位数
       ├─ _compute_pity_coverage()       ← 保底覆盖倍数
       ├─ prepare_simulation_config()    ← 构建 99 个虚拟池 + 保底引擎
       ├─ run_batch_parallel()           ← 以 draw_target 策略运行新模拟
       ├─ analyze_batch_results()        ← 统计连续成功池数的分布
       └─ 返回 WorstImpactResult
            │
            ▼
       WorstImpactPanel._on_finished()
         ├─ PityCoverageGauge 更新
         ├─ 摘要文字（最差资源 / 覆盖倍数 / 期望池数）
         ├─ 详情表（k / P(X=k) / P(X>=k) / 累计概率）
         └─ Plotly 柱状图（新池子数分布 + 期望值标记线）
```

### 3.4 关键细节

- **多保底匹配**：`_build_pity_engine()` 对每个虚拟池使用 `fnmatch.fnmatch(pid, pdef.pools)` 匹配保底规则的通配符模式——P17 修复后确保虚拟池 ID 正确匹配 `*` 模式
- **目标解析**：`_resolve_targets_for_pool()` 将保底配置中的抽象目标（`limited_ssr`/`featured`/`standard_ssr`/`ssr`）解析为具体卡 ID，每个虚拟池的 featured 卡被替换为 `_wi_featured_{i}`
- **SSR 识别**：从 `store.card_defs` 收集所有 SSR 并抽取（`_main_ssr_ids`），用于主模拟的 GDR 判定
- **串行/单进程运行**：虚拟池模拟强制 `max_workers=1`——因为它是面板后台线程内的嵌套调用，不能再开子进程池
- **缺失模拟处理**：`analyze_batch_results()` 中 `n_failed = n - len(aggregate_data)` 补全因资源耗尽未产生完整记录的模拟

### 3.5 自定义虚拟池

用户可通过「编辑分布」按钮自定义虚拟池的卡牌分布和单抽消耗，覆盖从原始池配置复制而来的默认值。
