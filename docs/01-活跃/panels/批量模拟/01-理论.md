# 批量模拟面板 -- 理论

> 创建日期：2026-06-02（首次建档）
> 关联模块：`service/batch_simulator.py`、`core/streaming.py`

---

## 一、核心假设

### 1.1 独立同分布（i.i.d.）假设

每次模拟的随机种子独立（通过 `seed + task_id` 生成），模拟之间不共享任何可变状态。内核假设每次模拟是对同一随机过程的独立观测，因此：

1. **样本均值向总体期望收敛**（大数定律）
2. **样本方差向总体方差收敛**
3. **中心极限定理适用**——样本均值在大 N 下近似正态分布

违反条件：
- 如果使用固定种子且 `task_id` 范围重叠，可能产生重复序列
- 如果 `_wk_init()` 注入的全局变量在模拟之间被意外修改（单进程路径的已知风险）

### 1.2 流式提取的无偏性

`SharedResultCollector` 的提取器（extractor）在每次 `on_result(compact)` 调用时立即提取所需统计量并丢弃 compact 数据。关键假设：

1. **顺序无关**：提取器的累积统计量（如均值、直方图）不依赖 compact 的处理顺序
2. **无信息丢失**：提取器的聚合函数（如 `np.histogram2d` 累加）在数学上等价于先收集全部 compact 再做批量聚合

如果使用顺序敏感的分析方法（如在线方差计算的 Welford 算法），此假设仍成立；但如果需要逐抽细节（如过程分析的事件序列），需在提取器中保留完整序列或通过 `kept_sequences` 字段显式保留。

### 1.3 并行性的正确性

`multiprocessing.Pool` 的 `apply_async` + `SharedResultCollector` 架构假设：

1. **无共享可变状态**：每个子进程独立运行 `GachaService.run_simulation()`，不访问主进程内存
2. **收集器线程安全**：`SharedResultCollector` 的 `on_result()` 方法在子进程中调用，`extraction` 在子进程内累积。主进程通过 `BatchResult.extraction` 在 join 后读取结果
3. **全局变量隔离**：`_wk_init()` 设置模块级全局变量，但子进程的模块级命名空间是 fork 后的副本，修改不影响主进程或其他子进程

### 1.4 no_draw 基线策略

每次批量模拟后额外运行 1 次 `no_draw` 策略模拟（不抽卡，仅收集资源）。假设：

1. **no_draw 模拟与正常模拟共享同一环境**：资源收入函数、初始资源完全一致
2. **no_draw 资源量是理论上限**：`no_draw_resource` 表示在完全不消耗资源的情况下可获得的最大资源量

此基线用于各分析面板中的「资源剩余」类 GDR 指标的参考线。

## 二、已知局限性

### 2.1 Worker 全局变量（历史遗留）

`service/batch_simulator.py` 使用 11 个模块级全局变量（`_wk_pools`、`_wk_schedule_mgr`、`_wk_end_time` 等）通过 `_wk_init()` 注入子进程。虽然策略类已不再直接依赖这些全局变量（通过 `StrategyContext` 传入信息），但 worker 机制仍使用全局变量模式。

**状态**：`docs/03-归档/plans/Worker全局变量消除方案.md` 已制定合并为 1 个 `_wk_env: SimulationEnv` 的方案，但尚未实施。

### 2.2 单进程路径的全局变量污染

当 `max_workers <= 1` 时，`run_batch_parallel()` 直接在主进程调用，全局变量可能被意外修改。这违反了 i.i.d. 假设的并行独立性要求。

**影响**：`no_draw` 基线计算使用 `max_workers=1`，存在此风险。实际影响较低——`no_draw` 策略不抽卡，不读取大多数全局变量。

### 2.3 进度回调的精度

进度信号使用 `N/50` 的批处理间隔，在 N<50 时进度更新可能不精确（每次间隔 ≤ 1 → 每步都 emit，但间隔计算逻辑取 `max(1, N//50)`，N<50 时 `N//50=0`，`max(1,0)=1`，仍为每步 emit）。在 N 极大（>100000）时，每 2000 次才更新一次，用户可能感觉进度条卡顿。
