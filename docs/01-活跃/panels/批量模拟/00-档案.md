## 业务问题

抽卡模拟的核心价值在于「大量重复模拟后的统计推断」——单次抽卡运气成分过高，只有通过成百上千次模拟才能获得可靠的概率估计。批量模拟面板是用户从「配置」到「分析」之间的桥梁：它负责触发放量并行模拟、显示实时进度、将结果分发给下游分析面板。

## 设计意图

### 核心抽象

- **GachaPanel**（`gui/gacha_panel.py`，406 行）：QWidget，提供模拟参数（次数、并行数、随机种子）的 UI 控件、进度条、以及「开始模拟」按钮。模拟完成后通过 `finished` 信号将 `result_bundle` 字典分发给所有注册的下游面板。
- **SimulationThread**（`gui/gacha_panel.py` 内部类）：QThread 子类，将耗时的批量模拟移至工作线程，避免阻塞 GUI 主循环。通过 `progress`/`finished`/`error`/`status_update` 四个信号与 UI 通信。
- **SimulationEnvBuilder**（`service/batch_simulator.py`）：静态工厂类。`from_config_store()` 方法将 `ConfigStore` 转换为 `SimulationEnv`（包含池子列表、保底引擎、资源收入函数、GDRContext 等运行时对象）。
- **BatchResult**（`service/batch_simulator.py`）：包装 `List[CompactResult]` 和可选的 extraction 字典，向后兼容 list 接口。

### 数据流

```
用户点击「开始模拟」
  → GachaPanel._start_simulation()
    → SimulationThread.run()
      → SimulationEnvBuilder.from_config_store(config_store)  [解析配置]
      → run_batch_parallel(env, target_specs, N, workers, seed, ...)  [并行模拟]
        → multiprocessing.Pool(initializer=_wk_init)  [注入全局变量到子进程]
        → 每个子进程: GachaService.run_simulation(collector=CompactCollector())
        → SharedResultCollector.on_result(compact)  [边提取边丢弃]
      → 结果打包 result_bundle
    → finished.emit(result_bundle)
  → GachaPanel._on_simulation_finished()
    → 统计面板、过程分析面板、策略比较面板等接收 result_bundle
```

## 技术实现

### SimulationThread 关键逻辑

1. **环境构建**：`SimulationEnvBuilder.from_config_store(config_store)` — 将配置对象展开为运行时环境
2. **并行模拟**：`run_batch_parallel()` — 使用 `multiprocessing.Pool` 启动 N 个子进程，每个子进程通过 `_wk_init()` 注入模拟所需的全局变量
3. **进度批处理**：每 `N/50` 次模拟 emit 一次 `progress` 信号，减少 Qt 事件循环压力
4. **基线计算**：额外运行 1 次 `no_draw` 策略模拟，作为「资源零消耗」参考线
5. **结果打包**：将 worker 内提取的聚合数据（aggregates、heatmap、cumulative_snapshots 等）打包为 `result_bundle` 字典

### result_bundle 结构

```python
{
    'aggregate_data': List[dict],          # 各模拟的聚合统计
    'draw_sequences': List[dict],          # 保留的逐抽序列（用于过程分析）
    'heatmap_data': dict,                  # 热力图数据（achievement / resource 两维）
    'cumulative_snapshots': dict,          # 逐池累积快照
    'transition_flags': List[dict],        # 转换标记（策略切换记录）
    'target_ids': Set[str],                # 目标卡 ID 集合
    'ssr_ids': Set[str],                   # SSR ID 集合
    'gdr_context': GDRContext,             # GDR 计算上下文
    'pool_end_times': Dict[str, float],    # 各池结束时间
    'target_specs': Dict[str, int],        # 目标卡规格
    'n_results': int,                      # 实际模拟次数
    'n_requested': int,                    # 请求模拟次数
    'no_draw_resource': float,             # 零消耗基线资源量
    'no_draw_pool_resources': dict,        # 零消耗基线各池资源
}
```

### 实施历史

- **v1.0~v1.8**：使用 `multiprocessing.Pool` + worker initializer 模式，策略通过全局变量注入子进程
- **v1.9**：引入 `SharedResultCollector`（`core/streaming.py`）实现边模拟边提取，消除主进程中的冗余数据累积
- **当前状态**：稳定。`SimulationThread` 是全项目唯一触发批量模拟的入口，所有分析面板通过它获取模拟数据

## 关键依赖

| 依赖 | 方向 | 说明 |
|------|------|------|
| `gui/config_panel.py` → `ConfigStore` | 上游 | 模拟输入来自用户编辑的配置 |
| `service/batch_simulator.py` | 被调用 | `run_batch_parallel()` + `SimulationEnvBuilder` |
| `core/streaming.py` → `SharedResultCollector` | 被调用 | Worker 内边提取边丢弃 |
| 所有分析面板 | 下游 | 通过 `finished` 信号接收 `result_bundle` |
