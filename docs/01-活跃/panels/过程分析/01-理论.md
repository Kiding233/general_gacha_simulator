> 创建日期：2026-06-02（首次建档）
> 关联模块：`core/process_trace.py`、`core/process_analysis.py`

## 一、核心假设

### 1.1 事件类型的互斥完备性

7 种事件类型（`pity_hit`、`early_hit`、`miss`、`skip`、`ignore`、`exchange`、`no_exchange`）被设计为在每个池上互斥且完备——任何池在任何一次模拟中必落入唯一的事件类型。

**判定优先级**（按代码逻辑）：池类型 > 抽卡次数 > 目标卡命中 > 保底触发

1. 资源池 → `resource_draw` / `resource_ignore`（不在 7 种事件中，跳过）
2. 兑换池 → `exchange` / `no_exchange`
3. 抽卡池 → 0 抽 → `skip` / `ignore`（由 `_resolve_skip_ignore()` 判定）
4. 抽卡池 → >0 抽 + 命中 + 保底 → `pity_hit`
5. 抽卡池 → >0 抽 + 命中 + 非保底 → `early_hit`
6. 抽卡池 → >0 抽 + 未命中 → `miss`

**完备性验证**：以上 6 条覆盖全部（池类型 × 抽卡次数 × 命中 × 保底）的组合空间，无遗漏。

### 1.2 池成败的独立判定

每个池的成功/失败通过池级 GDR 阈值独立判定。假设：

1. **池间独立**：一个池的成败不影响另一个池的判定（在 B 维度）
2. **阈值一致性**：所有池使用相同的 GDR 指标（key）和阈值
3. **判定函数**：`SuccessChecker.is_success(pool_compact)` —— 将单池数据包装为伪 compact 并传入统一的 `SuccessChecker`

如果池之间存在真实的相关性（如资源池提供金币、抽卡池消费金币），这一「独立判定」假设不反映因果依赖——但交叉统计（AB/BA）正是为了揭示这种依赖模式。

### 1.3 交叉统计的合理性

四种交叉统计（AA/BB/AB/BA）来自 2×2 列联表分析的标准框架：

- **AA（事件→事件）**：联合分布 `P(E₁=e₁, E₂=e₂, ...)`，回答「哪些事件组合最常见？」
- **BB（成败→成败）**：联合分布 `P(S₁=s₁, S₂=s₂, ...)`，回答「哪些池的成败联动？」
- **AB（事件→成败）**：条件分布 `P(Sⱼ=sⱼ | Eᵢ=eᵢ)`，回答「给定某池的事件类型，该池成功的概率？」
- **BA（成败→事件）**：条件分布 `P(Eᵢ=eᵢ | Sⱼ=sⱼ)`，回答「给定某池的成败，事件分布如何？」

理论依据：列联表分析（Pearson, 1904），在此维度灾难通过事件组合模式的离散化缓解。

### 1.4 Laplace 平滑的无信息先验

在条件概率估计中，对零计数单元格添加 Laplace 平滑（+1 伪计数）：

```
P̂(X=x | Y=y) = (count(x, y) + 1) / (count(y) + K)
```

其中 K=事件类型数量。此平滑等价于 Dirichlet(1, 1, ..., 1) 先验下的贝叶斯后验均值（Jaynes, 2003, *Probability Theory* §18.4.2）。

### 1.5 SVG 两层事件分类体系（未来细化方向）

理论参考.svg 建立了一套比当前 7 种事件更精细的**两层事件分类体系**。两层相对独立但存在嵌套关系——单抽层的稀有卡事件嵌套在池子层的决策过程中。当前 7 种事件是此体系的简化实现，两层体系为事件类型未来的细化方向。

#### 第一层：单抽层 —— 出了什么卡？（稀有卡事件分类）

入口为「（某一抽）出了稀有卡」→「出了什么卡」→ 四条路径，每条经「是否触发保底」（是/否）分叉，共 **8 种标签**：

```
（某一抽）出了稀有卡
  └─ 出了什么卡？
      ├─ "当期up"目标卡 ─── 是否触发保底？
      │   ├─ 是 → 保底[i]出
      │   └─ 否 → 提前出
      │
      ├─ 非"当期up"目标卡 ─── 是否触发保底？
      │   ├─ 是 → 保底[i]歪得目标卡
      │   └─ 否 → 提前歪得目标卡
      │
      ├─ "当期up"非目标卡 ─── 是否触发保底？
      │   │   （注：需要有一个以上up才存在此分支）
      │   ├─ 是 → 保底[i]歪非目标up
      │   └─ 否 → 提前出非目标up
      │
      └─ 普通稀有卡非目标卡 ─── 是否触发保底？
          ├─ 是 → 保底歪常驻
          └─ 否 → 提前出常驻
```

8 种标签的互斥完备性由决策树保证（每条路径终点唯一）。标签中的「目标卡」参照用户设定的**整体**目标集合，而非仅限当前池子的 up——因此单抽层也适用于 up 不含目标卡的池子（只是「当期 up 目标卡」路径永远不会触发）。

#### 第二层：池子层 —— 这个池子的整体结果？

入口为「该池子"当期 up"是否包含未抽出的目标卡」（是/否）→ 四个主分支均经「是否投入非免费抽」↔「是否抽出当期 up」两层判断，共 **12 种终态**：

```
该池子"当期up"是否包含未抽出的目标卡？
│
├─【是：up 包含未抽出的目标卡】
│  │
│  ├─ 是否投入非免费抽？──【是】
│  │  └─ 是否抽出"当期up"的目标卡？
│  │     ├─【是】→ 全部or部分？
│  │     │   ├─ 部分 → 出部分卡
│  │     │   └─ 全部 → 出全部卡
│  │     └─【否】→ 抽了没出
│  │
│  └─ 是否投入非免费抽？──【否】
│     └─ 是否抽出"当期up"的目标卡？
│        ├─【是】→ 全部or部分？
│        │   ├─ 部分 → 免费抽出部分卡
│        │   └─ 全部 → 免费抽出全部卡
│        └─【否】→ 跳过
│
└─【否：up 不包含未抽出的目标卡】
   │
   ├─ 是否投入非免费抽？──【是】
   │  └─ 是否抽出"当期up"？
   │     ├─【是】→ 全部or部分？
   │     │   ├─ 部分 → 计划外抽卡出部分up
   │     │   └─ 全部 → 计划外抽卡出全部up
   │     └─【否】→ 浪费资源
   │
   └─ 是否投入非免费抽？──【否】
      └─ 是否抽出"当期up"？
         ├─【是】→ 全部or部分？
         │   ├─ 部分 → 计划外免费抽出部分up
         │   └─ 全部 → 计划外免费抽出全部up
         └─【否】→ 忽略
```

12 种终态：**出全部卡、出部分卡、免费抽出全部卡、免费抽出部分卡、计划外抽卡出全部 up、计划外抽卡出部分 up、浪费资源、计划外免费抽出全部 up、计划外免费抽出部分 up、跳过、抽了没出、忽略**。

> **未决问题**：当一个池子出两个及以上 SSR 时，单抽层的多次稀有卡事件如何汇总为池子层的一个终态标签，需要进一步设计。SVG 中两层之间的连线关系表明它们不应简单独立——池子层某种意义上包含单抽层，但确切的关系尚未理清。

#### 当前代码如何映射到两层体系

| 代码事件 | 涵盖的 SVG 标签（单抽层 → 池子层） |
|----------|----------------------------------|
| `pity_hit` | 保底[i]出 + 提前歪得目标卡 → 出全部/部分卡 |
| `early_hit` | 提前出 + 提前歪得目标卡 → 出全部/部分卡 |
| `miss` | 保底歪非目标 up / 保底歪常驻 / 提前出非目标 up / 提前出常驻 → 抽了没出 |
| `skip` | （无 SSR 产出）→ 跳过 |
| `ignore` | （无 SSR 产出）→ 忽略 / 计划外免费抽出 up（合并） |

**代码合并、SVG 区分的关键维度**：

| 维度 | 代码 | SVG | 影响 |
|------|------|-----|------|
| 保底出 vs 歪得 | 合并为 pity_hit/early_hit | 区分 4 种（出/歪得 × 保底/非保底） | 无法区分「精准命中 up」与「歪出目标卡」的运气质量 |
| up 非目标 vs 常驻 | 合并为 miss | 区分歪非目标 up / 歪常驻 | 无法区分「歪了但仍是 up」与「歪了常驻」 |
| 免费 vs 非免费投入 | 合并为 skip/ignore | 区分免费出/跳过/忽略 | 无法识别「仅靠免费资源即达成目标」的高效情形 |
| 计划内 vs 计划外 | 合并为 ignore | 区分 4 种计划外情形 | 无法区分「主动放弃的池子」与「意外收获的池子」 |
| 超出需求 | 未追踪 | 浪费资源 | 无法检测「超额抽取」的资源低效行为 |

#### 两层交叉的理论可能性

单抽层 8 种 × 池子层 12 种 → 理论上 **96 种组合**。实际可通过维度规约降维：
- **单抽层规约**：保底/非保底（2 值）× 是否歪（2 值）× 卡类型（目标/非目标 up/常驻，3 值）→ 12 种，而非 8 种（因为「当期 up 目标卡」路径下「歪」分支永不触发）
- **池子层规约**：up 含目标（2 值）× 是否投入（2 值）× 是否抽出（2 值）× 全部/部分（2 值）→ 16 种理论组合，减去 4 种不可能组合（不投入且不抽出时不适用全部/部分）→ 12 种
- **两层交叉的稀疏性**：许多组合在实际中几乎不可能（如「保底歪常驻」+「免费抽出全部卡」），有效组合远少于 96

> **⚠ 注意**：两层交叉分析（C10）是架构设计中的条目，当前尚未在任何计划文件中展开。SVG 原作者也未给出两层之间确切的汇总规则——这是一个开放的设计问题，而非简单实现任务。

## 二、已知局限性

### 2.1 维度灾难

随着池子数量增加，事件组合空间指数增长（K 个池 × 7 种事件类型 → 理论上有 7^K 种组合）。实际通过 5 种事件组合模式（sequence/set/count_set/custom）降维，但对 5 个以上池的场景，即使 `count_set` 模式的组合数也可能超过有效样本量。

### 2.2 low_sample 阈值的校准

`low_sample` 标记的阈值（默认 30）基于经验法则（n>30 近似正态），但对：
- 高度偏斜的概率（p<0.1 或 p>0.9）：需要更大样本量
- Wilson 区间的小样本覆盖率：虽然优于 Wald，但在 n<10 时仍有不足

### 2.3 事件类型的语义合并与信息损失

当前 7 种事件类型是对 SVG 两层体系（单抽层 8 种 × 池子层 12 种）的**大幅简化**。以下分析每个合并点的信息损失：

#### 2.3.1 「miss」的歧义

「miss」表示「抽了但未获得目标卡」——但实际原因可能包括：
- 保底概率不够高（软保底未生效或生效但未触发）
- 运气差（即使在硬保底计数器顶部，也可能因分布权重未命中目标卡）
- 策略选择（在特定池只抽有限次数）
- **出卡了但非目标**：出了 SSR，但属于「歪非目标 up」或「歪常驻」，当前全部归入 miss

按 SVG 单抽层，miss 实际合并了 4 种不同的稀有卡事件（保底歪非目标 up / 保底歪常驻 / 提前出非目标 up / 提前出常驻），丢失了「歪了什么」的信息。这限制了事件→成败分析的因果解释力——用户无法区分「纯运气差（一个 SSR 没出）」和「出了 SSR 但歪了」。

#### 2.3.2 pity_hit / early_hit 的合并

`pity_hit` 和 `early_hit` 将「保底出目标卡」与「保底歪得目标卡」、「提前出目标卡」与「提前歪得目标卡」合并。在以下场景中损失关键信息：
- **精准命中 up 目标**：保底出（在 up 池中通过保底恰好命中目标）
- **歪出目标卡**：在非 up 池中意外获得目标（保底歪得/提前歪得）

对用户而言，这两个场景的「运气质量」截然不同，但当前统计中无法区分。

#### 2.3.3 skip / ignore 的合并

`skip`（有目标卡但选择不投入）和 `ignore`（无目标卡自然不投入）在 SVG 池子层中进一步细分：
- **免费抽出**：仅靠免费资源即达成目标（高价值场景）
- **计划外抽出**：在 up 不含目标的池子中意外抽出 up（意外收获）
- **计划外免费抽出**：在无目标池中免费抽出 up（双重惊喜）
- **浪费资源**：超出需求后继续抽（资源低效）

当前 `ignore` 将计划外免费抽出 up 与「根本没出 SSR」混为一谈，丢失了正面意外收获的信号。

#### 2.3.4 缺失的维度

SVG 明确区分但代码完全未实现的维度：

| 维度 | SVG 标签 | 当前状态 | 细化方向 |
|------|---------|---------|---------|
| **保底/非保底** | 8 种单抽层标签均有保底/非保底分支 | 仅 pity_hit 区分保底名（raw 模式），early_hit/miss 不区分 | 为所有事件类型增加 `pity_triggered: bool` 子字段 |
| **计划内/外** | 池子层区分「up 含目标」vs「up 不含目标」 | skip/ignore 用 pool_target_map 近似区分 | 增加「计划外收获」事件类型 |
| **免费/非免费** | 池子层每个分支均区分 | 未追踪投入资源是否为免费 | 增加 `free_only: bool` 子字段 |
| **全部/部分** | 池子层每个「抽出」分支均区分 | 未区分 | 增加 `all_targets_in_pool: bool` 子字段 |
| **超出需求** | 浪费资源（SVG 中标注在「是否超出需求」节点） | 未追踪 | 增加 `overdraw: bool` 子字段 |

### 2.4 两层交叉的设计挑战

单抽层（每次 SSR 的事件）× 池子层（每池的终态）的 96 种理论组合中，有效组合远少于此数。但需解决一个**开放性设计问题**：

> 「当一个池子出两个及以上 SSR 时，单抽层的多次稀有卡事件如何汇总为池子层的一个终态标签？」

可能的汇总策略（均未实施）：
1. **保守汇总**：取「最差」的单抽事件作为池子标签（如 miss 覆盖 early_hit）
2. **主导事件**：取第一次命中目标卡的 SSR 事件
3. **多标签**：允许一个池子携带多个单抽标签（破坏「每池唯一事件」的互斥假设）
4. **延迟决策**：在交叉统计阶段而非事件推断阶段处理多 SSR 汇总

这是一个需要进一步理论探讨的设计问题，而非简单实现任务。

### 2.5 转移矩阵分析（C9）的缺失

SVG 中标注的「转变分析」——每个池子后成功/失败的状态转移矩阵——当前在 `process_analysis.py` 中完全未实现（`analysis_panel.py` 仅预留了 `transition_analysis` 键名）。转移矩阵能回答：「在哪个阶段最容易从『成功轨道』跌入『失败轨道』？」以及「给定前 N 个池的成败序列，后续池的成败概率如何？」——这些是当前 AA/BB/AB/BA 四种交叉统计无法直接回答的问题。
