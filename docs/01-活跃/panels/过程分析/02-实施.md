## 一、代码位置

### 事件推断：`gacha_simulator/core/process_trace.py`

**行数**：~230 行

**核心数据结构**：

```python
@dataclass
class PoolEvent:
    pool_id: str
    pool_type: str           # 'draw' / 'exchange' / 'resource'
    event_type: str           # 7 种事件之一
    pity_name: Optional[str]  # 保底名称（仅 pity_hit）
    draws: int                # 该池抽卡次数
    counter_max: int          # 最大保底计数器

@dataclass
class SampleTrace:
    events: List[PoolEvent]
    pool_success: Dict[str, bool]    # 各池成功/失败
    is_success: bool                  # 总体成功/失败
    gdr_value: float                  # 总体 GDR 值
    pool_gdr_values: Dict[str, float] # 各池 GDR 值
```

**核心函数**：

| 函数 | 用途 |
|------|------|
| `infer_events(compact, target_ids, pool_types, pool_target_map)` | 主入口——从 CompactResult 推断每池事件 |
| `_infer_from_draw_sequence(...)` | 从逐抽序列推断（优先级路径） |
| `_infer_from_aggregate(...)` | 从聚合数据推断（降级路径） |
| `_resolve_skip_ignore(pool_id, pool_card_counts, target_ids, pool_target_map)` | 判定 0 抽池为 skip 还是 ignore |

## 二、事件组合模式实现

### 5 种模式

| 模式 | 函数 | 输出类型 | 说明 |
|------|------|---------|------|
| raw | 直接使用 `event_type` str | `str` | 保底按 `pity_hit:{name}` 细化 |
| sequence | `to_event_type_sequence()` | `Tuple[str, ...]` | 按池顺序排列事件类型 |
| set | `to_event_type_set()` | `Tuple[str, ...]` | 无序去重集合（排序） |
| count_set | `to_event_count_set()` | `Tuple[Tuple[str, int], ...]` | 事件类型+出现次数 |
| custom | `to_custom_pattern(events, constraints)` | `Dict[str, str]` | 用户自定义约束分桶 |

### 事件类型排序

`get_event_type_order()` 定义事件在图表中的固定显示顺序：
```
early_hit → miss → skip → ignore → exchange → no_exchange → pity_hit:xxx (排序)
```

## 三、交叉统计的实现

### AA：事件模式分布

```python
patterns = [to_xxx_pattern(sample.events) for sample in samples]
counts = Counter(patterns)
freq = {k: v / total for k, v in counts.items()}
# 输出：模式 → 频率的字典，按频率降序排列
```

### BB：成败模式分布

类似 AA，但粒度从「事件类型」变为「每池成功/失败」的布尔值元组。

### AB：事件→成败条件概率

```python
# 给定事件类型 E=e，池 P 成功的条件概率
count_e = sum(1 for s in samples if s.events[p].event_type == e)
success_e = sum(1 for s in samples if ... and s.pool_success[p])
p = (success_e + 1) / (count_e + K)  # Laplace 平滑
ci = wilson_ci(success_e, count_e)    # Wilson CI
```

### BA：成败→事件条件概率

类似 AB，但条件方向反转：`P(E=e | S=s)`。

## 四、low_sample 标记

当交叉统计中某组的样本量 < `low_sample_threshold`（默认 30）时：
1. 图表中该组的柱状图/单元格用斜线填充或淡化颜色
2. tooltip 中标注「低样本 (n=X)」警告
3. 条件概率值保留但附带可信度警告

## 五、当前实现与 SVG 设计的差距

理论参考.svg 定义了更精细的两层事件分类体系（详见 `01-理论.md` §1.5），当前代码实现与之存在以下差距：

| 差距项 | 代码现状 | SVG 设计 | 影响 |
|--------|---------|---------|------|
| 单抽层细化 | `pity_hit`/`early_hit` 合并保底出与歪得 4 种标签 | 区分保底[i]出/提前出/保底歪得/提前歪得 | `process_trace.py` 的 `_infer_from_draw_sequence()` 已逐抽遍历，具备所需的原子数据（每抽的 pool_id/card_id/pity_triggered），细化需要的是分类逻辑而非新数据 |
| 歪非目标/歪常驻区分 | `miss` 合并了 4 种非目标 SSR 事件 | 区分歪非目标 up / 歪常驻 × 保底/非保底 | 需要目标卡与 up 卡的交集比对——`target_ids` 和 `pool.cards` 均已可用 |
| 免费/非免费投入 | 未追踪 | 池子层每个分支均区分 | 需要追踪「免费抽卡次数」概念——当前 `CompactResult` 不包含此字段 |
| 计划内/外抽卡 | `skip`/`ignore` 用 pool_target_map 做「有/无目标」二分 | 区分 4 种计划外情形 + 免费/非免费组合 | pool_target_map 已覆盖「是否含目标」的判定，计划外的细分需要额外维度 |
| 全部/部分区分 | 未区分 | 池子层每个「抽出」分支均区分全部/部分 | `target_ids` 和 `card_counts` 已可判定「池中目标卡是否全部获得」 |
| 超出需求（浪费资源） | 未追踪 | SVG 标注在「是否超出需求」节点 | 需要定义「需求量」——目标卡的要求数量可来自 `ConfigStore.targets` |
| 转移矩阵（C9） | `analysis_panel.py` 仅预留 `transition_analysis` 键名，`process_analysis.py` 无实现 | SVG 标注「转变分析」——每池后成功/失败的状态转移 | 需要新增 `compute_transition_matrix()` 函数 |

> 以上差距并非全部需要立即解决——部分细化方向（如免费/非免费投入、浪费资源）在对应配置场景（免费抽卡机制、需求数量约束）尚未就绪时缺乏实际分析价值。优先级见 `04-问题.md`。
