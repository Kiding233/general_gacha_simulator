# 配置面板 -- 理论

> 创建日期：2026-06-02（首次建档）
> 关联模块：`core/config_store.py`、`core/config_io.py`

---

## 一、核心假设

### 1.1 ConfigStore 作为单一事实来源

所有面板通过 `ConfigStore` 获取配置信息，而非直接从文本文件或全局变量读取。这一设计假设：

1. **内存一致性**：`ConfigStore` 是完整配置的唯一 in-memory 表示。面板不应绕过它直接读文件。
2. **值语义**：`set_store()` 传入的是同一个对象引用，编辑原地发生。如果多个面板同时持有同一个 `ConfigStore` 引用并修改，将产生竞态。当前设计依赖 Qt 单线程事件循环保证安全。
3. **完整闭包**：`ConfigStore` 包含所有可配置项，无隐式全局默认值（除少数历史遗留，如 `weapon_character_map` 无配置入口）。

### 1.2 竖线分隔文本格式的语义约束

配置文件使用 `|` 作为字段分隔符。每行第一列为键或 ID，后续列为参数。格式假设：

1. **顺序无关**：除 `schedule.txt`（排期表按行顺序表示时间先后）外，其他文件的条目顺序无语义意义。
2. **可选列**：通过列位置推断含义（如 `schedule.txt` 第 4 列可空、第 5 列可选、第 6 列为 `target:` 前缀的目标卡列表）。向后兼容依赖「新列只能追加在末尾」的约定。
3. **换行即分隔**：每条记录占一行。不支持多行值字段（字段内不允许换行符）。
4. **编码假设**：文件读写作 UTF-8，无 BOM。`parse_schedule_file()` 等函数不显式处理编码参数，依赖 Python 默认值。

### 1.3 ConfigStore 与 SimulationEnv 的转换

`SimulationEnvBuilder.from_config_store()` 是配置层与模拟层之间的唯一桥梁。它负责：

- 将 `PoolEntry` 列表转换为 `Pool` 对象（解析分布、创建卡池实例）
- 将 `PityConfig` 转换为 `PityEngine`（创建保底规则树）
- 将 `GainRule` 列表转换为 `ResourceGain` 对象（合并为时间线函数）
- 提取 `target_ids`、`ssr_ids` 等辅助集合

此转换是单向的：`SimulationEnv` 不包含回写 `ConfigStore` 的路径。仿真过程中修改配置（如自适应模拟调整参数）需在更高层协调。

## 二、配置项维度

### 2.1 池子维度

| 属性 | 说明 | 配置来源 |
|------|------|---------|
| `pool_id` | 唯一标识（如 `pool_character_01`） | `schedule.txt` 第 1 列 |
| `pool_type` | 池类型：角色/武器/兑换/复刻 | 自动推断或 `schedule.txt` 类型列 |
| `start_day` / `end_day` | 开放时段 | `schedule.txt` |
| `cost` | 单抽消耗（如 `draw_resource:160`） | `schedule.txt` |
| `distribution` | 卡掉落分布（含概率和额外资源奖励） | `pools/{分布名}.txt` |
| `bindings` | 绑定的保底/兑换参数 | `schedule.txt` 可选参数列 |
| `target_specs` | 此池可掉落的目标卡 | `schedule.txt` `target:` 列 |

### 2.2 保底维度

`PityConfig` 含一个 `PityDef` 列表，每个 `PityDef` 定义：
- 保底类型（`soft`/`hard`）
- 生效参数（`start_at`、`end_at`、`step_at`、`base_prob` 等）
- 目标分布（触发保底时按权重分配 SSR）
- 重置条件（`any_ssr`/`featured`/`never`）
- 适用的池列表（`*` 表示所有池）

### 2.3 目标卡与权重

- **目标卡（`TargetCardEntry`）**：`card_id:quantity` 格式，定义玩家想要抽到的卡及期望数量
- **渴望度权重（`desire`）**：每张目标卡的渴望程度（0-10），用于 `weighted_satisfaction` GDR
- **未命中成本（`miss_cost`）**：未获得目标卡的损失系数（0-10），用于 `weighted_satisfaction` GDR
- **卡牌价值（`card_value`）**：任意卡（含非目标卡）的绝对价值权重，用于 `total_card_value` GDR

## 三、已知局限性

### 3.1 竖线分隔格式的表达力限制

1. **无嵌套**：分布文件中的「额外资源奖励」字段需通过 `_bonus_to_text()` 序列化为紧凑文本格式（`ft:res:10,res2:5 & nth:3=res:20 & xs:5>res:2`），解析逻辑复杂且易出错
2. **无类型注释**：解析器通过列位置推断字段类型，无法区分「空值」与「默认值」
3. **无 schema 校验**：配置加载时不做 schema 校验，非法值（如负数概率）可能在运行时抛异常

### 3.2 weapon_character_map 无配置入口（已知问题）

`GDRContext.weapon_character_map` 在配置系统中无填写入口，始终为空 `dict`。属于 CLAUDE.md 已知问题 #1。

### 3.3 ConfigStore 与 UI 的紧密耦合

`ConfigPanel` 在 `set_store()` 后直接修改传入的 `store` 对象。虽然这避免了 copy-on-write 的开销，但也意味着：
- 「取消编辑」不可行——所有修改即时生效
- 多个面板持有同一 `ConfigStore` 引用时，一个面板的修改可能影响另一个面板
