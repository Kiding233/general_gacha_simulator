## 一、背景与现状

### 1.1 当前架构（审查后修正）

资源获取有**两条独立路径**，存在代码重复和实现不一致：

```
gains.txt
   │
   ├── 路径A（死代码）：parse_gains_file()          ← 从未被生产调用！
   │    └── 4 个 helper(_weekday_of_day 等)         ← 使用 fromordinal，逻辑正确
   │
   └── 路径B（生产代码）：_load_gains() → ConfigStore
            └── SimulationEnvBuilder._build_resource_gain()  ← 独立重写了展开逻辑
                 ├── line 722,749: fromordinal(day + 735000) ← 硬编码 epoch
                 ├── line 731: range(0, total_days, 30)      ← monthly_day 用固定30天！
                 └── line 730: fallback=1 而非 rule.param    ← 不读 param 字段
```

### 1.2 审查发现的缺陷

| 缺陷 | 位置 | 严重度 |
|------|------|--------|
| `parse_gains_file()` 是死代码，零调用点 | `resource_gain.py:137` | P1 |
| `_build_resource_gain` 的 `monthly_day` 用固定 30 天而非真实日历 | `batch_simulator.py:731` | P1 |
| `_build_resource_gain` 的 `monthly_day: month,day` 逗号格式不支持（死代码已支持，生产代码只处理单数字 → `int()` 直接报错） | `batch_simulator.py:730` | P1 |
| `_build_resource_gain` 与 `parse_gains_file` 的 `monthly_week` 分隔符不一致（生产用 `-`，死代码用 `,`，`gains.txt` 注释用 `,`） | `batch_simulator.py:738` vs `resource_gain.py:225` | P1 |
| `_build_resource_gain` 的 `monthly_day` fallback=1 不读 `param` | `batch_simulator.py:730` | P2 |
| `<!-- REVIEW-R1-FIX: ISSUE-001 -->` `_build_resource_gain` 的 `monthly_day`/`monthly_week` 分支使用 `startswith('monthly_day:')`/`startswith('monthly_week:')` 要求冒号前缀——GUI 通过 `_gui_gain_type_to_store()` 创建的无冒号 `rule_type='monthly_day'`/`'monthly_week'` 被静默跳过，且回退值为硬编码（`else 1` / `else '1-1'`）完全忽略 `rule.param` 字段 | `batch_simulator.py:729-757` | P1 |

| 两条路径各自维护日历展开逻辑 | 架构层面 | P2 |

### 1.3 为何不直接用 `parse_gains_file`

1. 它是死代码——从未在生产路径中被调用
2. 它从文件读取（`gains.txt`），绕过了 `ConfigStore`——用户在 GUI 中编辑后，日历预览会显示过期数据
3. 它产出的 `schedule` dict 结构虽然适合日历预览，但需要和 `_build_resource_gain` 保持行为一致

### Phase 1 — 抽取共享展开函数

**新增函数**（`resource_gain.py`）：

```python
def expand_gain_rules_to_schedule(
    gain_rules: List[GainRule],
    day_overrides: List[DayOverride],
    total_days: int,
    start_date: datetime.date | None = None,
) -> Dict[int, Dict[str, float]]:
```

**职责边界**：新函数**只负责**将规则列表展开为 `{day: {rid: amount}}` 字典——不涉及 `ResourceGainFunction` 包装（`ScheduleResourceGain`/`CompositeResourceGain`/`LinearResourceGain`）。包装逻辑仍由调用方 `_build_resource_gain()` 负责（当前 `batch_simulator.py:769-776`），日历预览直接消费字典。

- 合并 `_build_resource_gain` 的展开逻辑 + 4 个 helper 的正确日历算法
- `monthly_day` 改用 `fromordinal` 获取真实日期（修复 `batch_simulator.py:731` 的固定30天bug）
- `monthly_day` 支持 `month,day` 逗号格式——死代码 `parse_gains_file` 已实现，生产代码 `_build_resource_gain` 缺失（修复 P1）
- `monthly_week` 统一分隔符为逗号 `,`（死代码用 `,` 正确，生产代码用 `-` 错误。修复 P1）
- 若 `start_date=None`，回退 `datetime.date(2013, 6, 2)`（即 ordinal=735000）
- `day:` 覆盖采用**累加**语义（`get(rid, 0) + amount`），与生产代码 `_build_resource_gain` 行为一致。死代码 `parse_gains_file` 用的是覆盖语义（`=`），不采纳

**`total_days` 来源**：由调用方传入。模拟路径中由 `end_time / 86400 + 1` 计算（逻辑不变）；日历预览路径从池配置推导：`max((p['start_day'] + p['duration']) for p in config['pools']) if config['pools'] else 365`（空池 fallback 365 以提供合理的日历展示范围；现有 `_update_preview` 文本回退 0，日历预览的 365 是独立决策，后续考虑统一两者的 fallback 行为）<!-- REVIEW-R1-FIX: ISSUE-007 -->。

**`start_date` 的 str→date 转换**（P1 设计澄清）：

`ConfigStore.sim_start_date` 存储为 `str`（如 `'2013-06-02'`），而 `expand_gain_rules_to_schedule` 的参数声明为 `datetime.date | None`。转换职责归属：**调用方负责转换**——模拟路径 `_build_resource_gain()` 和日历预览各在自己入口处执行 `datetime.date.fromisoformat(store.sim_start_date)`，异常时回退默认值。这样做的好处：(1) 新函数保持纯数据类型签名，不依赖字符串解析；(2) 两端可以有各自的错误处理策略。

**防御性检查**（参数范围校验 + 空值防御）：

| 规则类型 | 参数范围校验 | 参数越界处理 | `param=''` 空值处理 |
|---------|-------------|-------------|-------------------|
| `every_n_days` | `n >= 1` | 回退 `n=1` + `logging.warning` | 回退 `n=1` + `logging.warning` |
| `weekly` | `1 <= weekday <= 7` | 回退 `weekday=1` + `logging.warning` | 回退 `weekday=1` + `logging.warning` |
| `monthly_day`（单数字） | `1 <= day <= 31` | 回退 `day=1` + `logging.warning` | 回退 `day=1` + `logging.warning` |
| `monthly_day`（月,日） | `1 <= month <= 12` 且 `1 <= day <= 31` | 回退并跳过该规则 + `logging.warning` | 回退并跳过该规则 + `logging.warning`（无法解析为 `m,d` 格式时同样处理） |
| `monthly_week` | `1 <= week <= 5` 且 `1 <= weekday <= 7` | 回退并跳过该规则 + `logging.warning` | 回退并跳过该规则 + `logging.warning`（无法解析为 `w,d` 格式时同样处理） |

> 注：`n=0` 防御仍需保留（避免 `range(0, total_days, 0)` 抛出 `ValueError`），其他校验/空值防御为新增。归一化前 `param` 字段始终为 dataclass 默认值 `'1'`，归一化后 `param` 独立承载参数——GUI 清空参数列、旧格式文件转换后等场景均可能产生空 `param`，需系统性防御。`<!-- REVIEW-R1-FIX: ISSUE-003 -->`

**`<!-- REVIEW-R2-FIX: AUDIT-BREAK-5 -->`类型转换与格式解析防御（`expand_gain_rules_to_schedule` 内部实现要求）**：

`GainRule.param` 为 `str` 类型，而日历计算需要 `int` 值。所有参数解析必须包裹 `try/except ValueError` 并遵循以下规则：

| 规则类型 | 解析方式 | 需要 `int()` 转换 | NULL BREAK 风险 |
|---------|---------|------------------|----------------|
| `every_n_days` | `n = int(param)` | 是 | `param=''` → `int('')` → `ValueError`。防御：`try: n = int(param) if param else 1` + `n<=0` 回退 |
| `weekly` | `weekday = int(param)` | 是 | `param=''` → `int('')` → `ValueError`。防御：`try: weekday = int(param) if param else 1` |
| `monthly_day`（单数字） | `day = int(param)` | 是 | `param=''` → 回退 `day=1`。需先判断是否含逗号——若 `',' in param` 走月,日分支，否则走单数字分支。**不可仅凭 `split(',')`**：`'15'.split(',')` 产生 `['15']`，`len==1` 不能区分单数字与格式错误 |
| `monthly_day`（月,日） | `m, d = map(int, param.split(','))` | 是 | `param=''` → `['']` → `int('')` → `ValueError`。防御：先检查 `',' in param`，split 后验证 `len==2` |
| `monthly_week` | `week, wday = map(int, param.split(','))` | 是 | `param=''` → `int('')` → `ValueError`。**旧格式兼容**：归一化前 `monthly_week` 分隔符为 `-`，需同时防御 `param.split('-')` 旧格式输入——检测逻辑：先尝试 `split(',')`，若 `len!=2` 则尝试 `split('-')`，两者都失败则跳过该规则 + `logging.warning` |

**关键实现约束**：
- 所有 `int(param)` 调用必须用 `try/except ValueError` 包裹——归一化后 `param` 可能为空字符串或非数字字符串（用户 GUI 输入错误/旧文件格式残留）
- `param` 空字符串（`''`）的优先级高于数值校验：先检查空值 → 回退默认值 → 再检查数值范围
- `monthly_day` 的分支判断顺序：先 `',' in param` 走月,日逻辑 → 否则走单数字逻辑
- `monthly_week` 的旧格式兼容：尝试 `','` 分隔 → 失败则尝试 `'-'` → 都失败则跳过规则，不抛异常
- 所有回退行为均需 `logging.warning` 记录参数值、规则类型和回退后的默认值，便于排查数据问题
- 这些防御独立于上表中「参数范围校验」列——先完成类型转换和格式解析，再对转换后的数值进行范围校验`<!-- REVIEW-R2-FIX: AUDIT-BREAK-5 END -->`

**前置归一化 — `rule_type` 格式统一**（2026-06-11 审查补充）：

当前 `GainRule` 的 `rule_type` 字段在三条路径中格式不一致，`expand_gain_rules_to_schedule` 若同时面对所有格式，解析逻辑将脆弱且难以测试：

| 来源 | `rule_type` 示例 | `param` 示例 | 问题 |
|------|-----------------|-------------|------|
| `_load_gains()`（文件加载） | `every_n_days: 1` | `"1"`（dataclass 默认值，未被填充） | 参数嵌入 `rule_type`，`param` 未用 |
| `_gui_gain_type_to_store()` — "每天" | `every_n_days:1` | `""`（来自 GUI param） | 只有「每天」嵌入参数 |
| `_gui_gain_type_to_store()` — 其他类型 | `monthly_day` | `"15"` | `rule_type` 不含参数，`param` 独立 |

建议在 Phase 1 实施前先归一化：
- **`_load_gains()`**（`config_io.py:230`）：创建 `GainRule` 时拆分 `rule_type`——提取 `:` 后的参数填入 `param`，`rule_type` 只保留类型名

归一化后 `param` 始终包含参数值，不再需要从 `rule_type` 字符串中提取。`expand_gain_rules_to_schedule` 直接读取 `param` 即可。若 `param` 为空字符串，对 `every_n_days` / `weekly` / `monthly_day` 回退默认值 `1`。

> `<!-- REVIEW-R1-FIX: ISSUE-004 -->` **缓存 key 序列化实施细节**：
> - `GainRule` 序列化为 `(rule_type, param, tuple(sorted((k, round(v, 6)) for k, v in gains.items())))`。其中 `gains` 的值是 `float`（如 `60.0`），取 6 位小数 `round(v, 6)` 消除浮点精度漂移（`60.00000000000001` 与 `60.0` 的 `repr` 不同可能导致语义相同规则产生不同缓存 key）。资源金额通常 < 10^6，6 位小数精度足够且可逆。

> - `sorted(gains.items())`（键值对排序）而非 `sorted(gains)`（仅键列表）——前者才能捕捉值的差异，后者丢弃值信息导致不同收益的规则产生相同缓存 key。
