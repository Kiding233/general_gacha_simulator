# 最差后期影响完整流程调查报告

**状态：仅报告，不涉及编码。**

以下为对 `WorstImpactPanel` → `core/worst_impact.py` 完整数据流的调查记录，供后续参考。

## 概述

评估在批量模拟的条件筛选结果中，最差资源水平下能连续拿下多少个新池子限定卡。

## 完整数据流（8阶段）

```
阶段1: 输入
  GachaPanel完成模拟 → MainWindow.on_simulation_finished()
  → WorstImpactPanel.set_simulation_results(aggregate_data, target_specs)
  输入: List[Dict] (extract_aggregate输出) + Dict[str,int] (目标卡)

阶段2: 用户配置 (worst_impact_panel.py:209-269)
  ┌─ 条件选择：全部/成功/失败 (QRadioButton, 默认success)
  ├─ GDR指标：QComboBox (populate_gdr_combo填充, 默认索引1)
  ├─ 成功阈值：QDoubleSpinBox (默认1.0, 范围±9999999)
  ├─ α分位数：QDoubleSpinBox (默认0.05)
  ├─ 模拟次数：QSpinBox (默认1000, 范围50-100000)
  ├─ 池持续时间：QSpinBox (默认21天, 范围1-365)
  ├─ 单抽消耗：QLineEdit (默认"draw_resource:160")
  └─ 卡牌分布：PoolDistributionDialog (编辑99个新池的奖励分布)

阶段3: analyze() 核心流程 (core/worst_impact.py:136-191)
  3a. _prepare_pool_info() (line 138):
      - 遍历原始池，提取 rewards、probabilities、featured_ssr_ids
      - 从 store.card_defs 收集所有SSR卡ID (line 127-134)
      - 确定池类型(角色/武器)、单抽消耗、每日资源收入
  3b. _build_success_checker() (line 140-143):
      - 使用 GDR key + threshold + desire/miss_cost/card_value 权重
      - 构建 SuccessChecker，用于判断原始模拟是否"成功"
  3c. ConditionalResourceDistribution (line 145-147):
      - 遍历所有模拟结果 (line 26-39):
        for r in simulation_results:
            if success_checker(r): success_resources.append(r['final_resources'][resource])
            else:                  failure_resources.append(r['final_resources'][resource])
      - 提供 get_conditional_distribution(condition) 返回 EmpiricalDistribution
  3d. get_worst_case_resource(condition, alpha) (line 149):
      - 调用 EmpiricalDistribution.quantile(alpha)
      - 插值公式 (distribution.py:96-109):
        idx = p * (n-1), lo = floor(idx), hi = ceil(idx)
        若 lo==hi → sorted[lo]
        否则 → sorted[lo]*(1-frac) + sorted[hi]*frac  (线性插值, 等价于R type=7)
      - alpha=0.05 → 第5百分位, 即最差5%情况中的资源上限
  3e. _compute_pity_coverage() (line 150):
      - pity_coverage = worst_resource / (pity_threshold × cost_per_draw)
      - 含义：最差资源能覆盖多少倍"大保底"所需抽数
  3f. prepare_simulation_config(worst_resource) (line 193-293):
      - 详细见阶段4
  3g. run_batch_parallel() (line 171-183):
      - max_workers=1 (强制串行), strategy_name='smart'
      - 串行路径 (batch_simulator.py:274-298):
        _wk_init() 在主进程注入全局变量 → for循环 _wk_run_single()
      - 并行路径 (batch_simulator.py:300-325):
        multiprocessing.Pool(initializer=_wk_init) → imap_unordered()
        分块大小 = max(1, n // (workers*4))
  3h. analyze_batch_results() (line 295-317):
      - 详细见阶段4末尾

阶段4: 新池子生成 (prepare_simulation_config, line 193-293)
  MAX_POOLS = 99
  对 i in 0..98:
    池ID:       f'_worst_impact_pool_{i}'
    Featured卡: f'_wi_featured_{i}'
    奖励复制:   原始池 rewards → 替换 Featured SSR → 保留非Featured卡
    时间安排:   start = i * pool_duration, end = (i+1) * pool_duration
    池目标:     {f'_wi_featured_{i}': 1}
    卡定义:     为每个新Featured卡和原始Standard SSR注册card_def

  _build_pity_engine() (line 447-508):
    - 从 store.pity.pities 读取保底定义
    - 创建 SoftPityBehavior 或 HardPityBehavior
    - 对每个虚拟池:
        fnmatch 匹配 pools 通配符模式
        _resolve_targets_for_pool() :
          'limited_ssr' → 当前池的 _wi_featured_{i}
          'standard_ssr' → 原始非Featured SSR集合
          'ssr' → featured ∪ standard
    - 构建 PoolPitySpec → PityEngine

  停止条件 (line 266-271):
    ConsecutivePoolTargetCondition(
        pool_schedules: [(pool_id, start, end), ...] 按end_time升序排列,
        pool_targets:    {pool_id: featured_card_id},
        resource_name:   'draw_resource',
        end_time:        99 * pool_duration,
    )
    三个OR条件 (stop_condition.py:120-133):
    ① 资源耗尽: resources[resource_name] <= 0
    ② 时间耗尽: real_time >= end_time
    ③ 连续中断: 遍历已结束池子, 任一目标卡获得数<1 → 立即停止
       关键: pool_schedules 按 end_time 升序, 遇到第一个 real_time < end 的池子就 break
       "连续"语义: 从第0个池子开始, 每个都必须成功, 第k个失败则k+1...根本没有机会被抽

  初始资源: {'draw_resource': worst_resource}

阶段5: 策略
  analyze() 使用 strategy_name='smart' (SmartStrategy)
  SmartStrategy 通过 _pool_needs_target() 判断每个池是否需要追卡

  DrawTargetStrategy 已注册但未被 analyze() 使用:
    - select_action(): 遍历当前池, 找到 pool_id 匹配的 → DrawAction, 否则等待
    - 设计为"只从指定池抽卡", 不如 SmartStrategy 灵活
    - 可能是预留扩展或遗留代码

阶段6: analyze_batch_results() (line 295-317)
  对每条模拟结果:
    consecutive = 0
    for i in 0..98:
      if card_counts.get(f'_wi_featured_{i}', 0) >= 1:
        consecutive += 1
      else:
        break  ← 严格连续: 第一个失败就停止计数
    未返回的结果(异常)视为 consecutive=0

  统计:
    distribution[k] = count / n
    expected = Σ k * prob

阶段7: 结果展示 (worst_impact_panel.py)
  ┌─ PityCoverageGauge (line 86-107):
  │    coverage = worst_resource / pity_cost
  │    三档颜色: ≥1.0绿色(#27ae60) / ≥0.5橙色(#f39c12) / <0.5红色(#e74c3c)
  │    CoverageBar.paintEvent() (line 29-56):
  │      填充条宽度 = (w-2) * min(pct, 1.0)
  │      红色标记线位置 = 1 + (w-2) * (1.0 / max_display)
  │
  ├─ detail_table (line 433-445):
  │    5列: k | P(X=k) | P(X>=k) | 累计概率 | 说明
  │    get_p_ge(k) = Σ prob[j] for j>=k
  │    get_max_consecutive_at_threshold(t): max{k | P(X>=k) >= t}
  │
  └─ _plot_distribution() (line 450-490):
       ax.bar(ks, probs, color='coral')
       ax.axvline(expected_val, 红色虚线)
       柱上标注百分比(仅概率>1%的柱)
       临时PNG → QPixmap → scaled → QLabel

阶段8: 数据依赖链
  ConfigStore
    ├─→ store.pools → _prepare_pool_info() → rewards, featured_ids, standard_ssr_ids
    ├─→ store.card_defs → SSR ID 收集
    ├─→ store.pity.pities → _build_pity_engine()
    └─→ store.resource_defs → resource_name 解析

  模拟结果 (List[Dict])
    └─→ extract_aggregate() → final_resources, card_counts, total_consumed, ...
        └─→ ConditionalResourceDistribution → EmpiricalDistribution → quantile(alpha)
            └─→ prepare_simulation_config(worst_resource)
                └─→ run_batch_parallel()
                    └─→ analyze_batch_results()
                        └─→ WorstImpactResult → UI 展示
```

## 潜在问题

1. **DrawTargetStrategy 未使用**：注册于 `STRATEGY_REGISTRY['draw_target']` 但 `analyze():176` 硬编码 `strategy_name='smart'`，可能是预留接口
2. **max_workers=1**：首轮分析已使用条件分布分类，第二轮串行模拟 99 池 × N 次 → 较慢
3. **99池同质化假设**：所有新池继承相同的基础奖励分布（仅替换 featured 卡），实际新池可能各不相同
4. **临时文件泄漏**：`_plot_distribution()` 使用 `NamedTemporaryFile(delete=False)` 但未清理 PNG
5. **resource 硬编码**：`ConditionalResourceDistribution` 默认 `resource='draw_resource'`，不支持多资源类型
