## 1. 图表右侧边缘截断问题

**归属模块**：`subsystems/图表引擎/`
**搁置日期**：2026-05-22

**问题描述**：
统计分析界面「截止每池的GDR分布」山脊线图，最右侧柱子的右边线被切掉。其他图表可能也存在类似问题。

**已排查的根因方向**：
- step/fill_between 逻辑已修正（`bin_edges` + `np.append`），非代码逻辑问题
- 所有 25 处 `savefig` 已添加 `pad_inches=0.15`，但未改善
- `tight_layout()` + `savefig(bbox_inches='tight')` 组合可能是根因，但单纯加 `pad_inches` 无效

**可能需要尝试的方向**：
1. 放弃 `bbox_inches='tight'`，改用显式 `subplots_adjust(left=..., right=..., top=..., bottom=...)` 手动控制边距
2. 对山脊线图单独设置 `ax.set_xlim(x_min, x_max * 1.05)` 留右侧余量
3. 检查 matplotlib 版本是否有已知的 tight bounding box 计算 bug
4. 尝试用 `constrained_layout` 替代 `tight_layout`

**影响范围**：
6 个文件、25 处 savefig：analysis_panel.py (17)、vulnerability.py (4)、resource_search_panel.py (1)、strategy_panel.py (1)、retreat_search_panel.py (1)、worst_impact_panel.py (1)

## 2. 流式分析 — Dict → List 打包优化（方案C）

**归属模块**：`subsystems/流式分析/`
**搁置日期**：2026-05-23（来源：`streaming_memory_optimization_plan.md` 归档时提取）

**问题描述**：
`SharedResultCollector._extractors` 中每个字段以 Dict 存储，10 万次模拟下 14 个字段 × 每个 Dict 的 Python 开销约 1.17 GB。方案 C 将小字段从 Dict 改为 List 打包以节省内存。

**搁置原因**：
方案 A（heatmap 流式预分箱）已实施，10 万次模拟内存从 ~3.8 GB 降至 ~1.9 GB，当前内存压力可接受。方案 C 的 Dict → List 重构涉及较多 API 变更，投入产出比不高。

**未来可能重新评估的条件**：
- 模拟次数需要达到 50 万+ 级别
- 需要支持更多 extractor 字段（会放大 Dict 开销）

## 4. 转变分析：UI 暴露 GDR 选择器 + 范围 + 阈值

**归属模块**：`panels/统计分析/`
**搁置日期**：2026-05-24（来源：P11 转变分析统一计划讨论）

**问题描述**：
当前转变分析的成功判定 UI 是三个固定选项的下拉框（全部目标卡达成 / 至少一张SSR / 每池至少一张目标卡），内部映射到 GDR key + scope + threshold。一种更灵活的设计是直接暴露 GDR 下拉选择器（与统计分析面板复用 `populate_gdr_combo()`）+ 范围切换（累积到该池 / 仅该池）+ 阈值微调控件。

**搁置原因**：
当前三选项简单够用，满足用户主要分析场景。暴露完整 GDR 配置会增加 UI 复杂度，且大部分 GDR 指标的语义在「逐池判定」场景下不够直观（如 `resource_efficiency` 在单池 scope 下含义模糊）。

**未来可能重新评估的条件**：
- 用户需要自定义阈值（如「出至少 2 张目标卡」）
- 新增更多适合逐池判定的 GDR 指标
- 需要与过程分析面板的每池 GDR 判定保持一致的配置体验

**放弃原因**：

## 7. 最差影响分析：MAX_POOLS 动态上限（修复 3）

**归属模块**：`panels/最差影响/`
**搁置日期**：2026-05-25（来源：`docs/最差影响分析修复方案.md` 修复 3）

**问题描述**：
`worst_impact.py` 中 `MAX_POOLS = 99` 硬编码为模块常量。绝大多数场景实际仅需 0-10 个池子，99 浪费构造开销；极端情况下（运气极好每池 1 抽出货），99 池天花板会截断真实结果。

**方案**：改为动态上限 `ceil(worst_resource / cost_per_draw) + 5`，从模块常量变为运行时常量。

**搁置原因**：
当前模拟速度已经足够快，99 池的构造开销在实际使用中不可感知。动态上限的收益（节省几十毫秒构造时间）远小于改造风险（需将 `MAX_POOLS` 从模块常量改为函数参数，涉及 4 处引用和 `PoolScheduleManager` 初始化逻辑）。

**未来可能重新评估的条件**：
- 成本大幅降低（如 cost=1）导致构造池数激增
- 出现 99 池天花板截断真实结果的用户报告
- 性能分析显示池构造成为瓶颈

## 8. P11 可选方案：渲染器抽象 + matplotlib 保底（第八章）

**归属模块**：`subsystems/图表引擎/`
**搁置日期**：2026-05-26（来源：`docs/Plotly图表迁移实施方案.md` 第八章）

**方案描述**：
在 ChartSpec → PlotlyRenderer → ChartWebView 架构之上增加一层 `ChartWidget(QStackedWidget)` 渲染器抽象。运行时自动检测 PyQt6-WebEngine 是否可用：已安装走 Plotly 交互图表，未安装回退 matplotlib 静态图表。支持 `pip install gacha_simulator`（轻量版）和 `pip install gacha_simulator[full]`（完整版）两种分发形态。增量约 +400-500 行（mpl_charts.py + chart_widget.py）。

**搁置原因**：
1. **双路径维护负担**：9 种图表类型需在 PlotlyRenderer 和 MatplotlibRenderer 中各实现一遍，每次 ChartSpec 变更需同步两个渲染器
2. **功能不对等**：matplotlib 回退路径无法支持 `update_chart()` 增量更新、`set_charts()` JS 标签切换、交互式功能（悬停/缩放/保存），analysis_panel 和 retreat_panel 体验显著降级
3. **收益未兑现**：当前项目无 pip 发布、无 exe 打包计划，132MB WebEngine 节省无实际受益用户（YAGNI）
4. **mpl_charts.py 非纯搬迁**：现有 matplotlib 代码入参为散装参数、出参为临时文件，适配 ChartSpec 接口需实质性重写

**未来可能重新评估的条件**：
- 实际开始打包分发（pyinstaller/nuitka → exe），安装包体积确实成问题
- 有用户反馈 PyQt6-WebEngine 无法安装（企业环境、特殊 Linux 发行版）
- 项目从单人开发变为多人协作，有 dedicated 维护带宽分担双渲染器成本
