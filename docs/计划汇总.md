### P12：模拟性能优化

- **实施计划**：`docs/模拟性能优化方案.md`（v1，2026-05-25）
- **状态**：未开始
- **目标**：1000 样本从 3-5s → 0.3-0.5s；1000 万样本从 8-14h → 4-20 分钟（20 核）
- **分三阶段渐进式实施**：

| 阶段 | 方案 | 预期加速 | 风险 | 周期 |
|------|------|---------|------|------|
| Phase 1 | 算法优化（消除 dict 拷贝、预构建不可变数据、策略复用、池过滤缓存、defaultdict 计数器） | 20-30% | 极低 | 1-2 天 |
| Phase 2 | Numba 数值化内核（ID 映射系统 + 结构化 numpy dtype + @njit 核心循环 + 双路径回退） | 10-25x | 中 | 1-2 周 |
| Phase 3 | 并行架构优化（共享内存替代 pickle 传输 + 改进 chunksize + 流式聚合） | 30-40%（在 Phase 2 之上）| 中 | 3-5 天 |

- **依赖**：Phase 1 零风险独立可执行；Phase 2 需 `numba` 可选依赖（`pip install numba`）；Phase 3 依赖 Phase 2
- **关键约束**：GUI、策略系统、停止条件的外部接口保持不变；Numba 不可用或策略不兼容时自动回退 Python 路径
- **不推荐方案**：GPU 加速（大概率负优化，数据传输开销 > 计算收益）、C/C++ 扩展（需用户手动搭编译工具链）
- **关联**：P10（Worker 全局变量消除）完成后有利于 Phase 1 的 `_wk_init` 改造

## 五、建议执行顺序

```
第0步：决定 P9 vs P11（图表体系统一方向）
         ↓
第1轮（可并行）：P7-P11（转变分析）| P2（过程分析续）| P10（Worker变量）|
                P7-P12（策略参数UI）| P12-Phase1（算法优化）
         ↓
第2轮（可并行）：P7-P4→P5→P13（GDR框架+三种新GDR）| P7-P9（卡牌初始持有）| P3（Bootstrap）
         ↓
第3轮：P3 → P4（自适应+EVT，在Bootstrap框架上扩展）
        P12-Phase2 → P12-Phase3（性能优化，在算法优化基础上深化）
         ↓
第4轮：图表统一（P9 或 P11 之一）完成后 → P7-P8（多资源类型）+ P8（面板合并）
         ↓
第5轮：P7-P10（额外资源，依赖 P7-P9）
```

**并行度最大化**：第 1 轮 5 项互不依赖；第 2 轮两大主线（GDR 框架 + 机制层 + Bootstrap）可同时推进。

**关键决策点**：P9 vs P11 的选择影响第 4 轮及以后所有涉及图表的条目。建议在进入第 4 轮之前做出最终选择。

**快赢项**：P12-Phase1（1-2 天，20-30% 加速，零风险）和 P10（Worker 变量消除）可在不阻塞其他计划的情况下提供立即可见的收益。
