## 1. 图表右侧边缘截断问题

**搁置日期**：2026-05-22

**问题描述**：
统计分析界面「截止每池的GDR分布」山脊线图，最右侧柱子的右边线被切掉。其他图表可能也存在类似问题。

**已排查的根因方向**：
- step/fill_between 逻辑已修正（`bin_edges` + `np.append`），非代码逻辑问题
- 所有 25 处 `savefig` 已添加 `pad_inches=0.15`，但未改善
- `tight_layout()` + `savefig(bbox_inches='tight')` 组合可能是根因，但单纯加 `pad_inches` 无效

**可能需要尝试的方向**：
1. 放弃 `bbox_inches='tight'`，改用显式 `subplots_adjust(left=..., right=..., top=..., bottom=...)` 手动控制边距
2. 对山脊线图单独设置 `ax.set_xlim(x_min, x_max * 1.05)` 留右侧余量
3. 检查 matplotlib 版本是否有已知的 tight bounding box 计算 bug
4. 尝试用 `constrained_layout` 替代 `tight_layout`

**影响范围**：
6 个文件、25 处 savefig：analysis_panel.py (17)、vulnerability.py (4)、resource_search_panel.py (1)、strategy_panel.py (1)、retreat_search_panel.py (1)、worst_impact_panel.py (1)

## 2. 流式分析 — Dict → List 打包优化（方案C）

**搁置日期**：2026-05-23（来源：`streaming_memory_optimization_plan.md` 归档时提取）

**问题描述**：
`SharedResultCollector._extractors` 中每个字段以 Dict 存储，10 万次模拟下 14 个字段 × 每个 Dict 的 Python 开销约 1.17 GB。方案 C 将小字段从 Dict 改为 List 打包以节省内存。

**搁置原因**：
方案 A（heatmap 流式预分箱）已实施，10 万次模拟内存从 ~3.8 GB 降至 ~1.9 GB，当前内存压力可接受。方案 C 的 Dict → List 重构涉及较多 API 变更，投入产出比不高。

**未来可能重新评估的条件**：
- 模拟次数需要达到 50 万+ 级别
- 需要支持更多 extractor 字段（会放大 Dict 开销）

## 3. 过程分析：success_distribution UI 展示

**搁置日期**：2026-05-25

**问题描述**：
过程分析 AB 分析（事件→成败）中，`compute_ab()` 已计算各事件类型下的成功率分布（`success_distribution`），但 UI 仅展示条件概率表（`p(success|event)` 数值），未以图表形式展示分布。

**搁置原因**：
当前表格展示已满足基本分析需求。图表展示需额外设计（直方图 vs 箱线图 vs KDE），且 AB 分析结果已足够支撑决策，优先度不高。

**未来可能重新评估的条件**：
- 用户对「同一事件类型下成功率的离散程度」产生疑问
- 过程分析面板 UI 有足够的展示空间

## 4. 转变分析：UI 暴露 GDR 选择器 + 范围 + 阈值

**搁置日期**：2026-05-24（来源：P11 转变分析统一计划讨论）

**问题描述**：
当前转变分析的成功判定 UI 是三个固定选项的下拉框（全部目标卡达成 / 至少一张SSR / 每池至少一张目标卡），内部映射到 GDR key + scope + threshold。一种更灵活的设计是直接暴露 GDR 下拉选择器（与统计分析面板复用 `populate_gdr_combo()`）+ 范围切换（累积到该池 / 仅该池）+ 阈值微调控件。

**搁置原因**：
当前三选项简单够用，满足用户主要分析场景。暴露完整 GDR 配置会增加 UI 复杂度，且大部分 GDR 指标的语义在「逐池判定」场景下不够直观（如 `resource_efficiency` 在单池 scope 下含义模糊）。

**未来可能重新评估的条件**：
- 用户需要自定义阈值（如「出至少 2 张目标卡」）
- 新增更多适合逐池判定的 GDR 指标
- 需要与过程分析面板的每池 GDR 判定保持一致的配置体验
