# Plan Skill — AI 主导的计划生成

将用户给的种子方向细化为可执行计划文件，并自动注册到模块状态矩阵。

## 触发

- `/plan <种子描述>` — slash command
- 自然语言中检测到"制定计划"/"创建P编号"/"细化方案"等意图时也应触发

## 流程

### 1. 理解种子方向

- 如果是简单一句话（如"给所有面板加导出CSV功能"），先搜索影响面再确认
- 如果是详细描述，基于描述展开
- 如果不明确（范围/优先级/成功标准），先问 1-2 个澄清问题

### 2. 搜索影响面

使用 Grep/Glob 搜索：
- 受影响的面板文件 (`gui/*_panel.py`)
- 受影响的 core 模块
- 现有相关计划（避免重复）
- 相关文档

### 3. 评估复杂度

| 复杂度 | 标准 | 行动 |
|--------|------|------|
| 小 (P2) | 单文件修改，1-2个函数，无架构变更 | 创建单一P编号计划 |
| 中 (P1) | 多文件，需要跨模块协调 | 创建单一P编号，含分阶段 |
| 大 (P0) | 架构变更，3+模块，需要多种技能 | 建议拆分为 2+ 子计划 |

### 4. 确定归属模块

- 若影响单个面板 → `docs/01-活跃/panels/{面板名}/`
- 若影响单个子系统 → `docs/01-活跃/subsystems/{子系统}/`
- 若跨模块 → 选择主要模块，在计划中标注波及范围
- 若全新模块 → 在 `docs/01-活跃/` 下创建新目录（含三文件骨架：模块.md + 理论.md + 05-笔记.md）

### 5. 确定 P 编号

- 读取 `docs/00-meta/模块状态矩阵.md` → 找到当前最大 P 编号
- 新编号 = 最大 + 1（跳过已用编号）
- P 编号全局唯一

### 6. 写入计划文件

**文件名格式：** `P{n} {中文名称}.md`

**文件内容模板：**

```markdown
<!-- META: P{n} | module:{模块文件夹名} | status:designing | last:{YYYY-MM-DD} -->

# P{n} {名称}

> 日期：{日期} | 状态：设计中
> 触发：{种子描述或问题来源}

## 一、问题

{当前状态——什么不工作/缺失/待改进}

## 二、目标

{成功标准——完成后是什么样}

## 三、方案

{实施路径——阶段划分、关键决策、技术选型}

## 四、波及范围

{受影响文件列表——面板/core/服务/文档}

## 五、风险

| 风险 | 缓解 |
|------|------|
| ... | ... |

## 六、验收标准

- [ ] ...
```

### 7. 注册到模块状态矩阵

在 `docs/00-meta/模块状态矩阵.md` 中：
- 在对应的优先级段（P0/P1/P2）添加新行
- 格式：`| P{n} | {名称} | {一句话描述} | {计划文件路径} | {归属模块} |`
- 更新"最后更新"日期

### 8. 输出确认

```
已创建 P{n} {名称}
  文件: {计划文件路径}
  模块: {归属模块}
  优先级: {P0/P1/P2}
  预估: {N}阶段 / {N}文件

波及范围: {一句话总结}
```

## 约束

- 计划文件名中文，不含特殊字符
- META 头**必须在首行**，供 C2 cron agent grep 提取
- 不重复已有计划的 P 编号
- 计划文件遵循增量修改原则（后续编辑时标注/替换/更正/补充，不整段删除）
- 若种子方向与已有计划重叠 → 先报告已有计划，建议合并而非新建
